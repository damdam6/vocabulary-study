# 범용화 PRD — 프로필(비밀번호 ↔ 시트 ↔ 모드) 기반 다중 시트 지원

## 1. 개요

**한 줄 요약**: 메인(로그인)에서 입력한 비밀번호가 곧 "어떤 시트로, 어떤 방식으로 공부하는가"를 결정한다. 비밀번호마다 다른 스프레드시트가 열리고, 시트(프로필)마다 퀴즈 모드 구성(모드 1만 / 모드 2만 / 둘 다)이 달라진다.

| | v1 (PRD.md) | 범용화 (이 문서) |
|---|---|---|
| 비밀번호 | 1개 (`APP_PASSWORD`) | 프로필당 1개 (전 프로필에서 유일) |
| 스프레드시트 | 1개 (`SHEET_ID`) | 프로필당 1개 |
| 퀴즈 모드 | 모드 1·2 고정 | 프로필당 구성: `["m1"]` / `["m2"]` / `["m1","m2"]` |
| 콘텐츠 | 중국어 단어 전용 | 프로필당 콘텐츠 타입: `zh`(중국어) / `generic`(일반 암기 항목) |
| 사용자 | 소유자 1인 | 프로필 수만큼 — 여전히 회원가입 없음, 비밀번호 입력이 곧 프로필 선택 |

- 사용 예: 소유자의 중국어 단어 시트(모드 1+2, `zh`), 영어 표현·일반 암기용 시트(모드 1만, `generic`) 등.
- 서비스의 정체성 — 구글 시트가 원천, 고정 간격 반복, 문제 단위 즉시 기록 — 은 그대로다. 이 문서는 "누가 어느 시트를 어떤 모드로 보는가"라는 차원 하나를 추가할 뿐이다.

## 2. 용어

- **프로필(profile)**: 비밀번호 1개 ↔ 스프레드시트 1개 ↔ 모드 구성 1개의 묶음. 이 문서의 핵심 단위.
- **시트**: 요구사항 원문("비밀번호에 따라 접근할 수 있는 시트가 달라짐")의 "시트"는 **스프레드시트(문서) 단위**다(소유자 확인으로 확정 — §10 Q1). PRD.md §4.1은 문서 내 시트 탭을 "탭(시트)"이라 부르므로, 혼동을 피하기 위해 이 문서에서는 문서 단위를 "스프레드시트", 문서 내 단위를 "탭"으로만 쓴다.
- **활성 모드 집합 M**: 프로필의 `modes`. 원소는 `m1`(표제어→뜻 자가판정), `m2`(뜻→표제어 입력 정확일치 채점). §4의 학습 모델은 전부 M으로 파라미터화되며, M = {m1, m2}이면 v1과 완전히 동일하게 동작한다.
- **콘텐츠 타입**: 프로필의 `contentType` — `"zh"`(중국어 단어, v1과 동일) 또는 `"generic"`(일반 암기 항목 — 영어 표현 등). 학습 모델·시트 계약에는 영향이 없고, 화면 렌더링과 등록 화면 노출만 가른다(§7).
- **표제어**: A열 텍스트의 일반화 명칭. `zh`에서는 한자, `generic`에서는 임의 텍스트(문제의 정답).

## 3. 프로필 구성

### 3.1 스키마

Worker 시크릿 `PROFILES`에 JSON 배열로 저장한다.

```json
[
  {
    "id": "zh",
    "name": "중국어 단어",
    "password": "…",
    "sheetId": "1WEo1b15RRq2NfrJqzsLA1_-6Rs3nIZl2q5sn7doBvw8",
    "modes": ["m1", "m2"],
    "contentType": "zh"
  },
  {
    "id": "en",
    "name": "영어 표현",
    "password": "…",
    "sheetId": "1AbC…",
    "modes": ["m1"],
    "contentType": "generic"
  }
]
```

| 필드 | 규칙 | 용도 |
|---|---|---|
| `id` | 전 프로필 유일, 짧은 슬러그. 비밀 아님. **한 번 정하면 바꾸지 않는다** (재시도 큐 태그가 고아가 됨, §8) | 재시도 큐 태깅(§8), 로그 식별 (비밀번호는 절대 로그 금지) |
| `name` | UI 표시명 | 홈 화면 표기(§7) |
| `password` | 전 프로필 유일, 빈 문자열 금지 | 인증 겸 프로필 선택(§6) |
| `sheetId` | 스프레드시트 ID. 중복 금지는 하지 않으나 권장하지 않음(카운트 열이 공유돼 두 프로필의 진도가 섞인다) | 모든 Sheets 호출 대상(§5) |
| `modes` | `"m1"`/`"m2"`의 비어 있지 않은 부분집합, 중복 없음 | 학습 모델 파라미터(§4) |
| `contentType` | `"zh"` 또는 `"generic"`. 생략 시 `"zh"` (폴백 프로필 포함 하위 호환) | 등록 화면 노출·표제어 렌더링 분기(§7) — 학습 모델에는 무관 |

### 3.2 검증과 실패 동작

- Worker는 `PROFILES`를 파싱·검증하고 isolate 수명 동안 캐시한다 (v1의 비밀번호 다이제스트 캐시와 같은 패턴).
- 검증 실패(JSON 파싱 불가, 필드 누락·형식 위반, `password`/`id` 중복, `modes` 빈 배열 등) 시 **fail closed**: 모든 `/api/*`가 `500`(서버 설정 오류)으로 응답하고 원인을 `console.error`로 남긴다. `401`(비밀번호 불일치)과 구분하는 이유: 설정 사고를 "비밀번호가 틀렸다"로 오독하지 않고 즉시 관측하기 위함.
- **폴백**: `PROFILES` 미설정이면 기존 `APP_PASSWORD` + `SHEET_ID`로 단일 프로필을 합성한다 — `{ id: "default", name: "단어 암기", modes: ["m1","m2"], contentType: "zh" }`. Workers Builds가 머지 즉시 자동 배포하므로(#67), 시크릿 설정과 코드 배포 사이의 순서 제약을 없애고 롤백(구 코드는 `PROFILES`를 읽지 않음)을 안전하게 한다. 운영 전환 완료 후 정리 단계에서 폴백과 구 변수를 제거한다(§9).

### 3.3 스프레드시트 내부 계약 (불변 — 의미만 일반화)

- 각 프로필의 스프레드시트 내부는 PRD.md §4를 그대로 따른다: 탭 여러 개 = 출처별 분류, `_` 접두 탭 제외, 각 탭 1행 헤더 스킵, A~G열 컬럼 계약 동일, 프로필 안에서는 전 탭 통합 학습 큐.
- 컬럼의 **의미는 콘텐츠 중립으로 읽는다**: A열 = 표제어(모드 2 채점의 정확일치 대상), B열 = 보조 표기(발음 등 — **빈칸 허용**), C열 = 뜻. `zh`에서는 v1과 동일하게 한자/병음/뜻이다. 탭 내 유일 키가 A열이라는 규칙도 동일.
- 모드 구성과 무관하게 **컬럼 배치는 고정**: D열 = 모드 1 정답 수, E열 = 모드 2 정답 수. 비활성 모드의 열은 빈칸/0으로 남을 뿐이다(§4.1). 계약을 프로필별로 바꾸지 않는 이유: 파서·등록·기록 코드가 하나로 유지되고, 나중에 모드 구성을 바꿔도 데이터 마이그레이션이 없다.
- 서비스 계정은 **모든 프로필 스프레드시트에 편집자로 공유**돼 있어야 한다. 새 프로필 추가 절차는 §9.

## 4. 학습 모델 파라미터화 (PRD.md §5·§6 개정)

### 4.1 상태 머신 (§5.1 대체)

상태는 여전히 D·E·F열에서 파생하되, 판정이 M 기준이 된다.

| 상태 | 판정 조건 (M 기준) |
|---|---|
| **학습 중** | M의 어느 한 모드라도 카운트 < 3 |
| **복습 대기** | M의 모든 모드 카운트 ≥ 3 그리고 F열 날짜 ≤ 오늘 |
| **복습 예약** | M의 모든 모드 카운트 ≥ 3 그리고 F열 날짜 > 오늘 |

- 졸업 = "M의 모든 모드 ≥ 3" 최초 달성. M = {m1}이면 D ≥ 3만으로 졸업한다. 졸업 순간 F열에 `내일|1`을 쓰는 규칙은 동일.
- **비활성 모드의 카운트 열은 판정에서 무시하고, 어떤 경로로도 쓰지 않는다** (Worker가 강제 — §5.2).
- 오답 시 카운트 리셋 없음, 복습 오답 = 간격 한 단계 후퇴 등 나머지 규칙 전부 불변.

### 4.2 출제 모드 (§5.2·§5.3·§6.1 개정)

- **복습 문제**: M 중 무작위 (M이 1개면 항상 그 모드).
- **학습 중 문제**: M 중 미달(<3) 모드에서 선택 — 하나만 미달이면 그 모드, 둘 다 미달이면 무작위. v1 #44 규칙("단어당 1문제")의 M 제한판이다.
- 모드 1·2 각각의 문제 플로우(플립·자가판정 / 입력·정확일치 채점)와 세션 규칙(60문제 상한, 셔플, 학습 중 오답 1회 재삽입, 복습 오답 재출제 없음)은 변경 없다.

### 4.3 모드 구성 변경 시 재해석

상태를 저장하지 않고 파생하므로, 프로필의 `modes`를 바꾸면 **다음 조회부터 새 구성으로 재해석**된다. 별도 데이터 마이그레이션은 없다. 두 가지 엣지를 의도된 동작으로 문서화한다:

- **축소(예: {m1,m2} → {m1})**: m2만 미달이던 단어가 일괄 졸업 상태가 되는데 F열이 빈칸이다 → 기존 데이터 이상 처리 규칙(wordState — `nextReview` null인 졸업 단어는 복습 대기 취급)에 따라 복습 대기로 잡히고, 첫 복습 정답에서 F열이 기록되며 자가치유된다.
- **확대(예: {m1} → {m1,m2})**: 졸업했던 단어들이 E < 3이므로 일괄 "학습 중"으로 되돌아온다. 확대 전 소유자가 인지하고 결정할 사항.

## 5. 시스템 아키텍처 변경 (PRD.md §7 개정)

```
[브라우저: React SPA]
   │  fetch /api/* (Authorization: Bearer <비밀번호>)
   ▼
[Cloudflare Worker]  ── 비밀번호 → 프로필 해석(§6) 후 라우트에 전달
   │  Google Sheets API v4 (서비스 계정 JWT, 전 프로필 공용)
   ▼
[프로필별 구글 스프레드시트 × N]
```

- Worker 1개·배포 1개·SPA 1개는 유지. 요청마다 인증 단계에서 프로필을 해석하고, 이후 모든 Sheets 호출이 그 프로필의 `sheetId`를 쓴다 (v1의 `env.SHEET_ID` 직접 참조 제거).
- 서비스 계정과 액세스 토큰 캐시는 전 프로필 공용이다 — 토큰 스코프가 spreadsheets 전체라 프로필별 토큰이 필요 없다.

### 5.1 환경변수/시크릿 (§7.2 대체)

| 이름 | 내용 | 비고 |
|---|---|---|
| `PROFILES` | 프로필 JSON 배열 (secret) | §3.1 |
| `GOOGLE_SERVICE_ACCOUNT_KEY` | 서비스 계정 키 JSON (secret) | 불변, 전 프로필 공용 |
| `SHEET_ID` · `APP_PASSWORD` | 폴백 합성 전용(§3.2) | 전환 완료 후 제거 |

### 5.2 API 명세 변경 (§7.3 개정분)

인증 형식은 동일하게 `Authorization: Bearer <비밀번호>`, 불일치 시 `401` — 다만 비밀번호가 프로필을 선택한다. 아래 언급 없는 엔드포인트(`POST /api/review-fail`, `GET /api/tabs`, `POST /api/words/register`)는 계약 불변이며 대상 스프레드시트만 프로필로 결정된다.

- **`GET /api/health`** — 응답에 인증된 프로필 정보를 추가한다.
  ```json
  { "ok": true, "time": "…", "version": "…",
    "profile": { "id": "zh", "name": "중국어 단어", "modes": ["m1", "m2"], "contentType": "zh" } }
  ```
  `sheetId`·`password`는 응답에 포함하지 않는다. 로그인 화면이 이 응답으로 프로필을 저장한다(§7).
- **`GET /api/words`** — 응답 최상위에 같은 형태의 `profile` 블록을 추가한다. `words` 항목 형태는 불변. 홈이 진입마다 재조회하므로(design-prd §3) 모드 구성 변경이 다음 홈 진입에 자연히 반영된다 — 클라이언트 학습 로직은 이 응답의 `profile.modes`를 쓴다.
- **`POST /api/answer`** — 요청 `mode`가 프로필 `modes`에 없으면 `400`. 정상 클라이언트는 비활성 모드 문제를 만들지 않으므로 이는 방어선이다 — "비활성 모드 열은 어떤 경로로도 쓰지 않는다"(§4.1)의 서버 측 강제. 실경로는 모드 축소 직후 재시도 큐가 옛 모드 기록을 재전송하는 경우뿐이고, 이때 무효가 된 기록은 버려지는 게 맞다(§8).

## 6. 인증·보안 (PRD.md §8 개정)

- Bearer 비밀번호를 각 프로필 비밀번호의 SHA-256 다이제스트와 **상수 시간 비교**해 일치하는 프로필로 해석한다 (v1 auth 패턴의 다중화 — 유일성은 구성 검증이 보장하므로 첫 일치가 곧 유일 일치다). 전 프로필 불일치면 `401`.
- 다이제스트는 isolate 수명 동안 캐시한다. 프로필 수는 한 자릿수 전제라 순회 비용은 무시 가능.
- 클라이언트 동작(localStorage 저장, 모든 호출에 Bearer 첨부, `401` 수신 시 저장값 삭제 후 로그인 복귀)은 v1 그대로.
- 위협 모델 불변: 개인·지인 용도, 레이트리밋·세션 만료는 여전히 비범위. 프로필 간 경계는 "비밀번호를 모르면 그 시트를 열 수 없다"가 전부다. 단, **프로필 간 데이터 격리는 구현이 보장해야 한다** — 모든 Sheets 호출 경로가 인증된 프로필의 `sheetId`만 쓰는지가 코드 리뷰의 핵심 확인점이다.

## 7. 화면 변경 (PRD.md §9 개정 — design-prd 개정은 구현 작업에 포함)

- **로그인**: UI 불변 (문구 "구글 시트의 단어로 오늘의 퀴즈를 만듭니다"는 범용화 후에도 유효). 검증 성공 시 `GET /api/health` 응답의 `profile`을 localStorage(키 `vocab-study:profile`)에 저장한다.
- **홈**:
  - 프로필 `name`을 저강조로 표기한다 (지금 어느 시트에 들어와 있는지 확인용 — 위치·스타일은 design-prd 개정에서 확정, §10 Q5).
  - **"프로필 전환" 저강조 링크 추가** — 저장된 비밀번호·프로필 캐시를 지우고 로그인 화면으로 보낸다. v1에는 로그아웃이 없어서, 이 링크가 없으면 다른 프로필로 갈아탈 방법이 없다.
  - `/api/words` 응답의 `profile`로 저장본을 갱신한다 (모드 구성 변경 반영 경로).
- **학습·완료**: 플로우 불변. 문제별 모드 칩·진행 규칙 그대로. 렌더링만 콘텐츠 타입으로 분기한다:
  - **`zh` 전용 처리**: 한자 글자 수 적응 크기(design-prd §1.2), `lang="zh-Hans"` 지정, 모드 2 입력란의 중국어 속성(placeholder "汉字" 등), 모드 칩 문구 "한자 → 뜻"/"뜻 → 한자". `generic`은 일반 텍스트로 표시하고 칩 문구도 중립("단어 → 뜻"/"뜻 → 단어")으로 한다. 구체 크기·스타일은 design-prd 개정에서 확정.
  - **B열(보조 표기) 빈칸이면 그 줄을 숨긴다** — 콘텐츠 타입 무관 공통 규칙 (`zh` 시트에서 B열이 빈 행도 동일).
- **단어 등록**: **`contentType: "zh"` 프로필에만 홈 진입 링크를 노출한다.** 등록 파이프라인(추출 킷·pinyin-pro 검증·Worker 재검증)은 중국어 전제 그대로 유지한다. `generic` 프로필의 단어 추가는 구글 시트 직접 편집으로 한다 — 등록 화면 도입(2026-07-19) 전 v1의 원래 방식이며, 시트 계약이 단순(A~C열)해 충분히 실용적이다. 등록 화면의 콘텐츠 타입별 일반화(검증 규칙 분기·추출 킷 변형)는 **후속 플랜으로 분리**한다(§10 Q4).

## 8. 재시도 큐 (PRD.md §10 개정)

- 큐 엔트리에 `profileId`를 태깅한다. flush는 **현재 로그인한 프로필의 항목만** 적재 순서(FIFO)로 전송하고, 다른 프로필의 항목은 건드리지 않고 보존한다 — 그 프로필로 재로그인하면 그때 전송된다. 홈의 미전송 인디케이터도 현재 프로필 항목만 센다.
  - 태깅이 없으면: 프로필 전환 후 flush가 이전 프로필의 기록을 **새 프로필의 시트에** 던진다. 대상 단어가 없으면 `404`로 큐 머리가 영구히 막히고(현 구현은 첫 실패에서 중단·유지), 같은 표제어가 우연히 있으면 **다른 시트에 오기록**된다. 이 방지가 태깅의 존재 이유다.
- 태그 없는 레거시 항목(단일 프로필 시절 적재분)은 읽기 시점에 현재 프로필 소속으로 승격한다 — retryQueue의 기존 무태그 kind 승격(#43)과 같은 접근.
- **영구 실패 폐기**: 재전송이 `4xx`(비활성 모드 `400`, 단어 삭제 `404` 등)를 받으면 그 항목은 재시도해도 영원히 실패하므로 **버리고 다음 항목을 계속** 전송한다. "첫 실패에서 중단·유지"는 일시 실패(네트워크 예외·`5xx`)에만 적용한다. (v1의 중단 규칙을 실패 성격별로 정밀화한 것 — 4xx 하나가 큐 전체를 영구히 막는 기존 동작은 프로필 전환이 생기면 실제로 밟게 되는 지뢰다.)

## 9. 마이그레이션·운영

- **전환 순서** (폴백 덕에 순서 제약은 없지만 권장 순서):
  1. 코드 머지 — `PROFILES` 미설정이므로 폴백으로 기존과 동일하게 동작.
  2. `PROFILES` 시크릿 설정 — 1번 프로필에 현 `APP_PASSWORD`·`SHEET_ID` 값을 그대로 옮기면 기존 사용자는 재로그인 불필요(비밀번호 동일). 프로필 캐시가 없어도 다음 `/api/words` 응답에서 채워진다.
  3. 새 프로필 추가: ① 스프레드시트 생성(기존 시트 사본 또는 헤더 행만) → ② 서비스 계정 이메일을 편집자로 공유 → ③ `PROFILES`에 항목 추가(시크릿 갱신은 새 Worker 버전을 만들어 즉시 반영).
  4. 안정화 후 `APP_PASSWORD`·`SHEET_ID`·폴백 코드 제거.
- **롤백**: 구 코드는 `PROFILES`를 읽지 않으므로 코드 롤백만으로 v1 동작 복귀.
- **데이터 마이그레이션**: 없음 (시트 계약 불변, 상태는 파생 — §4.3).
- **스모크(#23·#67)**: 무인증 `401` 검사 불변. `deploy-auth-smoke.sh`의 `SMOKE_PASSWORD`는 아무 프로필의 비밀번호면 200 경로가 유효하다.
