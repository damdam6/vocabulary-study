#58문서 정합화

PRD §10 재시도 큐 서술 갱신 — review-fail 포함 (#43 반영)

damdam6/vocabulary-study 브랜치 docs/58-prd-retry-queue-wording ← main 라벨 documentation 2026-07-19

1요약

#43(PR #45)에서 POST /api/review-fail 실패도 재시도 큐에 적재되도록 확장됐지만, docs/PRD.md §10 서술은 아직 #43 이전 상태로 POST /api/answer 실패만 재시도 대상으로 적고 있다. 실제 코드(src/lib/retryQueue.ts)는 answer·review-fail 두 종류를 하나의 큐에 kind 태그를 붙여 적재하고 FIFO로 재전송한다. §10을 양쪽 실패가 재시도 큐 대상임을 반영해 고치고, answer(타임스탬프 = 최초 발생 시각 보존)와 review-fail(간격 후퇴 재전송, tab·hanzi만 재생) 차이를 한 줄로 명시한다. §7.3 review-fail 절에도 "실패 시 재시도 큐" 한 줄을 더한다. 코드 변경 없는 순수 문서 정합화.

2현황 분석

재시도 큐의 실제 동작(src/lib/retryQueue.ts)이 사실상의 명세이며, 이미 answer·review-fail 두 종류를 모두 다룬다:

  • 엔트리 타입이 kind 태그로 두 종류를 구분한다 — { kind: "answer" | "review-fail", record } (retryQueue.ts:21-23).
  • 적재 진입점이 둘 — enqueueAnswer·enqueueReviewFail이 같은 배열 끝에 순서대로 push (retryQueue.ts:67-74).
  • 플러시가 kind로 분기 — answer는 postAnswer(record), review-fail은 postReviewFail(record.tab, record.hanzi) (retryQueue.ts:96-100).
  • 두 종류의 성격 차이 — answer는 timestamp(최초 판정 시각)를 그대로 보존해 재전송(retryQueue.ts:7-8), review-fail 레코드는 tab·hanzi만 담아 간격 후퇴 재요청을 재생한다(타임스탬프 개념 없음).

문서 쪽 현황:

  • docs/PRD.md:276 §10 "기록 유실 방지" — POST /api/answer 실패만 재시도 대상으로 기재. #43 이전 상태.
  • docs/PRD.md:211-214 §7.3 POST /api/review-fail 절 — Worker 동작만 서술, 전송 실패 시 재시도 큐로 간다는 언급 없음.
  • docs/design-prd.md:115 §4.5 — "기록 전송은 비동기 … 재시도 큐(#18)로"로 이미 넓게 서술되어 정합. 변경 불필요.

3설계

선택: 코드가 이미 정답이므로, 문서를 코드에 맞춘다. 두 곳만 손댄다.

  • §10 "기록 유실 방지" 한 항목을 answer·review-fail 양쪽 실패가 재시도 큐 대상이도록 다시 쓰고, 두 종류의 재전송 성격 차이를 한 줄로 붙인다.
  • §7.3 POST /api/review-fail 절 끝에 "전송 실패 시 재시도 큐(§10)에 적재·재전송" 한 줄 추가.

배제한 대안:

  • §7.3를 크게 재작성 — 이슈 범위(한 줄 추가)를 벗어남. answer 절과의 서술 톤도 무너뜨림.
  • design-prd §4.5 수정 — 이미 재시도 큐를 넓게 포괄하므로 건드리면 오히려 중복. 정합 확인만.
  • 큐 종류를 별도 표/절로 신설 — 비기능 요구사항 한 항목에 담기는 분량이라 과설계.

4작업 단계

  1. ✅ §10 "기록 유실 방지" 항목 수정 — POST /api/answer·POST /api/review-fail 양쪽 실패를 localStorage 재시도 큐 대상으로 기술. answer는 타임스탬프(최초 발생 시각) 보존, review-fail은 간격 후퇴 재전송이라는 차이를 한 줄로 명시. docs/PRD.md:277 · 완료 — 종류 태그·FIFO 재전송 문구 포함
  2. ✅ §7.3 POST /api/review-fail 절에 "전송 실패 시 재시도 큐(§10)에 적재 후 재전송" 한 줄 추가. docs/PRD.md:215 · 완료 — §7.3 answer 절엔 재시도 큐 줄이 없어 "대칭" 대신 §10을 참조하는 한 줄로 추가
  3. ✅ design-prd §4.5가 여전히 재시도 큐를 포괄하는지 재확인(모순 없음 확인만, 수정 없음). docs/design-prd.md:115 · 완료 — "재시도 큐(#18)로" 서술 그대로 정합, 변경 없음
  4. ✅ 최종 대조 — 갱신된 §10 서술이 retryQueue.ts의 실제 kind 분기·타임스탬프 처리와 문자 그대로 일치하는지 확인. src/lib/retryQueue.ts:21-23,96-100 · 완료 — kind 태그·answer 타임스탬프 보존·review-fail(tab·hanzi) 재전송 일치

5리스크

리스크영향완화책
서술이 review-fail도 타임스탬프를 보존하는 것처럼 읽힐 오해 낮음 — 문서 정확성 review-fail 레코드는 tab·hanzi만 담아 간격 후퇴만 재생함을 명시 (타임스탬프 보존은 answer에만 걸리도록 문장 구조 분리)
코드 변경 없음 → 기능 회귀 없음 순수 마크다운 diff, 실행 경로 무변경

6테스트 계획

  • 문서 전용 변경이라 실행 테스트 없음 — 마크다운 diff만 발생하며 빌드·런타임 경로에 영향 없음.
  • 코드-문서 대조: 갱신된 §10 문장을 retryQueue.ts:21-23(kind 태그), :67-74(두 적재 진입점), :96-100(kind 분기 재전송)와 한 줄씩 맞춰 불일치 없음 확인.
  • design-prd §4.5 재독 — "재시도 큐(#18)로" 서술과 갱신된 §10 사이에 모순 없음 확인.
  • 렌더 확인 — §10·§7.3의 마크다운(코드 백틱, 강조)이 깨지지 않는지 diff로 육안 점검.

7완료 조건

  • PRD §10이 answer·review-fail 양쪽을 재시도 큐 대상으로 기술
  • answer(타임스탬프 보존)와 review-fail(간격 후퇴 재전송)의 차이가 한 줄로 명시됨
  • §7.3 review-fail 절에 실패 시 재시도 큐 한 줄 추가됨
  • 코드(retryQueue.ts)의 실제 동작과 서술이 일치
  • design-prd와 모순 없음
