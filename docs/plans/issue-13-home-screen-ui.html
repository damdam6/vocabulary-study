#13신규 기능

홈 화면 UI 구현

damdam6/vocabulary-study 브랜치 feat/13-home-screen-ui ← main 라벨 enhancement 2026-07-18

1요약

design-prd §3이 정의하는 홈 화면(오늘 날짜 → 타이틀 → 현황 카드 3개 → 미전송 인디케이터 → "오늘 세션 · n문제" → "학습 시작" 버튼)을 구현한다. GET /api/words(#7)로 받은 단어 목록을 getWordState(#12)로 분류해 복습 대기/학습 중/졸업 수와 오늘 세션 문제 수(기능 PRD §6.1 큐 산식과 동일한 계산)를 계산하고, 로딩/에러/정상 3가지 화면 상태를 처리한다. 로그인 화면(#11)과 재시도 큐(#18)는 아직 구현되지 않았으므로, 이번 이슈에서 두 기능이 이후 그대로 재사용할 로컬스토리지 키·이벤트 계약을 먼저 정의해 둔다.

2현황 분석

  • src/screens/HomeScreen.tsx:1-16는 플레이스홀더 — 타이틀과 버튼 하나뿐이며 데이터 페칭도, 상태 판정도 없다. App.tsx:16이 onStart만 넘겨준다.
  • 의존 이슈 둘 다 완료됨: GET /api/words(#7)는 worker/routes/words.ts:10-26에 구현되어 { fetchedAt, words: WordEntry[] }를 반환하고, WordEntry(worker/lib/words.ts:8-17)는 이미 m1/m2/nextReview를 정규화해 넘긴다. getWordState·getSeoulToday(src/lib/wordState.ts:17-51)도 완성돼 있고 순수 함수라 그대로 재사용 가능.
  • 반면 로그인 화면(#11)은 아직 미착수 — src/screens/LoginScreen.tsx:1-16도 플레이스홀더이며 onLogin을 누르면 아무 저장 없이 바로 홈으로 이동한다. 즉 Authorization: Bearer <비밀번호>(worker/lib/auth.ts:2, 기능 PRD §7.2)를 localStorage에서 읽어올 저장 로직 자체가 레포에 아직 없다 — 이 이슈가 사실상 그 계약을 처음 정의하게 된다.
  • 재시도 큐(#18)도 미착수. design-prd §3(docs/design-prd.md:79)은 "재시도 큐에 항목이 있을 때만" 인디케이터를 표시하라고 하지만, 큐를 채우는 쪽(#18, POST /api/answer 실패 시 저장)이 아직 없어 이번 이슈는 "읽는 쪽" 계약만 먼저 세워야 한다.
  • design-prd §3 전문(docs/design-prd.md:71-80): 구성 순서, 현황 카드 3개 색 토큰(accent-light/text/green), "졸업 수 = reviewDue + reviewScheduled"(복습 대기 포함), 세션 수 산식 min(60, min(복습대기,60) + 학습중 후보 문제 수), 로딩/에러/미전송 상태 사양, "홈 진입 시마다 재조회"가 모두 명시돼 있다.
  • 기능 PRD §6.1(docs/PRD.md:129-136)의 후보 문제 수 정의: 학습 중 단어당 최대 2문제(D<3 → 모드1 후보, E<3 → 모드2 후보). 세션 수는 실제 큐 구성(#14, 별도 이슈) 없이 이 카운트만으로 계산 가능 — §6.1 전체를 이 이슈에서 구현할 필요는 없다.
  • src/index.css:134-144의 .screen-placeholder는 임시 스타일이며 주석에 "#11/#13/#15가 실제 화면으로 교체하면서 제거"라고 명시 — 홈 화면은 이 클래스를 더 이상 쓰지 않게 된다. 색 토큰(--bg-panel/--bg-disabled/--accent/--green/--accent-light/--red 등)은 src/index.css:1-23에 이미 정의돼 있다.
  • package.json:9-28에는 리액트 컴포넌트 테스트 도구(@testing-library 등)가 없다 — src/lib/wordState.test.ts 선례처럼 순수 함수 단위로 vitest 테스트를 쓰고, 컴포넌트 자체는 브라우저 수동 검증으로 확인하는 편이 기존 관례에 맞다.
  • 메모리 기록(2026-07-14 세팅): 실 스프레드시트 HSK6급 탭은 헤더만 있고 데이터 행이 비어 있다 — "실제 데이터로 카드 숫자 확인"을 검증하려면 검증용 임시 행이 필요하다(#7이 자체 완료 조건에서 쓴 것과 같은 패턴).

3설계

  • src/lib/auth.ts 신설 — getStoredPassword()/setStoredPassword()/clearStoredPassword(), localStorage 키 vocab-study:app-password. 홈 화면이 처음으로 실제 인증 헤더가 필요한 화면이라 이 계약을 여기서 만들고, #11이 그대로 재사용하게 한다. (대안: #11이 정의할 때까지 기다림 — 의존 방향이 반대다. #13은 #7·#12에만 의존하며 #11에 의존하지 않으므로, 저장 규약을 먼저 세우는 쪽이 맞다.)
  • src/lib/api.ts 신설 — apiFetch(path, init?): 저장된 비밀번호를 Authorization: Bearer로 붙여 fetch하고, 401이면 저장값을 지운다(기능 PRD §8). 화면 전환(로그인으로 복귀)까지는 #11이 없어 이번 이슈 범위 밖 — 홈 화면 입장에서 401은 다른 실패와 동일하게 에러 상태로 처리한다.
  • src/lib/wordsApi.ts 신설 — fetchWords(): GET /api/words를 apiFetch로 호출, 비-2xx면 Error throw, 성공 시 { words: WordProgress[] }(WordProgress는 src/lib/wordState.ts 기존 export 재사용 — m1/m2/nextReview만 필요, hanzi/pinyin/meaning은 홈 집계에 불필요).
  • src/lib/homeStats.ts(+ .test.ts) 신설 — computeHomeStats(words: WordProgress[], today: string) 순수 함수. getWordState로 각 단어를 분류해 reviewDue/learning/reviewScheduled 카운트, graduated = reviewDue + reviewScheduled, learningCandidates(학습 중 단어마다 m1<3·m2<3 각각 +1), sessionCount = Math.min(60, Math.min(reviewDue,60) + learningCandidates)를 반환. UI와 분리해 wordState.test.ts와 같은 방식으로 vitest 검증한다.
  • src/lib/retryQueue.ts 신설 — getRetryQueueLength()(localStorage 키 vocab-study:retry-queue를 JSON 배열로 읽고, 없거나 파싱 실패면 0)와 RETRY_QUEUE_CHANGED_EVENT(문자열 상수, window 커스텀 이벤트 이름) export. #18이 큐에 쓰기/비우기를 할 때마다 이 이벤트를 window.dispatchEvent해야 홈 화면이 같은 탭에서 즉시 반응한다 — PR 설명과 코드 주석에 "#18 구현 시 이 계약 재사용" 명시. (대안: storage 이벤트만 사용 — 그러나 storage는 다른 탭에서의 변경에만 발화하고 같은 탭 내 변경엔 발화하지 않아, 학습 세션 중 재전송 성공 후 홈으로 돌아왔을 때 즉시 반영되지 않는 문제가 생긴다. 두 이벤트를 함께 구독한다.)
  • src/lib/date.ts(+ .test.ts) 신설 — formatHomeDate(now = new Date()): Intl.DateTimeFormat("ko-KR", { timeZone: "Asia/Seoul", year:"numeric", month:"long", day:"numeric", weekday:"long" }). 로컬에서 직접 확인한 결과 이 옵션 조합이 정확히 "2026년 7월 17일 금요일"을 만든다(한국어 로케일은 월 "long"도 "7월"로 렌더링됨 — 영문처럼 "July"가 되지 않음).
  • HomeScreen.tsx 상태 모델: status: "loading" | "error" | "ready" + words/error/retryKey(재시도 버튼용)/retryQueueLength. 마운트 시 fetchWords() 호출 — App.tsx가 화면을 조건부 렌더링({screen === 'home' && <HomeScreen/>})하므로 홈을 벗어났다 돌아올 때마다 컴포넌트가 새로 마운트되어 "홈 진입 시마다 재조회" 요구사항을 별도 로직 없이 만족한다.
  • 에러 상태에서 기본 시작 버튼 숨김(판단 필요): design-prd는 에러 카드에 자체 "다시 시도" 버튼이 있다고만 하고, 하단 "학습 시작" 버튼의 노출 여부는 명시하지 않는다. 데이터가 없으면 세션 수를 계산할 수 없으므로, 에러 상태에서는 세션 수 줄과 시작 버튼을 모두 숨기고 에러 카드의 "다시 시도"를 유일한 CTA로 둔다(로딩 상태의 "세션 수 줄 숨김" 규칙과 일관). PR 설명에 이 해석을 명시해 리뷰에서 쉽게 뒤집을 수 있게 한다.
  • 미전송 인디케이터 표시 조건: retryQueueLength > 0이면 로딩/에러/정상 상태와 무관하게 항상 표시(재전송 여부는 단어 목록 로딩과 무관한 별개 관심사). window의 storage 이벤트와 RETRY_QUEUE_CHANGED_EVENT 둘 다 구독해 큐가 비면 즉시 사라지게 한다.
  • 스타일: src/index.css에 이어서 추가(레포에 CSS 모듈/CSS-in-JS 관례가 없고 기존 .app-*/.screen-placeholder도 전역 스타일이라 동일 관례를 따름). 새 키프레임 skeleton-pulse 1개, 클래스는 .home-* 접두사로 다른 화면과 충돌 방지.

5리스크

리스크영향완화책
로그인 화면(#11)이 미구현이라 실제 비밀번호 저장 플로우가 없음실행 상태에서 /api/words 호출이 저장된 비밀번호 없이 나가 항상 401(에러 상태)로 떨어짐devtools 콘솔에서 localStorage.setItem("vocab-study:app-password", "...")로 수동 세팅해 검증. 키 이름을 계약으로 문서화해 #11이 그대로 재사용하게 함
재시도 큐(#18)가 미구현이라 인디케이터를 채우는 쪽이 없음정상 플로우로는 인디케이터가 절대 표시되지 않아 완료 조건 검증 불가localStorage 키+이벤트 계약만 먼저 정의, 검증은 devtools로 vocab-study:retry-queue에 배열을 수동 주입해 확인
실 스프레드시트 HSK6급 탭에 데이터 행이 없음(2026-07-14 세팅 메모 기준)"실제 데이터로 카드 숫자 일치" 완료 조건을 빈 데이터로는 의미 있게 검증 불가#7 선례처럼 검증용 임시 행 추가 후 확인, 완료되면 즉시 삭제해 시트를 빈 상태로 복원. 실제 확인 결과 메모가 낡아 있었다 — 탭에 이미 학습 중 단어 46개가 있어 그대로 "학습 중" 실데이터로 쓰고, 복습 대기·복습 예약 2가지 상태만 임시 행으로 보강했다(검증 후 삭제, 46개로 원복)
에러 상태에서 "학습 시작" 버튼 노출 여부가 design-prd에 명시되지 않음구현자마다 다르게 해석하면 리뷰에서 재작업 발생설계 섹션의 판단(에러 시 시작 버튼·세션 수 줄 숨김, 에러 카드의 "다시 시도"만 노출)을 PR 설명에 명시
