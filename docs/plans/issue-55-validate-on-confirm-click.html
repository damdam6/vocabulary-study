#55기능 개선

등록 화면 검증을 '확인' 버튼 클릭 시에만 실행 — 붙여넣기 즉시 검증 제거

damdam6/vocabulary-study 브랜치 feat/55-validate-on-confirm-click ← main 라벨 enhancement 2026-07-19

1요약

등록 화면(#49)은 텍스트를 붙여넣거나 수정할 때마다 useMemo가 즉시 검증을 실행해 결과 테이블을 갱신한다. 이 이슈는 검증 실행 시점을 "텍스트 변경"에서 새로 추가하는 "확인" 버튼 클릭으로 옮긴다. 확인 후 텍스트를 다시 수정하면 기존 결과는 무효화되어, 재확인 전까지는 제출할 수 없다. 탭 선택만 바꿨을 때 즉시 재분류되는 기존 동작(#49 설계)은 그대로 유지한다.

2현황 분석

파일 헤더 주석이 현재 설계 의도를 명시한다: "검토는 버튼 없이 붙여넣는 즉시 반영된다 — 붙여넣기는 한 번의 onChange로 텍스트 전체가 들어오는 원자적 이벤트라 '입력 중 깜빡이는 오류' 문제가 없다" (src/screens/RegisterScreen.tsx:6-8). 이슈 본문은 이 전제를 "소유자 결정"으로 뒤집는다 — 타이핑뿐 아니라 붙여넣기 자체도 확인 버튼 없이는 검증되면 안 된다.

  • textarea의 value는 text state에 직접 바인딩되고, onChange가 매 입력마다 setText를 호출한다 (src/screens/RegisterScreen.tsx:49, 216).
  • parseResult는 useMemo(() => ..., [text, existingHanziInTab])로 text가 바뀔 때마다 validateRegistrationInput을 재실행한다 (src/screens/RegisterScreen.tsx:110-113).
  • 테이블/요약/중복배너/제출 버튼은 모두 parseResult 존재 여부로 렌더링되고 (src/screens/RegisterScreen.tsx:258-286), canSubmit도 parseResult?.ok === true를 요구한다 (src/screens/RegisterScreen.tsx:127-132) — 즉 현재는 "검증 결과가 있다"와 "텍스트가 바뀌었다"가 항상 동시에 참이라 구분되지 않는다.
  • validateRegistrationInput(text, existingHanziInTab) (src/lib/registerValidation.ts:50)는 순수 함수 — 부수효과 없이 언제 호출해도 동일 결과이므로, 호출 시점만 옮기면 되고 함수 자체는 변경할 필요가 없다.
  • 탭을 바꾸면 텍스트는 그대로 두고 분류(특히 중복)만 즉시 재계산되는 것이 기존 의도다 (src/screens/RegisterScreen.tsx:6-8 주석, docs/design-prd.md:129 두 번째 문장) — 이슈는 이 부분을 건드리지 않는다.
  • docs/design-prd.md:129 "검증 시점" 문장이 옛 동작("버튼 없이 붙여넣는 즉시 반영")을 명문화하고 있어, 코드만 고치면 문서가 stale해진다.
  • RegisterScreen.tsx에 대한 컴포넌트 테스트는 존재하지 않는다 (@testing-library/react 미설치, vitest.config.ts도 jsdom 미설정) — 회귀 대상은 src/lib/registerValidation.test.ts·src/lib/registerApi.test.ts 같은 순수 함수 테스트뿐이며, 둘 다 RegisterScreen을 임포트하지 않는다.

3설계

선택한 설계 — text(라이브 입력)와 별개로 confirmedText: string | null 스냅샷 state를 두고, "확인" 버튼 클릭 시에만 setConfirmedText(text)로 갱신한다. parseResult는 text 대신 confirmedText에 의존하도록 바꾸되 existingHanziInTab 의존성은 유지해, 탭 전환 시 자동 재분류는 그대로 살린다. isDirty = confirmedText !== null && text !== confirmedText로 "확인 이후 편집됨"을 판정하고, 테이블/요약/중복배너 렌더링과 canSubmit을 !isDirty로 게이팅한다. 이는 기존 중복-확인 로직이 쓰는 "서명(signature) 비교로 자동 재확인 요구" 패턴(src/screens/RegisterScreen.tsx:10-15 주석, acknowledgedDuplicateKey)과 동일한 파생-state 방식이라 일관적이고, 별도 리셋 useEffect가 필요 없다.

기각한 대안

  • 디바운스만 늘리기 — 여전히 텍스트 변경만으로 자동 실행되므로 "붙여넣기만으로는 검증 없음"이라는 완료 조건을 만족하지 못한다.
  • 탭 변경도 확인 필요하게 만들기 — 이슈 현상/기대 동작 모두 텍스트 입력만 언급한다. 탭-즉시반영 동작(#49 설계)까지 게이팅하면 범위를 벗어나고 불필요한 회귀 위험만 늘어난다.
  • 확인 결과를 별도 useEffect로 리셋 — isDirty 파생값 하나로 충분한데 굳이 effect를 추가하면 상태가 두 곳(effect 트리거 + state)으로 나뉘어 추적하기 어려워진다.

5리스크

리스크영향완화책
새 "확인" 버튼과 기존 중복-확인 배너의 "확인" 버튼(src/screens/RegisterScreen.tsx:275)이 같은 라벨을 공유해 사용자가 혼동할 수 있음 UX 혼란(치명적이지 않음) 배치(텍스트 영역 바로 아래 vs. 중복 배너 안)와 주변 문구로 구분; 필요 시 더 명확한 대체 라벨 고려
parseResult를 confirmedText 기반으로 바꾸는 과정에서 탭 전환 시 자동 재분류(#49 설계)를 실수로 깨뜨릴 위험 기존 동작 회귀 existingHanziInTab을 useMemo 의존성에 계속 포함하고, 수동 검증에 "확인 후 탭만 전환" 케이스 포함
isDirty 판정상 텍스트를 정확히 이전 확인값으로 되돌리면 재확인 없이 다시 유효 판정됨 낮음 — 완료 조건 위반 아님(오히려 합리적인 동작) 별도 처리 불필요, 현상 유지

6테스트 계획

  • npm test(vitest) — registerValidation.test.ts·registerApi.test.ts 등 기존 스위트 회귀 없음 확인(순수 함수 시그니처 무변경이라 그대로 통과 예상).
  • npm run lint(oxlint) — 신규/수정 코드 스타일 위반 없음.
  • npm run build(tsc -b && vite build) — 타입 에러 없음.
  • 수동 브라우저 검증(npm run dev, 등록 화면): (1) JSON 붙여넣기 직후 테이블/에러 미표시 (2) "확인" 클릭 시 테이블·요약·중복배너 표시 (3) 확인 후 텍스트 한 글자 수정 시 결과 무효화(테이블 숨김 또는 힌트) 및 제출 버튼 비활성 (4) 재확인 클릭 후 제출 가능 (5) 확인 후 탭만 전환했을 때 기존처럼 즉시 재분류(중복 갱신) — 회귀 없음.

7완료 조건

  • 붙여넣기/타이핑만으로는 검증이 실행되지 않는다 (테이블·에러 메시지 없음).
  • "확인" 버튼 클릭 시에만 파싱 + 기계 검토가 실행되고 결과 테이블이 표시된다.
  • 확인 후 텍스트를 수정하면 기존 결과가 무효화되고, 재확인 전에는 제출 버튼이 비활성 상태로 유지된다.
  • 탭 선택만 바꿨을 때(텍스트 미변경)는 기존처럼 즉시 재분류된다 — #49 설계 유지, 회귀 없음.
  • 기존 registerValidation/registerApi 테스트, lint, build가 회귀 없이 통과한다.
