#5기능 개선

구글 시트 API 연동 — 서비스 계정 인증 + read/write 유틸

damdam6/vocabulary-study 브랜치 feat/5-google-sheets-api-client ← main 라벨 enhancement 2026-07-15

1요약

Cloudflare Worker에서 구글 서비스 계정 키로 JWT(RS256)를 서명해 access token을 교환하는 인증 유틸과, Sheets API v4의 values.get / update / append를 감싸는 read/write 래퍼를 만든다. Node crypto 대신 Web Crypto API(crypto.subtle)로 서명하고, 토큰은 isolate 메모리에 캐시해 만료 전까지 재사용한다. 이후 모든 API 이슈(단어 목록 read, 학습 결과 write 등)가 이 유틸 위에 올라간다. 단, 이 브랜치는 main에서 분기됐는데 Worker 스캐폴딩은 아직 미머지 PR #2에만 있어, 시작 전 베이스 정리가 선행되어야 한다.

2현황 분석

⚠️ 선행 의존성: PR #2가 아직 머지되지 않았다. 현재 브랜치는 origin/main(커밋 1900de8, docs/PRD.md뿐)에서 분기됐지만, 이슈가 전제하는 Worker 스캐폴딩(worker/index.ts, wrangler.jsonc, Vite+React 셋업)은 전부 미머지 상태인 PR #2 브랜치 feat/1-project-stack-setup에만 존재한다. 이 워크트리에는 현재 docs/ 외에 아무 코드도 없다.

  • wrangler.jsonc: main: "worker/index.ts", nodejs_compat 플래그 켜짐, run_worker_first: ["/api/*"]. vars는 전부 주석 처리 상태(= SHEET_ID 미정의).
  • 시크릿: GOOGLE_SERVICE_ACCOUNT_KEY는 프로덕션에 등록돼 있으나 참조하는 코드가 0곳. 로컬 키 파일은 메인 체크아웃의 .secret/voca-502322-99c84e265c69.json에 있고, .gitignore가 .secret/·.dev.vars*를 이미 커버한다.

3설계

모듈 2개 + 설정 3곳. 인증과 시트 호출을 분리해 이후 이슈들이 sheets.ts만 import하면 되게 한다.

  • worker/lib/google-auth.ts — 토큰 발급 + 캐시.
    • GOOGLE_SERVICE_ACCOUNT_KEY(JSON 문자열)를 파싱해 client_email·private_key 추출. PEM(PKCS#8) 헤더/푸터 제거 → base64 디코드 → crypto.subtle.importKey("pkcs8", …, { name: "RSASSA-PKCS1-v1_5", hash: "SHA-256" }).
    • JWT 조립: header {alg:"RS256", typ:"JWT"}, claims {iss: client_email, scope: "https://www.googleapis.com/auth/spreadsheets", aud: "https://oauth2.googleapis.com/token", iat, exp: iat+3600}. base64url 인코딩 헬퍼 포함.
    • https://oauth2.googleapis.com/token에 grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer로 POST → access_token 수신.
    • 캐시: 모듈 스코프 변수 { token, expiresAt }. 만료 60초 전이면 재발급. 발급/캐시히트 시 console.log 한 줄씩 남겨 완료 조건 3을 검증 가능하게 함. 동시 요청 중복 발급 방지를 위해 in-flight Promise도 같이 캐시.
  • worker/lib/sheets.ts — Sheets API v4 래퍼.
    • getValues(env, range) → GET …/values/{range}, updateValues(env, range, values) → PUT (valueInputOption=RAW), appendValues(env, range, values) → POST :append (insertDataOption=INSERT_ROWS).
    • 범위 표기는 'HSK6급'!A1:G1처럼 탭명을 작은따옴표로 감싸고 전체를 encodeURIComponent — 한글 탭명이므로 인코딩 누락 시 400.
    • 비 2xx 응답은 status + 응답 body를 담은 에러로 throw — 이후 이슈들이 공통으로 쓸 에러 계약.
  • 설정: wrangler.jsonc의 vars에 SHEET_ID 추가(비밀 아님, 시트 ID는 URL에 노출되는 값) · 로컬용 .dev.vars에 GOOGLE_SERVICE_ACCOUNT_KEY 주입(gitignore 기커버) · npm run cf-typegen으로 Env 타입 재생성.
  • 기각한 대안: ① googleapis npm SDK — Node 전용 의존성이 많고 번들이 커서 Workers에 부적합, REST 직접 호출로 충분. ② nodejs_compat의 node:crypto 사용 — 동작은 하지만 이슈가 Web Crypto를 명시했고, 표준 API가 런타임 호환성 면에서 안전. ③ KV/캐시 API에 토큰 저장 — isolate 메모리 캐시로 충분하며 시크릿 파생물을 외부 저장소에 두지 않는 편이 낫다.

4작업 단계

  1. 베이스 정리 (선행). PR #2 머지 후 origin/main 리베이스가 정석. 머지를 기다리지 않으려면 feat/1-project-stack-setup 위로 리베이스하고 PR을 스택으로 연결 — 어느 쪽이든 사용자 결정 필요. git rebase (파일 변경 없음)
  2. 인증 유틸 작성: 키 JSON 파싱, PEM→CryptoKey import, base64url 헬퍼, JWT 서명, 토큰 교환, 만료 60초 버퍼를 둔 모듈 스코프 캐시(+in-flight Promise 공유), 발급/캐시히트 로그. worker/lib/google-auth.ts (신규)
  3. 시트 래퍼 작성: getValues/updateValues/appendValues + 범위 인코딩 헬퍼 + 비 2xx throw. 시그니처는 이후 이슈(단어 read, 학습 결과 write)가 그대로 쓸 수 있게 탭+범위를 인자로 받는다. worker/lib/sheets.ts (신규)
  4. 설정 반영: wrangler.jsonc에 vars.SHEET_ID 추가, .dev.vars 생성(로컬 전용, .secret/voca-502322-99c84e265c69.json 내용을 한 줄 JSON으로), npm run cf-typegen으로 Env에 SHEET_ID·GOOGLE_SERVICE_ACCOUNT_KEY 반영. wrangler.jsonc, .dev.vars (미커밋), worker-configuration.d.ts (재생성)
  5. 검증용 임시 라우트 추가: /api/dev/sheets-check — 헤더 행('HSK6급'!A1:G1) read → 테스트 셀(예: 'HSK6급'!A100) write → 즉시 재-read 후 결과 JSON 반환. 완료 조건 검증 후 커밋 전 제거(검증 증거는 PR 본문에 기록). worker/index.ts (임시 수정 후 원복)
  6. 검증 실행: wrangler dev에서 임시 라우트 연속 호출 — read 성공, write 후 시트 육안 확인 + 테스트 셀 원상복구, 로그에서 최초 1회만 토큰 발급이고 이후 캐시히트인지 확인. (코드 변경 없음 — 완료 조건 1·2·3 검증)
  7. 마무리: 임시 라우트 제거, npm run lint·npm run build(tsc 포함) 통과 확인, 커밋 및 PR 생성(검증 로그/스크린샷 첨부). worker/index.ts, worker/lib/*

PEM 파싱 실수 (JSON 이스케이프 \n, 헤더/푸터 변형)importKey 실패 또는 서명 오류로 invalid_grantJSON.parse가 \n을 실개행으로 복원함을 전제로 헤더/푸터·공백 제거를 방어적으로 구현, 실패 시 원인 구분 가능한 에러 메시지
한글 탭명 HSK6급 URL 인코딩 누락Sheets API 400 에러범위 인코딩을 sheets.ts 내부 헬퍼로 강제 — 호출자가 인코딩을 신경 쓸 수 없는 구조로
동시 요청 시 토큰 중복 발급 (thundering herd)불필요한 OAuth 왕복, 완료 조건 3 위반처럼 보일 수 있음in-flight Promise를 캐시에 함께 저장해 동시 호출이 같은 발급을 기다리게 함
isolate 교체 시 토큰 캐시 소실콜드 스타트마다 재발급 (기능상 문제 없음, 레이턴시 +1 왕복)정상 동작으로 수용 — 문서화만. KV 저장은 보안상 비권장 (설계 §3 대안 ③)
테스트 write가 실제 데이터 영역을 오염이후 단어 데이터와 충돌테스트 셀은 데이터 영역에서 먼 A100 사용, 검증 후 빈 값으로 원상복구

6테스트 계획

  • read: wrangler dev + curl localhost:8787/api/dev/sheets-check → 응답에 헤더 행 7개 값(한자/병음/뜻/…)이 그대로 옴 → 완료 조건 1.
  • write: 같은 라우트가 A100에 테스트 문자열 write → 브라우저에서 시트 열어 육안 확인 → 빈 문자열로 재-write해 원복 → 완료 조건 2.
  • 토큰 캐시: 라우트를 3회 연속 호출 → wrangler dev 로그에 "token issued"가 최초 1회, 이후 "token cache hit"만 찍히는지 확인 → 완료 조건 3.
  • 에러 경로: 존재하지 않는 탭명으로 read 시도 → 래퍼가 status·body 포함한 에러를 throw하는지 확인 (이후 이슈들의 에러 계약 검증).
  • 기존 동작 회귀: /api/health가 여전히 200 JSON을 반환하고, 임시 라우트 제거 후 npm run build(tsc -b 포함)·npm run lint가 깨끗한지 확인.
