커스텀 드롭다운 외부 탭 닫힘이 mousedown에만 의존 — 터치 환경에서 안 닫힐 수 있음

1요약

#56/PR #60에서 만든 커스텀 Dropdown의 바깥-탭 닫힘 리스너가 document의 mousedown 이벤트에만 의존한다. 이 앱은 모바일이 주 환경인데, 터치 입력은 브라우저가 만들어주는 합성 mousedown에 의존하는 구조라 일부 브라우저·제스처에서 바깥을 탭해도 패널이 닫히지 않을 수 있다(PR #60 리뷰 코멘트에서 지적). mousedown 리스너를 pointerdown으로 교체해 마우스·터치·펜을 단일 이벤트로 커버한다. 변경 범위는 리스너 하나뿐이고, ESC 닫힘·내부 클릭 유지·키보드 내비게이션 등 나머지 동작은 그대로 둔다.

2현황 분석

src/components/Dropdown.tsx:30-50 — 패널이 열려 있을 때만(if (!open) return) useEffect가 두 개의 document 리스너를 붙인다.

  • Dropdown.tsx:32-36 — handlePointerDown(event: MouseEvent): 이름은 handlePointerDown이지만 실제로는 mousedown 전용 핸들러다. 클릭 대상이 containerRef(트리거 버튼 + 패널을 감싸는 div) 밖이면 setOpen(false).
  • Dropdown.tsx:37-43 — keydown으로 Escape 처리, 별개 리스너라 이번 변경과 무관.
  • Dropdown.tsx:44 — document.addEventListener('mousedown', handlePointerDown).
  • Dropdown.tsx:47 — cleanup에서 동일하게 'mousedown'으로 제거.
  • 옵션 선택(Dropdown.tsx:132, <li onClick={() => selectOption(index)}>)과 트리거 토글(Dropdown.tsx:105)은 React 합성 onClick을 쓰는데, 이는 이미 모든 포인터 타입(마우스/터치/펜)에서 동작하므로 이번 이슈의 대상이 아니다 — 오직 바깥-탭 닫힘 리스너만 mousedown 원시 이벤트에 묶여 있다.
  • 이 프로젝트에는 아직 컴포넌트 레벨 테스트 인프라(jsdom/happy-dom, @testing-library/react)가 없다 — vitest.config.ts는 기본(node) 환경이고 기존 *.test.ts는 모두 순수 함수 단위 테스트다. PR #60도 같은 이유로 임시 하네스 + cmux 브라우저 자동화로 수동 QA만 하고 하네스를 삭제하는 방식을 택했다.

3설계

  • 채택: document.addEventListener('mousedown', …) → document.addEventListener('pointerdown', …)로 교체하고, 핸들러 파라미터 타입도 MouseEvent → PointerEvent로 바꾼다. PointerEvent는 MouseEvent를 확장하므로 event.target 기반 containerRef.current?.contains(...) 로직은 변경 없이 그대로 재사용된다. 이슈가 요구하는 "단일 리스너로 마우스·터치·펜 커버"에 정확히 대응하고, 변경 범위가 이벤트 이름 두 곳 + 타입 한 곳으로 최소화된다.
  • 기각 — mousedown + touchstart 이중 바인딩: PR #60 리뷰 코멘트가 대안으로 언급했지만, 이렇게 하면 터치 기기에서 touchstart와 뒤이은 합성 mousedown이 같은 탭에 대해 두 번 호출돼 방어 코드(예: 플래그, preventDefault)가 추가로 필요해진다. pointerdown 단일 리스너가 더 간단하고 이슈의 기대 동작과도 정확히 일치한다.
  • 기각 — mousedown과 pointerdown 동시 바인딩(구형 브라우저 폴백): 이 레포는 Vite + React 19 기반 최신 빌드 타깃이고 PointerEvent는 Safari 13+/모든 최신 모바일 브라우저에서 지원된다. 존재하지 않는 구형 브라우저 지원 요구를 가정한 폴백은 불필요한 복잡도만 늘린다(레포 관례상 불필요한 방어 코드 지양).
  • 키보드(ESC/화살표/Enter) 리스너와 포커스 관리(Dropdown.tsx:52-60)는 이번 변경과 완전히 독립적이라 손대지 않는다.

4작업 단계

  1. ✅ 바깥-탭 닫힘 핸들러의 이벤트 타입을 MouseEvent에서 PointerEvent로 변경 src/components/Dropdown.tsx:32
  2. ✅ 리스너 등록을 mousedown에서 pointerdown으로 변경 src/components/Dropdown.tsx:44
  3. ✅ cleanup의 리스너 해제도 동일하게 pointerdown으로 맞춤 src/components/Dropdown.tsx:47
  4. ✅ PR #60과 동일한 방식으로 로그인/Worker 없이 Dropdown만 마운트하는 임시 하네스(src/main.tsx 임시 교체)를 만들어 cmux 브라우저 자동화로 QA 후 원상복구. 데스크톱은 실제 click 자동화(마우스 pointerdown 경로)로, 모바일 터치는 WKWebView가 저수준 input touch 명령을 지원하지 않아(not_supported) pointerType: 'touch' PointerEvent를 직접 dispatch하는 방식으로 검증 — 두 경우 모두 바깥 탭 시 패널이 닫힘을 확인. 같은 세션에서 ESC 닫힘, 트리거 재클릭 시 토글(바깥 핸들러 간섭 없음), 방향키(↓↓)+Enter 선택(값 "탭 C"로 정상 반영)도 기존과 동일하게 동작함을 확인. 임시 파일, 커밋에 포함하지 않음
  5. ✅ npm test(17개 파일 158건 통과) · npm run lint(oxlint 통과) · npm run build(통과) 확인 전체 스위트

5리스크

리스크영향완화책
매우 오래된 브라우저(레거시 Safari/Android WebView 등)가 PointerEvent를 지원하지 않을 가능성 해당 환경에서 바깥-탭 닫힘이 동작하지 않음 이 앱의 실제 타깃은 최신 모바일 브라우저(Vite/React19 빌드 타깃과 일치) — 별도 폴백 없이 진행하고, 실사용 중 리포트되면 그때 대응
포인터 타입에 따라 pointerdown이 click보다 먼저 발생 → 트리거 버튼 토글과의 순서 상호작용 트리거를 눌러 닫을 때 이중 토글 등 예기치 못한 동작 가능성 트리거 버튼이 containerRef 내부에 있어 바깥-탭 핸들러가 아예 반응하지 않음(기존 mousedown 구조와 동일) — 현황 분석에서 확인, 하네스 QA에서 트리거 재클릭 케이스도 함께 확인
