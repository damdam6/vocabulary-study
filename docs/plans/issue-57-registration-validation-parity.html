단어 등록 검증 계층 정합화 — Worker 100건 한도·범위·성조 재검증 + 한자 범위 통일

1요약

등록 검증 계층 3곳에서 스펙(PRD §7.3, 플랜 §3·§6)과 코드 간 드리프트가 확인됐다. (1) Worker가 words 배열 100건 상한을 검사하지 않고, (2) 한자 유니코드 범위·병음 성조 부호 형식을 재검증하지 않아 클라이언트를 우회한 요청이 그대로 시트에 쓰이며, (3) schema_check.py의 한자 허용 범위가 클라이언트보다 넓어(CJK 확장 A 포함) 세 검증기 간 정의가 어긋나 있다. 이번 작업은 Worker 재검증을 스펙 수준으로 보강하고 한자 범위를 기본 블록(U+4E00–U+9FFF)으로 통일해, 플랜 §2가 명시한 신뢰 경계 원칙("2·3단계가 각각 재검증")을 코드에 맞춘다.

2현황 분석

등록 파이프라인은 3단계다: ① 추출(claude.ai, docs/registration-kit/schema_check.py로 자체 검증) → ② 등록 화면(src/lib/registerValidation.ts + pinyinValidation.ts) → ③ Worker(worker/routes/register.ts + worker/lib/register.ts). 플랜 §2는 "1단계 출력은 신뢰하지 않는다 — 2단계와 3단계가 각각 재검증"한다고 명시한다.

  • Worker 재검증(worker/lib/register.ts:13-38 parseRegisterWords)은 현재 배열 비어있지 않음(14행), 항목이 object인지, hanzi/pinyin/meaning 세 필드가 트림 후 빈 문자열이 아닌지, 배열 내 한자 중복(31-34행)만 검사한다. 배열 길이 상한, 한자 유니코드 범위, 병음 형식 검사는 전혀 없다.
  • 등록 화면(src/lib/registerValidation.ts:31)의 HANZI_RE = /^[一-鿿]+$/u는 실측 확인 결과 정확히 U+4E00–U+9FFF(기본 블록)와 일치한다 — 클라이언트는 이미 스펙대로다. 병음 형식은 pinyinValidation.ts의 isPinyinMatch(pinyin-pro 기반 한자-병음 의미 일치)가 대신한다 — 후보가 항상 성조 부호를 포함하므로 무성조·숫자성조 입력은 후보와 매치되지 않아 부수효과로 걸러진다.
  • 이 두 검증은 클라이언트에서만 돈다. worker/routes/register.ts는 Bearer 인증만 통과하면 POST /api/words/register를 직접 호출하는 어떤 요청도 받아들인다 — 등록 화면을 거치지 않고 curl 등으로 직접 호출하면 한자 범위·성조 형식 검사를 완전히 우회한다.
  • docs/registration-kit/schema_check.py:30의 HANZI_RANGES = ((0x4E00, 0x9FFF), (0x3400, 0x4DBF))는 CJK 확장 A(0x3400–0x4DBF)까지 허용한다. 이 검증기는 파이프라인 1단계(추출 직후, 사용자가 등록 화면에 도달하기도 전)에서 돈다 — 그래서 확장 A 문자가 섞인 배치도 여기서는 PASS로 통과했다가 2단계(클라이언트)에서야 비로소 막힌다.
  • words 배열 길이 상한은 코드 어디에도 없다(worker/routes/register.ts 전체에 .length 상한 검사 없음). PRD §7.3은 "words 배열 최대 100건, 초과 시 400"을 명시하지만(docs/PRD.md:226), 대량 배열도 전부 파싱돼 한 번의 updateValues 호출(worker/routes/register.ts:68-73)로 시트에 append된다.

3원인 분석

  • ① 100건 한도 미구현 — parseRegisterWords의 배열 단위 검사가 raw.length === 0 하나뿐이라 상한이 없다. PRD §7.3의 "words 최대 100건, 초과 시 400"이 코드로 옮겨지지 않은 단순 누락. 수정: MAX_REGISTER_WORDS = 100 상수를 두고 같은 위치(배열 단위 가드)에 상한 검사를 추가한다.
  • ② Worker 재검증 깊이 부족 — parseRegisterWords는 구조(타입·공백·중복)만 보고, 플랜 §3이 요구하는 두 스키마 규칙(한자 유니코드 범위, 병음 성조 부호 형식 — 숫자 표기 불가)은 전혀 재검증하지 않는다. 이 두 규칙은 지금 클라이언트(registerValidation.ts·pinyinValidation.ts)와 1단계(schema_check.py)에만 있다. 플랜 §2 신뢰 경계 원칙상 Worker가 최종 쓰기 지점이므로 이 결여는 실질적 방어선 구멍이다.
    설계 선택: schema_check.py의 병음 형식 검사(숫자 거부·허용 문자 집합·성조 부호 존재 요구)를 라이브러리 없이 TS로 이식한다 — 이슈 본문이 명시한 대로 "병음-한자 일치(pinyin-pro)는 클라이언트 전용 유지, Worker에 라이브러리 도입하지 않음"이 전제이므로, Worker가 검사하는 것은 병음 문자열의 "형식"이지 한자와의 "의미적 일치"가 아니다. 기각한 대안: Worker에 pinyin-pro를 그대로 도입 — 이슈 본문에서 명시적으로 배제.
  • ③ 한자 범위 정의 드리프트 — schema_check.py(1단계)만 CJK 확장 A를 추가로 허용해 클라이언트(2단계)보다 느슨하다. 결정(이슈 본문에 이미 명시): 기본 블록(U+4E00–U+9FFF)으로 통일 — 간체자는 전부 기본 블록 안에 있고, 확장 A는 고문헌·인명용 희귀 한자라 단어장에 등장하지 않는다. 좁은 쪽이 더 엄격한 방어선이므로 넓은 쪽(1단계)을 좁혀 맞춘다. 동시에 Worker(3단계)에도 같은 기본 블록 검사를 추가해야 세 검증기가 실제로 동일해진다.

4작업 단계

  1. MAX_REGISTER_WORDS = 100을 export하고, parseRegisterWords의 기존 배열 단위 가드(raw.length === 0 옆)에 상한 초과 시 null 반환을 추가한다. worker/lib/register.ts 완료 — 계획대로 구현.
  2. src/lib/registerValidation.ts의 HANZI_RE와 동일한 정규식을 Worker 쪽에 두고, parseRegisterWords의 항목 루프에서 트림된 hanzi가 이를 통과하지 못하면 null을 반환하도록 추가한다. worker/lib/register.ts (참조: src/lib/registerValidation.ts) 완료 — 계획대로 구현.
  3. schema_check.py의 병음 성조 부호 형식 검사(숫자 거부·허용 문자 집합·성조 부호 존재 요구)를 순수 함수 hasValidToneFormat로 이식해 같은 루프에서 호출한다. 새 의존성 추가 없음. worker/lib/register.ts (참조: docs/registration-kit/schema_check.py) 완료 — 계획대로 구현. pinyin-pro 등 새 의존성 없음.
  4. 스키마 위반 시 반환하는 400 오류 메시지를 새 제약(한자 범위·성조 형식·100건 상한 포함)까지 정확히 반영하도록 갱신한다 — 등록 화면 오류 배너가 부정확한 사유를 보여주지 않도록. worker/routes/register.ts 완료 — 계획대로 구현.
  5. 100건 경계(정확히 100건 통과 / 101건 거부), 한자 범위 밖(확장 A 문자·비한자 혼입), 병음 형식(숫자 성조·무성조·허용 외 문자) 각각에 대한 단위 테스트를 기존 파일의 describe 스타일대로 추가한다. worker/lib/register.test.ts 완료 — 8개 케이스 추가, 통과.
  6. POST /api/words/register 엔드투엔드로 같은 세 시나리오(100건 초과, 한자 범위 밖, 숫자 성조)가 400을 반환하는지 통합 테스트를 추가한다. worker/routes/register.test.ts 완료 — 3개 케이스 추가, 통과.
  7. HANZI_RANGES에서 CJK 확장 A 튜플을 제거해 기본 블록만 남기고, 설명 주석을 갱신한다. 수정 후 확장 A 문자 샘플로 수동 실행해 FAIL 확인. docs/registration-kit/schema_check.py 완료 — U+3400(㐀) 샘플 FAIL, 기본 블록 샘플 PASS 수동 확인.
  8. §3 필드 규칙 문장에 한자 유니코드 범위 결정(기본 블록 U+4E00–U+9FFF, 확장 A 제외 — 이슈 #57)을 명시해, 세 검증기가 실제로 가리키는 원본 문장을 남긴다. docs/plans/word-registration-system.md 완료 — 계획대로 구현.
  9. HANZI_RE 위에 플랜 §3을 가리키는 참조 주석 한 줄을 추가한다(값 변경 없음) — 세 구현 모두 참조 주석을 남긴다는 이슈 요구를 대칭적으로 충족. src/lib/registerValidation.ts 완료 — 계획대로 구현.
  10. npm test·npm run lint를 리포 루트에서 실행하고, python3 docs/registration-kit/schema_check.py를 확장 A 포함 샘플/기본 블록 정상 샘플 각각에 수동으로 돌려 FAIL/PASS를 확인한다. 검증 전용 — 코드 변경 없음 완료 — vitest 168/168, oxlint 0 issue, tsc -b 0 error, schema_check.py 수동 확인 통과.

5리스크

리스크영향완화책
새 Worker 검증이 실제 정상 등록 입력을 오탐 차단 등록 화면에서 원인 불명 400 오류 노출 기존 routes/register.test.ts 픽스처(经济/jīngjì 등 실사용 사례)로 회귀 확인, 정규식·문자 집합은 이미 검증된 schema_check.py·클라이언트 로직을 그대로 이식
클라이언트(pinyin-pro 의미 일치)와 Worker(정규식 형식 검사)의 판정 기준 불일치로, 클라이언트는 통과시켰는데 Worker가 막는 케이스 발생 등록 화면에서 정상 통과된 행이 제출 시점에 실패 Worker는 "형식"만 재검증(성조 부호 존재·숫자 표기 여부)하고 "한자-병음 의미적 일치"는 검사하지 않도록 스코프를 의도적으로 좁힘 — 클라이언트가 이미 허용한 정상 형식 병음이면 Worker 형식 검사도 통과
100건 상한이 실사용 대량 붙여넣기 패턴을 방해 사용자가 나눠서 여러 번 제출해야 함 PRD §7.3에 이미 명시된 기존 스펙 — 새 제약이 아님. 등록 화면 UX(예: 분할 안내) 변경은 이슈 범위 밖

7완료 조건

  • words 101건 이상 요청 시 400, 정확히 100건은 통과 (lib·route 테스트로 확인)
  • Worker가 한자 유니코드 범위(U+4E00–U+9FFF) 밖 문자·숫자 성조 표기 병음을 포함한 요청을 400으로 거부 (테스트로 확인)
