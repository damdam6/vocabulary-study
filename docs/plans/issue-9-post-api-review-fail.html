#9기능 개선

POST /api/review-fail 구현 — 복습 오답 간격 후퇴

damdam6/vocabulary-study 브랜치 feat/9-post-api-review-fail ← main 라벨 enhancement 2026-07-17

1요약

복습 대기 단어를 틀렸을 때 기록 없이 간격만 한 단계 후퇴시키는 POST /api/review-fail을 신규 구현한다. 현재는 이 엔드포인트가 아예 없다. 같은 요청 안에서 해당 탭의 A열 한자로 행을 재탐색하고, F열의 간격 사다리(1→3→7→14→30)를 한 단계 낮춰 오늘+후퇴간격|후퇴간격으로 덮어쓰며, D/E열과 G열 이후 타임스탬프는 절대 건드리지 않는다. 형제 이슈 #8(POST /api/answer)이 아직 착수 전이므로, 행 재탐색과 간격 사다리 계산을 이번에 재사용 가능한 형태로 먼저 만들어 둔다.

2현황 분석

현재 라우팅은 GET /api/words 하나뿐이다 (worker/index.ts:16-23). /api/* 요청은 isAuthorized가 먼저 걸러주므로(worker/index.ts:8-10, worker/lib/auth.ts) 새 라우트는 인증을 별도로 신경 쓸 필요가 없다.

  • worker/lib/sheets.ts: getValues/updateValues/appendValues 등 Sheets API v4 래퍼는 갖춰져 있다. 특정 셀 범위(예: F5:F5)만 RAW로 덮어쓰는 것도 updateValues로 바로 가능하다.
  • worker/lib/words.ts: parseWordRow가 A~F열 한 행을 WordEntry로 정규화한다(D/E 빈칸→0, F 빈칸→null 쌍). 다만 "탭+한자로 행 번호를 재탐색"하는 함수는 아직 없다 — GET /api/words는 탭 전체를 훑기만 하고 특정 행에 쓰지 않기 때문이다.
  • worker/lib/time.ts: formatSeoulDateTime은 YYYY-MM-DD HH:mm(시각 포함)만 만든다. F열에 필요한 "날짜만" 포맷과 "N일 더하기" 연산은 없다.
  • 간격 사다리(1→3→7→14→30) 계산 로직은 코드베이스 어디에도 없다.
  • 이슈 본문이 언급한 의존 이슈 #5(구글 시트 API 연동)는 이미 main에 병합됐다. 반면 #8(POST /api/answer)과 #12(간격 상태 머신 유틸)의 워크트리는 둘 다 main HEAD(02c6bb9)에서 커밋이 없어 아직 착수 전이다. 즉 "공유"할 기존 코드가 없으므로, 이번 이슈가 행 재탐색·간격 사다리 유틸을 먼저 만들고 #8이 그 위에 얹는 순서가 된다.
  • 테스트 프레임워크는 package.json에 아직 없다(oxlint만 존재) — 검증은 npm run dev(Cloudflare Vite Plugin으로 Worker까지 로컬 구동) + curl 수동 호출로 진행한다.

3설계

  • 단일 읽기로 행 탐색+파싱을 겸한다. words.ts에 findWordRow(env, tab, hanzi)를 추가해 WORD_ROW_RANGE("A2:F") 전체를 한 번 읽으며 A열이 일치하는 행을 찾고, 그 자리에서 parseWordRow로 파싱까지 끝내 { rowNumber, entry }를 반환한다(못 찾으면 null). "행 번호만 찾는 함수 + 행 내용 다시 읽는 함수"로 나누면 Sheets API 호출이 두 번 되므로 합쳤다 — 이 저장소는 비밀번호 해시 캐싱까지 신경 쓸 만큼 성능에 민감한 편이다(895ef4b). 이 함수는 #8이 D/E 증가·G열 append에 그대로 재사용할 수 있는 형태로 설계한다.
  • 간격 사다리는 worker/lib/interval.ts 새 모듈로 분리한다. REVIEW_INTERVAL_LADDER = [1,3,7,14,30] 상수와 이번 이슈에 필요한 stepBack(currentInterval)만 구현한다. #8이 필요로 할 상승(stepUp) 함수는 #8 착수 시점에 이 모듈에 추가하도록 상단 주석으로 안내만 해두고 미리 만들지는 않는다(아직 쓰는 곳이 없는 코드를 미리 짜지 않기 위함).
  • stepBack은 "정확히 사다리 위 값"이 아니어도 안전하게 동작해야 한다. 현재 간격보다 크지 않은 가장 큰 사다리 값을 "현재 위치"로 보고 그 앞 칸을 반환한다(하한은 인덱스 0=1일). 정상 케이스(1/3/7/14/30)뿐 아니라 시트를 수동으로 건드려 간격이 사다리 밖 값이 된 경우도 항상 낮은 쪽으로만 움직여 안전하다.
  • F열이 비어 있거나 형식이 깨진 상태로 호출되면(정상 흐름에서는 발생하지 않아야 함) 최소 간격(1일)로 폴백한다. parseWordRow가 이미 "형식이 깨지면 throw 대신 빈 상태로 취급"하는 방침(words.ts:35 주석)을 갖고 있어 같은 철학을 따른다 — 404 대신 조용히 가장 보수적인 값으로 처리해 사이트가 죽지 않게 한다.
  • 날짜 연산은 서울 달력 날짜를 UTC 앵커로 취급해 계산한다. Asia/Seoul은 DST가 없는 고정 UTC+9라, "오늘"의 y-m-d를 뽑아낸 뒤 그 값을 Date.UTC(y, m-1, d)로 취급하고 일 단위로 더해도 타임존 버그 없이 정확하다. time.ts에 날짜 전용 포맷터(formatSeoulDate)와 addDays를 추가한다 — #8의 "내일"(졸업)·"오늘+새간격"(복습 정답)도 같은 유틸을 쓰게 된다.
  • 응답은 GET과 동일한 단어 객체를 반환한다. PRD §7.3은 POST /api/answer는 "응답: 갱신된 단어 객체"라고 명시하지만 review-fail 항목에는 응답 형식이 없다. 형제 엔드포인트와의 일관성, 그리고 프론트가 재조회 없이 바로 로컬 상태를 갱신할 수 있다는 이점 때문에 같은 규약(WordEntry 형태)을 채택한다. 이미 파싱해 둔 entry에 새 nextReview/interval만 덮어써 반환하면 되므로 추가 API 호출도 없다.
  • 없는 tab+hanzi 조합은 404. PRD §4.2는 모든 쓰기 작업 공통 규약으로 "행 재탐색, 없으면 실패"를 요구하고 §7.3의 POST /api/answer가 이를 404로 명시한다. 이슈 #9 본문의 완료 조건에는 빠져 있지만 같은 규약이 적용되어야 하므로 포함한다.

4작업 단계

  1. 서울 기준 날짜 전용 포맷터 formatSeoulDate(date)(YYYY-MM-DD)와 날짜 덧셈 addDays(dateStr, days)를 추가한다. 기존 formatSeoulDateTime과 같은 Intl.DateTimeFormat 포매터를 재사용하되 시/분 파트를 뺀다. worker/lib/time.ts
  2. worker/lib/interval.ts를 신규 생성해 REVIEW_INTERVAL_LADDER = [1,3,7,14,30]과 stepBack(currentInterval)을 구현한다. 사다리 값이 아닌 입력에도 안전하게 동작하도록(설계 참고) 처리하고, #8이 상승 함수를 추가할 위치를 주석으로 안내한다. worker/lib/interval.ts (신규)
  3. worker/lib/words.ts에 findWordRow(env, tab, hanzi)를 추가한다: WORD_ROW_RANGE로 탭 전체를 읽고 A열이 일치하는 행을 찾아 { rowNumber, entry }(헤더 오프셋 +2) 또는 null을 반환한다. 모듈 상단 주석에 #9(및 향후 #8)가 이 함수를 공유한다고 갱신한다. worker/lib/words.ts
  4. worker/routes/review-fail.ts를 신규 생성한다: 요청 바디에서 tab/hanzi 파싱(문자열 아니거나 없으면 400) → findWordRow로 탐색(없으면 404) → 현재 간격(entry.interval ?? 1)에 stepBack 적용 → formatSeoulDate(new Date()) + addDays로 새 날짜 계산 → F{rowNumber}:F{rowNumber}만 updateValues로 덮어쓰기(RAW, D/E/G열 미접근) → 갱신된 entry를 Response.json으로 응답. worker/routes/review-fail.ts (신규)
  5. worker/index.ts에 POST /api/review-fail 라우팅을 추가한다. handleGetWords와 동일하게 try/catch로 감싸 예외 시 500 + console.error 로깅. worker/index.ts

5리스크

리스크영향완화책
시트를 수동 편집해 F열 형식이 깨지거나 사다리 밖 값이 된 상태에서 호출후퇴 폭을 잘못 계산할 수 있음stepBack이 "현재 값보다 크지 않은 가장 큰 사다리 칸"을 기준으로 항상 안전하게(더 낮은 쪽으로만) 움직이도록 구현. F 자체가 없으면 최소 간격(1일)로 폴백
존재하지 않는 tab/hanzi 조합으로 호출엉뚱한 시트에 쓰기 시도findWordRow가 null이면 즉시 404, 쓰기 자체를 하지 않음
#8(POST /api/answer)이 아직 미착수라 이번에 만드는 interval.ts/findWordRow가 #8 요구사항과 어긋날 가능성#8 작업 시 리팩터링 필요두 함수 모두 방향성 없는 범용 인터페이스로 설계(행 조회는 읽기 전용 반환, 사다리는 상수+단일방향 함수)해 확장 시 시그니처 변경 없이 stepUp만 추가하면 되도록 함
실제 HSK6급 탭은 라이브 데이터라 테스트로 오염시키면 안 됨진행 데이터 훼손_test 같은 임시 탭(언더스코어 접두 — GET 목록에는 안 잡히지만 review-fail은 탭 이름 제약이 없어 그대로 테스트 가능)에서 curl 테스트 후 행 삭제

6테스트 계획

  • npm run dev로 로컬 Worker 구동(Cloudflare Vite Plugin이 /api/*까지 처리). .dev.vars에 SHEET_ID·GOOGLE_SERVICE_ACCOUNT_KEY·APP_PASSWORD 필요(서비스 계정: voca-dam@voca-502322.iam.gserviceaccount.com, 키 파일 .secret/).
  • 시트에 임시 탭(예: _test)을 만들고 졸업 상태 행 하나 추가: D=3, E=3, F=2026-07-17|7, G에 임의 타임스탬프 1~2개.
  • curl -X POST http://localhost:5173/api/review-fail -H "Authorization: Bearer <APP_PASSWORD>" -H "content-type: application/json" -d '{"tab":"_test","hanzi":"经济"}' 호출 → 응답 interval: 3, nextReview가 오늘+3일인지 확인.
  • 시트를 열어 D/E열 값과 G열 이후 타임스탬프가 호출 전과 동일한지, F열만 오늘+3일|3으로 바뀌었는지 육안 확인.
  • F열을 임의날짜|1로 세팅한 뒤 다시 호출 → 응답 interval이 여전히 1(하한 유지), nextReview는 오늘+1일로 갱신됐는지 확인.
  • 존재하지 않는 hanzi로 호출 → 404 확인.
  • Authorization 헤더 없이 호출 → 기존 인증 미들웨어가 401을 그대로 반환하는지 확인(회귀 없음).
  • 테스트에 사용한 임시 탭/행 정리.

7완료 조건

  • 호출 시 F열만 변경되고 D/E열, G열 이후 타임스탬프는 불변
  • 이미 1일 간격 상태에서 오답 시 간격 1일 유지(날짜만 오늘+1로 갱신, 간격 값은 그대로)
  • 존재하지 않는 tab+hanzi 조합으로 호출 시 404
  • 응답은 갱신된 단어 객체(GET /api/words와 동일한 WordEntry 형태)로 반환
