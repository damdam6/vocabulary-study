1요약

#15 학습 셸이 머지되면서 모드 2의 뼈대(채점 규칙·오답 결과 화면·입력 속성·셸과의 진행 계약)는 플레이스홀더 Mode2Card.tsx에 이미 구현되어 있다. 남은 일은 design-prd §4.3의 실제 디자인으로 마감하는 것: 오답 결과 화면의 fadeup 등장·"오답" 타이틀 shake, 화면별 타이포 크기(출제 뜻 27px / 정답 한자 50px / 병음 19px / 뜻 21px), 목업 기준 레이아웃 간격이다. 더불어 채점 규칙(트림 후 A열 정확 일치, 빈 입력 오답)을 순수 함수로 추출해 vitest로 고정한다 — 컴포넌트 테스트 인프라(jsdom) 없이 완료 조건 1을 자동화 테스트로 커버하는 저비용 경로다.

2현황 분석

셸(#15)과 플레이스홀더가 이미 담당 중인 것:

  • 진행 템포는 셸 소유: 모드 2 정답 → O 글리프 오버레이 + 즉시 다음 문제(src/screens/StudyScreen.tsx:96-97), 오답 → 카드가 결과 화면을 띄운 채 머물고(src/screens/StudyScreen.tsx:90-94) "다음" 버튼의 onProceed(src/screens/StudyScreen.tsx:100)로만 진행. 이슈의 "정답 즉시 진행 / 오답만 결과 화면" 계약이 셸 레벨에서 이미 성립.
  • 채점 로직: src/screens/Mode2Card.tsx:24-34 — value.trim() === word.hanzi 정확 일치, 빈 입력 = 오답. PRD §5.2와 일치하나 테스트가 없다(스크린 컴포넌트라 vitest node 환경에서 실행 불가).
  • 입력 속성: src/screens/Mode2Card.tsx:63-76 — autoFocus, autoComplete/autoCorrect/autoCapitalize="off", spellCheck={false}, enterKeyHint="done", lang="zh-Hans", placeholder "汉字" 모두 충족. Enter 제출은 form onSubmit으로 동작.
  • 오답 결과 화면: src/screens/Mode2Card.tsx:36-55 — "오답" + 정답 한자·병음·뜻 + 입력했었다면 "내 답" 취소선. 마크업 구성은 목업(docs/design/prototype.dc.html:124-136)과 동일 구조.
  • 폰트/애니메이션 기반: Noto Sans KR/SC 웹폰트는 index.html:14-25에서 비차단 로딩, 한자 폰트는 :lang(zh-Hans) 셀렉터(src/index.css:40-42) 한 곳으로 수렴. shake·fadeup keyframes도 이미 정의됨(src/index.css:108-132).

design-prd §4.3·목업 대비 남은 갭 (CSS는 전부 "플레이스홀더 최소 스타일" 블록 src/index.css:325-454):

  • 오답 결과 카드에 등장 fadeup 0.3s 없음 (목업 prototype.dc.html:125).
  • "오답" 타이틀에 shake 0.4s 없음 — 이슈 기대 동작에 명시된 항목 (.mode-result-title, src/index.css:379-383).
  • 타이포 크기가 모드 1 뒷면 공용값: 정답 한자 44px↔사양 50px(src/index.css:357-360), 병음 20px↔19px, 뜻 23px↔결과 21px·출제 27px(§4.3 "뜻 카드(27px)", 목업 prototype.dc.html:116).
  • 레이아웃 간격이 목업과 다름: 목업은 카드-폼 사이 18px·폼 내부 12px·결과 카드 내부 gap 11px, 현재는 .mode-area gap 14px 균일(src/index.css:326-333).

충돌 요인: 모드 1 실 UI(#16)가 같은 .mode-card* 클래스군을 재작업할 예정. feat/16-quiz-mode1-ui 브랜치는 현재 main과 동일(커밋 없음)이라 지금 충돌은 없다.

3설계

  • 마크업 구조는 유지, 스타일만 §4.3로 마감: 플레이스홀더의 JSX 구조(출제: form 안 카드+입력+제출 / 결과: 카드+다음 버튼)가 목업과 이미 동일하므로 컴포넌트 구조 변경 없이 클래스 추가로 해결한다. 셸과의 계약(onJudged 1회 호출, 오답 시 머무름, key 리마운트)은 그대로.
  • 모드 2 전용 스타일은 모디파이어로 스코프: 기존 컨벤션(.login-input--error, .study-chip--review)대로 .mode-card--result(fadeup·gap 11px)를 카드에 붙이고, 결과 화면 타이포는 .mode-card--result .mode-card-hanzi { font-size: 50px }식 컨텍스트 셀렉터로 오버라이드한다. 출제 화면 뜻은 .mode-card-meaning--question(27px/600). 공용 기본값(44/20/23px)은 건드리지 않는다 — #16이 모드 1 쪽을 재작업할 때 충돌 면적을 최소화하기 위함. (기각안: .mode2-* 신규 클래스군 — 공용 카드 스타일 중복 정의가 생겨 기각.)
  • 채점을 순수 함수로 추출: gradeMode2(input, hanzi)를 src/lib/studySession.ts(학습 모델 도메인, PRD §5.2)에 추가하고 컴포넌트는 호출만 한다. jsdom/testing-library 의존성 추가 없이(현재 devDeps에 없음, 리포는 lib 단위 테스트 컨벤션) 채점 세 케이스 + 병음·부분일치·번체 불허를 vitest로 고정한다.
  • 정오 피드백 오버레이·기록 API는 손대지 않음: §4.4 오버레이(src/index.css:293-323)와 fire-and-forget 기록(StudyScreen.tsx:53-69)은 #15에서 완결.
