기록 유실 방지 — POST /api/answer 실패 시 localStorage 재시도 큐

damdam6/vocabulary-study 브랜치 feat/18-answer-retry-queue ← main 라벨 enhancement 2026-07-18

1요약

POST /api/answer가 네트워크 등으로 실패하면 요청 페이로드를 localStorage 재시도 큐에 적재하고, 앱 로드 시와 다음 API 호출 성공 시점에 FIFO로 재전송한다(PRD §10). 큐 계약(저장 키·변경 이벤트)과 홈 화면 미전송 인디케이터는 #13에서 이미 선구현되어 있으므로, 이번 작업은 큐 본체(적재·재전송)와 StudyScreen 배선, 재전송 트리거 배선이 전부다. 타임스탬프는 최초 판정 시각을 유지하고, 재전송 실패분은 큐에 그대로 남긴다.

2현황 분석

  • 기록 유실 지점: src/screens/StudyScreen.tsx:53-69의 fireEffect가 문제 단위 fire-and-forget으로 postAnswer를 발사하고, 실패는 .catch(() => {})로 무시한다 — 주석에도 "#18이 이 catch에 배선될 예정"이라고 명시되어 있다.
  • 큐 계약(읽는 쪽) 선구현: src/lib/retryQueue.ts에 저장 키 RETRY_QUEUE_STORAGE_KEY("vocab-study:retry-queue", JSON 배열), 같은 탭 변경 알림용 커스텀 이벤트 RETRY_QUEUE_CHANGED_EVENT, getRetryQueueLength가 이미 있다. #18은 이 키에 저장하고 추가·제거 시마다 이벤트를 발화해야 한다는 계약이 파일 헤더에 박혀 있다.
  • 홈 인디케이터 완료: src/screens/HomeScreen.tsx:52-68, 119-124가 큐 길이를 구독(storage 이벤트 + 커스텀 이벤트)해 "● 미전송 기록 n건 · 연결되면 자동 저장"을 이미 렌더링한다. 이벤트 발화 계약만 지키면 UI 추가 작업은 없다.
  • 전송 페이로드: src/lib/api.ts:83-89의 AnswerRecord(tab·hanzi·mode·timestamp·isReview)가 그대로 큐 항목 형태가 된다 — JSON 직렬화 가능하고 timestamp가 이미 판정 시각(Asia/Seoul YYYY-MM-DD HH:mm, src/lib/time.ts)이다.
  • 재전송 트리거 훅 부재: "다음 API 호출 성공 시점" 감지 수단이 없다. src/lib/api.ts:23-28의 setUnauthorizedHandler가 같은 성격(모듈 레벨 콜백 등록)의 기존 패턴이다. 앱 로드 시점 훅은 src/App.tsx:19-22의 마운트 useEffect가 자리다.

3설계

모듈 구도 — 순환 import 없이 세 지점을 배선한다:

  • retryQueue.ts가 큐 본체를 소유: enqueueAnswer(record)(읽기→append→저장→이벤트 발화), flushRetryQueue()(재전송). api.ts의 postAnswer·AnswerRecord를 import한다.
  • api.ts에 성공 알림 훅 추가: setApiSuccessHandler(handler) — apiFetch가 response.ok일 때 호출. setUnauthorizedHandler 패턴의 미러. api.ts는 retryQueue를 import하지 않는다(순환 차단).
  • App.tsx가 배선: 마운트 시 setApiSuccessHandler(flushRetryQueue) 등록 + flushRetryQueue() 1회 호출(앱 로드 시 재전송). 이러면 홈의 fetchWords 성공, 학습 중 postAnswer 성공 등 모든 API 성공이 재전송 기회가 된다 — 이슈의 "다음 API 호출 성공 시점"을 문자 그대로 충족.
  • StudyScreen.tsx의 fireEffect: AnswerRecord를 변수로 만들어 postAnswer에 넘기고, .catch에서 같은 객체를 enqueueAnswer로 적재 — 타임스탬프가 자동으로 최초 판정 시각으로 보존된다.

flush 알고리즘 (FIFO, 실패 시 중단, 재진입 가드):

  • 모듈 레벨 in-flight 플래그로 재진입 차단 — flush 중의 postAnswer 성공이 success 핸들러를 다시 부르므로 가드가 없으면 재귀한다. 가드에 걸리면 즉시 no-op 반환.
  • 루프: 큐를 읽어 비어 있으면 종료 → queue[0]을 원본 그대로 postAnswer → 성공 시 큐를 다시 읽어 맨 앞 항목 제거·저장·이벤트 발화 → 실패 시 즉시 중단(항목 유지). 네트워크가 죽어 있으면 나머지도 실패할 것이므로 첫 실패에서 멈추는 게 낭비가 없다.
  • flush 중 새로 실패한 항목은 enqueueAnswer가 뒤에 append하므로(제거는 항상 재읽기 후 맨 앞) 유실되지 않는다. flush의 postAnswer 응답(갱신 단어)은 세션 문맥이 없으므로 무시한다.
  • localStorage 파손 JSON은 빈 큐로 취급(기존 getRetryQueueLength와 동일 방침), enqueueAnswer의 저장 실패(쿼터 등)는 try/catch로 삼킨다 — 학습 진행을 막지 않는 원칙(§6.2)이 우선.

기각한 대안:

  • apiFetch 내부에서 직접 flushRetryQueue() 호출 — retryQueue↔api 순환 import가 생기고 api.ts가 큐 존재를 알게 됨. 콜백 등록으로 방향을 한쪽으로 유지.
  • StudyScreen의 postAnswer 성공 시에만 flush — 홈 fetchWords 성공(네트워크 복구 후 홈 재진입) 시 재전송 기회를 놓쳐 스펙("다음 API 호출 성공 시점")보다 좁다.
  • postReviewFail 실패도 큐잉 — PRD §10과 이슈가 POST /api/answer만 명시. review-fail은 멱등하지 않은 간격 후퇴라 재전송 의미론이 다르므로 비범위로 남긴다.

4작업 단계

  1. ✅ retryQueue.ts에 큐 본체 구현 — 내부 read/write 헬퍼, enqueueAnswer(record: AnswerRecord)(append+이벤트), flushRetryQueue()(재진입 가드, FIFO 재전송, 성공 항목만 맨 앞 제거+이벤트, 실패 시 중단). 파일 헤더 주석을 "구현 완료" 상태로 갱신. src/lib/retryQueue.ts
  2. ✅ api.ts에 성공 알림 훅 추가 — setApiSuccessHandler 등록 함수와 apiFetch의 response.ok 분기에서 호출. setUnauthorizedHandler(api.ts:23-28) 패턴 그대로. src/lib/api.ts
  3. ✅ App.tsx 배선 — 마운트 useEffect에서 setApiSuccessHandler(flushRetryQueue) 등록(+cleanup에서 해제) 및 앱 로드 시 flushRetryQueue() 1회 호출. src/App.tsx
  4. ✅ StudyScreen 배선 — fireEffect의 answer 분기에서 AnswerRecord를 변수로 추출해 postAnswer와 실패 시 enqueueAnswer에 공유. "#18 배선 예정" 주석 정리. src/screens/StudyScreen.tsx
  5. ✅ 단위 테스트 — retryQueue.test.ts 신규: enqueue 적재·직렬화·이벤트 발화 / flush FIFO 순서·원본 타임스탬프 보존·성공 항목 제거·첫 실패 중단(잔여 유지)·빈 큐 no-op·재진입 가드 / 파손 JSON 허용. api.test.ts에 success 핸들러 호출(200에서 호출, 401·500에서 미호출) 케이스 추가. node 환경이므로 localStorage·fetch·window 이벤트는 기존 스타일(api.test.ts:14-22)대로 스텁. src/lib/retryQueue.test.ts, src/lib/api.test.ts
  6. ✅ 검증 — npm test(76개)·npm run lint·npm run build 그린. E2E는 fetch 오버라이드 대신 dev 서버 중단으로 실제 네트워크 실패를 재현: 서버 다운 상태에서 정답 처리(以~为主, 10:33) → 큐 적재·홈 인디케이터 "미전송 기록 1건" 확인 → 서버 복구·재로드(10:35) → 큐 비움·인디케이터 제거 → 시트 행 m1=1, G열="2026-07-19 10:33|m1" 확인(최초 시각 보존·중복 없음). npm run dev, 구글 시트

5리스크

리스크영향완화책
서버는 기록에 성공했으나 응답이 유실된 경우 재전송으로 시트에 중복 행기록 1건이 2행으로 적힘PRD §10이 유실 방지를 우선하는 트레이드오프로 수용. 시트 특성상 수동 정리 가능. 리스크로 명시만.
flush 중 401 수신 — apiFetch가 비밀번호 삭제·로그인 복귀flush 중단의도된 동작: 실패 항목은 큐에 유지되고, 재로그인 후 다음 API 성공 시 자동 재전송.
영구 실패 페이로드(예: 서버가 400을 계속 반환)가 큐 맨 앞을 점유뒤 항목까지 재전송 지연이슈 스펙이 "재전송 실패 시 큐 유지(버리지 않음)"를 명시 — v1은 스펙대로 유지하고 비고로 남김.
flush의 postAnswer 성공 → success 핸들러 → flush 재귀무한 루프모듈 레벨 in-flight 가드로 재진입 no-op + 재진입 가드 단위 테스트.
localStorage 쿼터 초과·파손 JSON적재 실패·큐 읽기 실패enqueue는 try/catch로 삼켜 학습 진행 보호, 파손 JSON은 빈 큐 취급(기존 getRetryQueueLength와 동일 방침).

  • 기존 회귀: npm test 전체 그린 (기존 api·homeStats·sessionQueue·studySession 등), npm run lint, npm run build(tsc -b 포함).

7완료 조건

  • 네트워크 실패 상태에서 정답 처리 시 요청이 localStorage 큐(vocab-study:retry-queue)에 쌓인다
  • 네트워크 복구 후 앱 재로드 시(및 다음 API 호출 성공 시) 큐가 재전송되어 시트에 기록된다
  • 재전송된 기록의 타임스탬프(G열)가 최초 판정 시각이다 — 재전송 시각으로 덮어쓰지 않음
  • 성공 전송된 항목은 큐에서 제거되어 중복 기록되지 않고, 재전송 실패분은 큐에 유지된다
  • 큐에 항목이 있을 때 홈 인디케이터가 표시되고, 큐가 비면 즉시 제거된다 (같은 탭 변경도 커스텀 이벤트로 반영)
  • 학습 중에는 어떤 실패 표시도 하지 않는다 — 전송 실패가 진행을 막지 않는 기존 원칙 유지
  • npm test·npm run lint·npm run build 그린
