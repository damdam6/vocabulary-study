1요약

PRD §6.1의 세션 큐 구성 규칙을 순수 함수 모듈 src/lib/sessionQueue.ts로 구현한다. 복습 대기 단어 전부(초과 시 복습일 오래된 순 60개)를 먼저 담고, 남은 슬롯에 학습 중 단어를 총 정답 수 내림차순으로 채운 뒤(단어당 최대 2문제), 같은 단어의 문제가 연달아 나오지 않도록 셔플한다. 큐 구성은 PRD §7.3에 따라 클라이언트 책임이므로 src/lib에 두고, 기존 상태 판정 유틸(getWordState)을 재사용한다. UI 연결은 #15의 몫이고 이 이슈는 로직+테스트까지다.

2현황 분석

세션 큐 구성 로직은 현재 어디에도 없다. 관련 기반은 모두 준비된 상태다:

  • 상태 판정 유틸(#12 완료) — src/lib/wordState.ts:36의 getWordState(word, today)가 단어를 learning / reviewDue / reviewScheduled로 분류한다. 주석(src/lib/wordState.ts:2-4)에 "학습 세션 큐 구성에서 재사용"이 명시돼 있다. getSeoulToday(src/lib/wordState.ts:17)로 "오늘"을 계산한다.
  • 데이터 형태 — GET /api/words(#7 완료)가 WordEntry(worker/lib/words.ts:11-20: tab, hanzi, pinyin, meaning, m1, m2, nextReview, interval)를 시트 행 순서 그대로 배열로 반환한다. 즉 배열 인덱스가 곧 "시트 상 순서"다. nextReview는 YYYY-MM-DD zero-padded 문자열 또는 null로 정규화된다(worker/lib/words.ts:83-93).
  • 소비처 — src/screens/StudyScreen.tsx는 플레이스홀더뿐이다("실제 학습 화면은 #15·#16·#17에서 구현"). 이 이슈는 큐를 만드는 함수까지만 책임지고, 세션 진행·오답 재삽입(§6.2)은 #15가 이 모듈의 출력을 받아 구현한다.
  • 모드 표기 — POST /api/answer는 mode: "m1" | "m2"를 받는다(PRD §7.3). 큐 항목의 모드 표기도 이와 일치시켜야 #15에서 변환 없이 전달된다.
  • 데이터 이상 케이스 — 졸업했는데 nextReview: null인 단어는 getWordState가 의도적으로 reviewDue로 취급한다(src/lib/wordState.ts:40-48). 큐의 "복습일 오래된 순" 정렬에서 이 null을 어디에 둘지 결정이 필요하다.

3설계

API — 제네릭 순수 함수 하나:

  • buildSessionQueue<T extends WordProgress>(words: readonly T[], today: string, rng?: () => number): SessionQuestion<T>[]
  • SessionQuestion<T> = { word: T; mode: "m1" | "m2"; isReview: boolean } — mode는 POST /api/answer 요청 형식과 동일, isReview는 §5.3(복습 오답 처리 분기)에 필요.
  • 제네릭으로 두면 WordEntry 타입을 src에 중복 정의하지 않고도 호출자가 넣은 전체 객체를 그대로 돌려받는다 — wordState.ts가 WordProgress 최소 필드 방식을 쓴 것과 같은 결. 단, 셔플 분산의 "같은 단어" 식별에는 tab + hanzi 키가 필요하므로 T extends WordProgress & { tab: string; hanzi: string }로 제약한다.
  • rng 주입(기본 Math.random)으로 모드 무작위 선택·셔플을 테스트에서 결정론적으로 만든다.
  • SESSION_CAP = 60 상수를 export — #15의 오답 재삽입("60문제 상한 내에서")도 같은 상수를 쓴다.

알고리즘 (PRD §6.1 그대로 4단계):

  • ① 분류: getWordState(word, today)로 전 단어를 분류. reviewScheduled는 제외.
  • ② 복습 대기: nextReview 오름차순(오래된 순) 정렬 후 앞 60개. null(데이터 이상)은 가장 오래된 것으로 취급해 맨 앞 — getWordState의 "눈에 띄게 한다" 방침과 일치. 동률은 시트 순서 유지(JS toSorted는 stable). 단어당 1문제, 모드는 rng()로 m1/m2 균등 선택, isReview: true.
  • ③ 학습 중: 총 정답 수(m1+m2) 내림차순, 동률은 시트 순서(stable sort). 정렬된 단어를 앞에서부터 순회하며 m1 < 3이면 m1 문제, m2 < 3이면 m2 문제를 추가(단어당 최대 2, m1 후보 먼저). 전체 60에 도달하면 중단 — 슬롯이 1개 남았는데 후보가 2개면 m1 후보만 들어간다. isReview: false.
  • ④ 셔플+분산: Fisher–Yates(rng 사용)로 전체 셔플 후, 인접한 같은 단어(tab+hanzi) 쌍을 스캔해 새 인접 쌍을 만들지 않는 위치와 교환하는 보정 패스 1회. 교환 불가능한 배치(예: 같은 단어 2문제만 있는 큐)는 그대로 둔다 — PRD의 "가능한 한 분산"이 허용하는 범위.

기각한 대안: (a) 인터리빙 방식 재배치(빈도 기반 슬롯 배치) — 단어당 최대 2문제·상한 60이라 셔플+보정이 훨씬 단순하고 충분함. (b) 큐 항목에 단어 필드 복사(flat 구조) — #15에서 POST /api/answer 후 갱신된 단어로 교체하기엔 참조 방식이 낫고, 중복 타입 정의도 피함. (c) worker/lib/words.ts의 WordEntry를 직접 import — 클라이언트 코드가 worker 코드에 의존하는 방향이 생겨 경계가 흐려짐.

4작업 단계

  1. 타입·상수·시그니처 정의: SessionQuestion<T>, SESSION_CAP = 60, buildSessionQueue(words, today, rng) 골격과 모듈 헤더 주석(PRD §6.1 근거) 작성 src/lib/sessionQueue.ts (신규)
  2. 복습 대기 선별 구현: getWordState로 분류 → nextReview 오름차순(null 최우선, stable) 정렬 → 60개 컷 → 단어당 1문제·모드 무작위(rng) src/lib/sessionQueue.ts
  3. 학습 중 채우기 구현: m1+m2 내림차순(stable) 정렬 → 단어당 m1/m2 후보 생성(카운트 3 미만인 모드만) → 남은 슬롯만큼 순서대로 채움 src/lib/sessionQueue.ts
  4. Fisher–Yates 셔플 + 인접 동일 단어 보정 패스 구현 src/lib/sessionQueue.ts
  5. 단위 테스트 작성: 이슈 완료 조건 4건 + 경계 케이스(빈 입력, reviewScheduled 제외, null nextReview 우선, 정확히 60, 모드 후보 규칙, isReview 플래그). rng는 고정 수열/간단 LCG로 결정론화, 셔플 분산은 여러 시드로 검증 src/lib/sessionQueue.test.ts (신규)
  6. 검증: npm test · npm run lint · npm run build(tsc 포함) 통과 확인 후 리포 스타일 커밋 package.json scripts

5리스크

리스크영향완화책
분산이 수학적으로 불가능한 구성(예: 같은 단어 2문제뿐인 큐)보정 패스가 인접 쌍을 못 없앰PRD가 "가능한 한"으로 허용 — 보정은 최선 노력으로 두고, 테스트는 분리 가능한 구성에서만 무인접을 단정
nextReview: null인 복습 대기(데이터 이상)의 정렬 위치가 미정의60개 컷에서 포함/제외가 갈림"가장 오래된 것"으로 취급해 최우선 포함 — wordState.ts:40-42의 "눈에 띄게 한다" 방침과 일치, 주석·테스트로 고정
PRD의 "시트 상 순서"를 코드에서 식별할 별도 필드가 없음동률 정렬이 무근거해질 수 있음GET /api/words가 시트 행 순서대로 반환하므로 입력 배열 순서를 그 자체로 사용, stable sort(toSorted)로 보존 — 문서화 주석 필수
#15(진행 규칙)와의 인터페이스 어긋남재작업mode: "m1"|"m2"·isReview를 API 계약(PRD §7.3)에 맞추고 SESSION_CAP export로 §6.2 재삽입 상한도 대비

6테스트 계획

  • 복습 대기 61개 이상 → 복습일 오래된 순 60개만 포함, 가장 최신 복습일 단어가 잘리는지 확인 (이슈 완료 조건 1)
  • 학습 중 정렬: 총 정답 수(D+E) 큰 단어 우선, 동률이면 입력(시트) 순서 유지 — 슬롯을 제한해 어떤 단어가 들어왔는지로 검증 (완료 조건 2)
  • 복습 30 + 학습 후보 100+ → 큐 길이 정확히 60 (완료 조건 3)
  • 분리 가능한 구성에서 여러 rng 시드로 셔플 → 같은 단어(tab+hanzi) 인접 쌍 0건 (완료 조건 4)
  • 모드 후보 규칙: m1≥3, m2<3인 학습 단어는 m2 문제만 생성
  • 복습 문제는 isReview: true·단어당 1문제·모드가 rng에 따라 m1/m2 모두 나올 수 있음, 학습 문제는 isReview: false
  • reviewScheduled(복습 예약) 단어는 큐에서 완전 제외
  • nextReview: null인 졸업 단어(데이터 이상)가 60개 컷에서 최우선 포함
  • 빈 입력 → 빈 큐, 학습 단어만 / 복습 단어만인 경우도 정상 동작
  • 기존 테스트 회귀 없음: npm test 전체 통과, npm run lint·npm run build 클린

7완료 조건

  • 복습 대기 60개 초과 케이스에서 복습일 오래된 순으로 잘린다 (null은 최우선, 동률은 시트 순서)
  • 학습 중 단어가 총 정답 수 내림차순·동률 시 시트 순서로 선별된다
  • 단어당 후보 규칙(D<3→모드1, E<3→모드2, 최대 2문제)이 지켜진다
  • 전체 큐가 60문제 상한을 준수한다
  • 분리 가능한 구성에서 같은 단어의 문제가 연달아 나오지 않는다
  • 큐 항목이 mode: "m1"|"m2"·isReview를 담아 #15가 변환 없이 소비할 수 있다
