#48기능 개선

Worker 단어 등록 API — GET /api/tabs + POST /api/words/register

damdam6/vocabulary-study 브랜치 feat/48-worker-register-api ← main 라벨 enhancement 2026-07-19

1요약

단어 등록 시스템 플랜(docs/plans/word-registration-system.md §6)의 3단계(저장) Worker API 2개를 구현한다. GET /api/tabs는 학습 대상 탭 목록을 반환하고, POST /api/words/register는 클라이언트가 보낸 단어 배열을 서버에서 재검증한 뒤 탭 내 중복 한자는 스킵하고 나머지만 A·B·C열에 append-only로 기록한다. 기존 worker/lib/sheets.ts·google-auth.ts·Bearer 인증 미들웨어를 그대로 재사용하며, 새 탭은 제출 시점에만 기존 탭의 헤더 행을 복사해 생성한다(빈 탭 방지).

2현황 분석

  • Worker에는 GET /api/words·POST /api/answer·POST /api/review-fail만 존재한다(worker/index.ts:26-51). 등록 관련 엔드포인트 없음.
  • 모든 /api/* 요청은 isAuthorized 미들웨어를 먼저 통과해야 한다(worker/lib/auth.ts, worker/index.ts:12-17) — 새 라우트도 worker/index.ts에 분기만 추가하면 자동으로 이 게이트를 상속한다.
  • worker/lib/sheets.ts는 getValues·updateValues·batchUpdateValues·getSheetTitles·appendValues를 제공한다(worker/lib/sheets.ts:39-100). 그런데 appendValues(:append 엔드포인트, 암묵적 테이블 경계 감지)는 어떤 라우트도 쓰지 않는다 — 기존 라우트(worker/routes/answer.ts, worker/routes/review-fail.ts)는 전부 "A2:A 스캔으로 행 번호를 매 요청 재계산 → updateValues/batchUpdateValues로 명시적 range에 쓰기" 패턴을 쓴다(worker/lib/words.ts:44-61,67-80 주석 — PRD 4.2, "행 번호는 캐시하지 않고 재탐색").
  • routes/words.ts:11-12가 이미 _ 접두 탭 제외 필터(title.startsWith("_"))를 구현 중 — 새 GET /api/tabs가 같은 필터를 그대로 복제하면 로직이 두 곳에 흩어진다.
  • 탭 생성(스프레드시트 구조 변경, addSheet 요청)은 sheets.ts에 아직 없다 — 기존 batchUpdateValues는 /values:batchUpdate(값 쓰기 전용)라 별개 엔드포인트다.
  • 테스트는 worker/index.test.ts 한 파일뿐이고 인증 게이트·/api/health만 검증한다 — Sheets API를 모킹하는 라우트 레벨 테스트는 이 저장소에 아직 한 건도 없다. 클라이언트 쪽(src/lib/wordsApi.test.ts, retryQueue.test.ts)은 vi.stubGlobal("fetch", ...)로 전역 fetch를 스텁하는 패턴을 쓴다.
  • worker/lib/google-auth.ts는 isolate 스코프 모듈 캐시(cached 토큰, cachedPrivateKey, inFlight)를 갖는다 — 실제 RSA 서명 없이 테스트하려면 이 모듈을 통째로 모킹하는 게 가장 안전하다.
  • PRD 컬럼 계약(docs/PRD.md:57-67): A=한자(탭 내 유일 키)·B=병음·C=한국어 뜻은 "등록 skill"이 쓰는 열, D(모드1)·E(모드2)·F(복습)·G~(타임스탬프)는 "사이트"가 쓰는 열 — 즉 등록 API는 A·B·C만 채우고 신규 행의 D·E·F는 빈칸으로 남겨야 한다.

3설계

  • 라우트 분리: worker/routes/tabs.ts(GET /api/tabs), worker/routes/register.ts(POST /api/words/register) — 기존 1라우트-1파일 컨벤션 유지.
  • 순수 로직 분리: 새 worker/lib/register.ts에 parseRegisterWords(스키마 §3 재검증: 필드 존재·타입·트림 후 비어있지 않음·배열 내 중복 한자 금지), normalizeTabName(트림 + _ 시작 차단), partitionByExisting(기존 한자 목록 대비 skip/add 분리)를 둔다. lib/answer.ts의 computeAnswerUpdate와 같은 패턴 — Sheets 모킹 없이 순수 함수 유닛 테스트가 가능해진다.
  • 신규 행 쓰기 방식: appendValues(:append, 암묵적 테이블 감지)를 쓰지 않고, 이미 중복 검사를 위해 읽어둔 A2:A 결과의 길이로 다음 행 번호(rows.length + 2)를 계산해 updateValues(tab, "A{n}:C{n+k-1}", rows)로 명시적으로 쓴다. 저장소 전역 원칙("행 번호 캐시 금지, 매 요청 재탐색")과 일치하고, 이미 가진 데이터를 재사용하며, D열 이후를 절대 건드리지 않는다는 걸 range만 보고도 보장할 수 있다.
  • 탭 생성(addSheet): sheets.ts에 스프레드시트 레벨 :batchUpdate(addSheet 요청 — 값 batchUpdate와 다른 엔드포인트)를 새로 추가한다.
  • 헤더 복사 원본 선택(이슈 텍스트가 "기존 탭"이라고만 하고 특정하지 않음 — 추론): GET /api/tabs와 같은 필터로 구한 첫 번째 단어 탭의 1행을 헤더 원본으로 쓴다. 그런 탭이 하나도 없으면 400("헤더를 복사할 기존 탭이 없습니다").
  • 탭 미존재 + createTab 미지정(이슈 텍스트에 없는 조합 — 추론): 정상 클라이언트 플로우(드롭다운 선택 vs "+ 새 탭")에서는 나오지 않지만, 플랜 §2 "클라이언트를 신뢰하지 않는다" 원칙에 따라 Worker가 방어적으로 400 처리한다.
  • 탭 이름 규칙 위반 응답: 트림 후 빈 문자열이거나 _로 시작하면 400. 트림 후 기존 탭과 동일하면 createTab 값과 무관하게 그 기존 탭을 사용(신규 생성 안 함).
  • 응답 스키마: { tab, created, added: {hanzi,pinyin,meaning}[], skipped: string[] } — 이슈가 요구하는 "added/skipped 상세"를 만족.
  • 기각한 대안: Worker에서 pinyin-pro 사전 검증까지 재실행 — 기각. 플랜 §5(작업 3, 별도 이슈)의 클라이언트 책임이고, 플랜 §6은 Worker 역할을 "스키마 재검증"(구조적 검증)으로 한정한다. Worker는 필드 존재·타입·공백 여부·배열 내 중복까지만 본다.

4작업 단계

  1. Sheets API 래퍼에 탭 생성 함수 추가 — 스프레드시트 레벨 :batchUpdate(addSheet 요청)를 감싸는 addSheet(env, title). worker/lib/sheets.ts
  2. _ 접두 필터 공유 헬퍼 추출 — getWordTabTitles(env)(getSheetTitles + 필터)를 만들고 handleGetWords가 이걸 쓰도록 리팩터. worker/lib/words.ts 또는 worker/lib/sheets.ts, worker/routes/words.ts
  3. 등록 전용 순수 로직 모듈 작성 — parseRegisterWords·normalizeTabName·partitionByExisting. worker/lib/register.ts
  4. GET /api/tabs 라우트 구현 — getWordTabTitles 호출 후 { tabs: string[] } 반환. worker/routes/tabs.ts
  5. POST /api/words/register 라우트 구현 — 바디 파싱(400 처리 포함) → 탭 이름 정규화·해석(기존 탭 재사용/생성/400) → 생성 시 헤더 원본 탭 선택 + addSheet + 헤더 행 복사 → A2:A 스캔으로 기존 한자 목록 확보 → partitionByExisting → 신규 행을 다음 빈 행부터 A·B·C열에만 updateValues로 기록 → 응답 조립. worker/routes/register.ts
  6. 두 라우트를 worker/index.ts에 배선 — 기존 라우트와 같은 try/catch + console.error + 500 패턴. worker/index.ts
  7. 순수 로직 유닛 테스트 — parseRegisterWords(필드 누락/빈 문자열/배열 내 중복 → null), normalizeTabName(트림/_차단/빈 문자열 → error), partitionByExisting(중복 스킵 분리). Sheets 모킹 불필요. worker/lib/register.test.ts
  8. 라우트 레벨 테스트용 Sheets 모킹 하네스 구성 — google-auth.ts를 vi.mock으로 대체(고정 토큰 반환)하고 vi.stubGlobal("fetch", ...)로 sheets.googleapis.com 호출을 경로별로 가로채는 헬퍼 작성. worker/index.test.ts 또는 새 worker/routes/register.test.ts — 계획과 달리 공유 모듈로 뽑지 않고 각 테스트 파일에 인라인으로 둠(register 쪽은 GET/PUT/addSheet까지 흉내내는 전체 하네스, tabs 쪽은 titles만 흉내내는 5줄짜리 축약판) — 재사용 범위가 작아 분리 이득이 없었음
  9. 라우트 레벨 시나리오 테스트 — 인증 없음 401(기존 게이트 재확인), 탭 이름 규칙 위반 400, 페이로드 스키마 위반 400, 중복 한자 스킵(기존 행 미변경 검증), 신규 행 append-only(A-C열만, 다음 빈 행), createTab 헤더 복사 후 등록, 트림 후 기존 탭과 동일하면 재사용(생성 안 함). worker/index.test.ts 또는 worker/routes/register.test.ts, worker/routes/tabs.test.ts

5리스크

리스크영향완화책
라우트 레벨 Sheets 모킹 하네스가 이 저장소 최초 사례실제 Google API 응답 형태와 모킹이 어긋나면 테스트는 통과해도 실제 호출은 실패할 수 있음기존 sheets.ts 응답 타입(UpdateResult/BatchUpdateResult 등)에 맞춰 모킹 응답을 구성하고, 최종적으로 npm run deploy 전 실제 시트로 수동 확인 1회 권장
헤더 복사 원본 탭 선택 기준(첫 번째 단어 탭)이 이슈 텍스트에 명시되지 않은 추론사용자가 기대한 원본과 다를 수 있음구현 중 사용자에게 확인하거나 PR 설명에 명시
탭 미존재 + createTab 미지정 조합의 400 처리도 추론정상 플로우에서는 발생하지 않지만 계약 문서화 누락 가능에러 메시지에 원인을 명확히 남기고 PR에 언급
동시 등록 요청 레이스(같은 탭에 동시 두 요청 → 같은 다음 행 계산)낮은 확률로 행 충돌개인용 저트래픽 앱이라 범위 밖으로 간주 — 기록만 해둠

6테스트 계획

  • parseRegisterWords: 정상 배열 통과, 필드 누락/빈 문자열/타입 불일치/배열 내 한자 중복 → null 반환 확인.
  • normalizeTabName: 앞뒤 공백 트림, _ 시작 차단, 트림 후 빈 문자열 차단 확인.
  • partitionByExisting: 기존 한자와 겹치는 항목은 skipped로, 나머지는 toAdd로 정확히 분리되는지 확인.
  • 라우트: 인증 헤더 없이 POST /api/words/register → 401 (기존 /api/* 게이트 재사용 확인).
  • 라우트: 탭 이름이 _test처럼 _로 시작 → 400, 에러 메시지에 규칙 명시.
  • 라우트: words가 빈 배열이거나 스키마 위반 필드 포함 → 400.
  • 라우트: 대상 탭에 이미 존재하는 한자가 포함된 요청 → 해당 항목은 skipped로 응답되고, 기존 행에 대한 쓰기 호출(updateValues/batchUpdateValues)이 발생하지 않는지 모킹 호출 인자로 검증.
  • 라우트: 신규 단어만 있는 요청 → 다음 빈 행부터 A{n}:C{n+k-1} range로만 쓰기가 발생(D~G열 미포함) 확인.
  • 라우트: createTab: true + 신규 탭 이름 → addSheet 호출 후 헤더 행이 기존 탭에서 복사되고, 그 다음 단어 쓰기가 2행부터 시작하는지 확인.
  • 라우트: 트림 후 이름이 기존 탭과 동일 → createTab: true여도 addSheet가 호출되지 않고 기존 탭에 등록되는지 확인.
  • 라우트: GET /api/tabs → _ 접두 탭 제외한 목록만 반환되는지 확인.
  • npm test 전체 그린 확인, npm run lint 통과 확인.

7완료 조건

  • GET /api/tabs·POST /api/words/register 두 엔드포인트가 인증 하에 동작, 미인증 401.
  • 탭 내 A열 중복 한자 스킵 — 기존 행은 어떤 경우에도 수정되지 않음(테스트로 검증).
  • 신규 단어는 마지막 데이터 행 아래 A·B·C열에만 append(D열 이후 불가침) — 테스트로 검증.
  • createTab 요청 시 기존 탭 헤더 행 복사로 새 탭 생성(빈 탭 방지) — 테스트로 검증.
  • 탭 이름 규칙(트림, _ 시작 차단) 위반 시 400.
  • 스키마 위반 페이로드(필드 누락/타입 불일치/배열 내 중복) 400 — 기존 answer.ts/review-fail.ts 테스트 스타일과 동일한 검증 방식.
