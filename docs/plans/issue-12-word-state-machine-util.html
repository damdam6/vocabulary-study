1요약

PRD §5.1이 정의하는 단어 상태(학습 중/복습 대기/복습 예약) 판정을 순수 함수로 분리한다. GET /api/words(#7)가 이미 정규화해 내려주는 m1·m2·nextReview 세 필드만으로 판정하며, Asia/Seoul 자정 기준 "오늘" 계산도 별도 함수로 함께 제공한다. PRD §7.3에 따라 이 판정은 클라이언트 책임이므로 src/lib/에 신설하고, 레포 최초의 테스트 프레임워크로 vitest를 도입해 경계값 케이스를 검증한다. 홈 화면 집계(#13)와 세션 큐 구성(#14)은 이후 이 유틸을 그대로 소비할 예정이며, 이번 이슈 범위는 판정 함수와 테스트로 한정한다.

2현황 분석

  • PRD §5.1(docs/PRD.md:85-96)이 세 상태의 판정 조건을 정의한다 — 학습 중: D<3 또는 E<3, 복습 대기: D≥3 && E≥3 && F날짜≤오늘, 복습 예약: D≥3 && E≥3 && F날짜>오늘. §5.4(docs/PRD.md:125-128)는 "오늘"의 경계를 Asia/Seoul 자정으로 못박는다.
  • GET /api/words(#7, 완료)는 worker/routes/words.ts:10-26에서 전 탭을 통합해 WordEntry(worker/lib/words.ts:8-17) 배열로 응답한다. D·E열 빈칸은 parseWordRow(worker/lib/words.ts:20-33)가 이미 0으로, F열 빈칸은 parseNextReview(worker/lib/words.ts:36-46)가 이미 nextReview: null로 정규화한다 — 이 유틸은 문자열 파싱을 다시 할 필요 없이 m1·m2·nextReview 세 필드만 보면 된다.
  • PRD §7.3(docs/PRD.md:195): "큐 구성(§6.1)과 상태 판정(§5.1)은 클라이언트가 수행한다. Worker는 데이터 중계만." — 즉 이 유틸은 worker/가 아니라 src/ 코드다.
  • 현재 src/에는 App.tsx·main.tsx·assets뿐이고 lib/ 디렉터리도, 상태 판정 로직도 존재하지 않는다.
  • 이 유틸의 실제 소비처인 #13(홈 화면 UI)·#14(세션 큐 구성)는 아직 미착수 — 지금은 순수 함수와 그 테스트만 만든다.
  • 레포에 테스트 관련 devDependency·스크립트가 전혀 없다(package.json:14-28) — 이슈가 "vitest 도입(레포 첫 테스트 프레임워크)"을 완료 조건으로 명시.
  • worker/lib/time.ts:7-21에 Asia/Seoul Intl.DateTimeFormat + formatToParts 조합으로 날짜·시간을 문자열화하는 패턴이 이미 검증된 형태로 존재 — "오늘" 계산에 동일 기법을 재사용할 선례.
  • tsconfig.app.json:25(include: ["src"])와 tsconfig.worker.json:7(include: ["worker"])는 서로 다른 TS 프로젝트다. src에서 worker/lib/words.ts의 WordEntry를 직접 import하면 소속 tsconfig 프로젝트의 include 범위를 벗어난 파일을 참조하게 된다.
  • vite.config.ts:7-9는 @cloudflare/vite-plugin을 포함한다 — vitest가 이 파일을 그대로 읽으면 순수 함수 테스트에도 Workers 관련 플러그인이 로드될 수 있다.

3설계

  • 위치: src/lib/wordState.ts 신설. PRD §7.3에 따라 클라이언트 코드이며, worker/는 별도 tsconfig 프로젝트라 애초에 선택지가 아니다.
  • 입력 타입: WordEntry 전체를 import하지 않고, 이 함수가 실제로 쓰는 3개 필드만 담은 로컬 인터페이스 WordProgress { m1: number; m2: number; nextReview: string | null }를 새로 선언한다. 구조적 타이핑 덕분에 /api/words 응답 객체(WordEntry)를 그대로 넘겨도 자연히 호환된다. (대안: worker에서 타입 import — tsconfig 프로젝트 경계상 불가.)
  • 상태 값: WordState = "learning" | "reviewDue" | "reviewScheduled". 기존 API의 mode: "m1"|"m2"와 같은 영문 판별 유니언 관례를 따른다. 한국어 라벨 매핑은 이 유틸을 소비하는 UI(#13 등) 책임으로 남긴다.
  • today를 문자열로 명시적으로 받기: getWordState(word: WordProgress, today: string): WordState. 내부에서 new Date()를 매번 계산하지 않고 호출자가 전달한다. 실사용처(#13 홈 집계, #14 세션 큐)는 모두 "여러 단어를 한 번에" 분류하므로, 오늘 날짜를 단어마다 새로 계산하면 비효율일뿐더러 배치 도중 자정을 넘길 경우 같은 배치 안에서 기준일이 흔들리는 미묘한 버그 여지가 생긴다. 한 번 계산해 모든 단어에 동일하게 적용하는 편이 맞다.
  • getSeoulToday(now: Date = new Date()): string: "오늘"(Asia/Seoul, YYYY-MM-DD) 계산을 별도 함수로 분리해 getWordState와 나란히 export한다. worker/lib/time.ts:7-21과 동일한 Intl.DateTimeFormat+formatToParts 기법을 재사용한다(코드는 tsconfig 경계 때문에 직접 import 불가하므로 기법만 재사용, 코드는 새로 작성). now 기본 인자 덕분에 실사용은 getSeoulToday() 한 줄로 충분하고, 테스트는 특정 시각을 주입해 자정 경계를 검증할 수 있다. PRD §9.1(홈 화면 상단 "오늘 날짜" 표시)에서도 재사용 가능하도록 export해 둔다.
  • 판정 순서: PRD 문구 그대로 "학습 중" 조건(m1<3 || m2<3)을 가장 먼저 검사하고, nextReview가 무엇이든(심지어 null이어도) 그 시점엔 절대 들여다보지 않는다. 완료 조건이 요구하는 "nextReview: null(미졸업) 단어가 학습 중으로 분류"를 설계상 자동으로 만족시킨다.
  • 데이터 이상 상태(문서화된 결정): PRD상 F열은 졸업 시점에 항상 채워지므로 "m1≥3 && m2≥3인데 nextReview===null"은 정상 흐름에서 나오지 않는다. 다만 시트 수동 편집 등으로 이런 상태가 생기면, nextReview===null을 "복습 예약"(미래)이 아니라 "복습 대기"로 처리한다 — 알 수 없는 미래로 영원히 숨기는 것보다 사용자 눈에 띄게 노출하는 편이 안전하다. 코드 주석으로 근거를 남기고 전용 테스트를 추가한다.
  • vitest 설정 분리: vite.config.ts에 test 필드를 얹지 않고 별도 vitest.config.ts를 루트에 새로 만든다. Vitest는 vitest.config.ts가 있으면 vite.config.ts보다 그것을 우선 사용한다 — @cloudflare/vite-plugin이 순수 함수 단위 테스트 실행에 끼어들 여지를 원천 차단한다. (대안: vite.config.ts에 test 블록 병합 — cloudflare 플러그인의 dev/build 훅이 테스트 실행 시에도 함께 로드되어 불필요하게 무거워지고, 향후 워커 전용 테스트(#8, #9 등)와 프론트 테스트가 같은 설정을 억지로 공유하게 된다.)
  • 테스트 파일 위치: src/lib/wordState.test.ts(코로케이션). vitest 기본 include 글롭이 *.test.ts를 자동으로 잡으므로 vitest.config.ts에 별도 include 설정이 필요 없다. 테스트는 import { describe, it, expect } from "vitest"로 명시적 import만 쓰고 글로벌 타입에 의존하지 않아, tsconfig.app.json 수정도 필요 없다.

4작업 단계

  1. npm install -D vitest로 개발 의존성 추가 (peer dependency상 현재 vite ^8.1.1과 호환). package.json, package-lock.json
  2. 루트에 vitest.config.ts 신규 작성 — vite.config.ts와 분리된 최소 설정. vitest/config의 defineConfig만 사용하고 cloudflare 플러그인은 포함하지 않는다. vitest.config.ts
  3. package.json의 scripts에 "test": "vitest run" 추가. package.json
  4. src/lib/wordState.ts 작성 — WordState 유니언, WordProgress 인터페이스, getSeoulToday(now?), getWordState(word, today) 구현. 설계 섹션의 판정 순서·null 처리 결정을 그대로 반영. src/lib/wordState.ts
  5. src/lib/wordState.test.ts 작성 — 학습 중/복습 대기/복습 예약 각 상태, nextReview===today 경계값, nextReview:null+미졸업, nextReview:null+졸업(이상 상태) 케이스, getSeoulToday의 UTC↔Asia/Seoul 자정 경계 2건을 포함한다 (테스트 계획 섹션에 구체 값 명시). src/lib/wordState.test.ts
  6. npm run test, npm run lint, npm run build를 실행해 전부 통과하는지 확인. 검증 단계 — 파일 변경 없음

5리스크

리스크영향완화책
vite.config.ts에 test 설정을 합치면 cloudflare 플러그인이 테스트 실행에도 로드됨테스트가 느려지거나 Workers 런타임 관련 초기화가 불필요하게 얽힐 수 있음별도 vitest.config.ts로 분리 (설계 섹션 참고)
WordEntry를 재사용하지 않고 WordProgress로 로컬 재정의두 타입이 시간이 지나며 구조적으로 어긋날 잠재 여지(예: worker 쪽 필드 타입 변경을 프론트가 못 알아챔)필드 3개뿐이라 어긋날 표면적이 작고, 어긋나면 /api/words 응답을 그대로 소비하는 다른 프론트 코드에서도 먼저 드러남
졸업(m1≥3 && m2≥3)인데 nextReview===null인 데이터 이상 상태는 PRD에 명시 안 됨이 구현의 처리 방식(복습 대기로 간주)을 다른 개발자가 놓칠 수 있음코드 주석 + 이 플랜에 근거 명시, 전용 테스트 케이스로 동작 고정

6테스트 계획

  • npm run test — getWordState: m1<3(m2 무관) 및 m2<3(m1 무관) → "learning", nextReview 값과 무관(null 포함).
  • npm run test — getWordState: m1≥3 && m2≥3 && nextReview가 today보다 과거 → "reviewDue".
  • npm run test — getWordState: m1≥3 && m2≥3 && nextReview === today → "reviewDue" (경계값, 완료 조건 명시 케이스).
  • npm run test — getWordState: m1≥3 && m2≥3 && nextReview가 today보다 미래 → "reviewScheduled".
  • npm run test — getWordState: m1<3 && nextReview===null → "learning" (완료 조건 명시 케이스).
  • npm run test — getWordState: m1≥3 && m2≥3 && nextReview===null(이상 상태) → "reviewDue" (설계 섹션의 방어적 기본값 검증).
  • npm run test — getSeoulToday: 2026-07-19T14:59:00.000Z(Seoul 23:59) → "2026-07-19", 2026-07-19T15:00:00.000Z(Seoul 00:00) → "2026-07-20" — 1분 차이의 UTC 인스턴트가 Asia/Seoul 자정을 사이에 두고 날짜가 갈리는지 검증.
  • npm run lint(oxlint) — 신규 파일 통과 확인.
  • npm run build(tsc -b && vite build) — 타입 에러 없이 빌드 성공. wordState.test.ts가 tsconfig.app.json의 include에 포함돼도 vitest의 타입 선언(devDependency)만으로 통과하는지 확인.

7완료 조건
