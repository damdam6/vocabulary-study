#8기능 개선

POST /api/answer 구현 — 정답 기록·졸업 판정·간격 갱신

damdam6/vocabulary-study 브랜치 feat/8-post-api-answer ← main 라벨 enhancement 2026-07-17

1요약

PRD에서 가장 복잡한 조각인 POST /api/answer를 신규 구현한다. 요청받은 탭+한자로 시트 행을 캐시 없이 재탐색하고, 한 요청 안에서 모드별 카운트(D/E) 증가, 졸업 판정(F열 최초 기록), 복습 간격 사다리 상승(F열 갱신), G열 이후 첫 빈 칸에 타임스탬프 append를 순차 수행한 뒤 GET과 동일한 단어 객체를 반환한다. GET /api/words가 쓰는 parseWordRow 정규화와 sheets.ts의 read/write 래퍼를 그대로 재사용해 새 코드는 최소화한다.

2현황 분석

  • worker/index.ts:1-27 — 현재 라우팅은 /api/health, GET /api/words 둘뿐이고 그 외는 404 폴백. POST /api/answer는 없음.
  • worker/routes/words.ts:10-26 — GET 핸들러 패턴: getSheetTitles → getValues(tab, "A2:F") → parseWordRow. 새 라우트도 같은 파일 분리 방식(lib/ 순수 로직 + routes/ HTTP 핸들러)을 따른다.
  • worker/lib/words.ts:1-4 — 파일 헤더 주석에 이미 "POST /api/answer(#8)도 'GET과 같은 형태'로 응답해야 하므로 여기 정규화 로직을 재사용한다"고 명시돼 있음. parseWordRow(tab, row)는 [hanzi, pinyin, meaning, m1, m2, nextReviewRaw] = row만 구조분해하므로, G열 이후를 포함한 더 넓은 행 배열을 그대로 넘겨도 안전하게 동작한다.
  • worker/lib/sheets.ts:34-79 — getValues/updateValues/appendValues 전부 (env, tab, range) 시그니처이고 range는 A1 표기 그대로 전달돼 임의의 셀 좌표(D5, 12:12 등)를 그대로 쓸 수 있다. appendValues는 "표 마지막 행 뒤에 새 행 추가"용이라 G열처럼 기존 행의 오른쪽 빈 셀에 쓰는 요구에는 맞지 않는다 — updateValues로 특정 셀을 직접 갱신해야 한다.
  • worker/lib/time.ts:1-21 — formatSeoulDateTime(date)(날짜+시간)만 있고, 날짜만(YYYY-MM-DD) 포맷 + "N일 후" 계산 함수는 없음 — 신규 추가 필요.
  • worker/lib/auth.ts + worker/index.ts:8-10 — /api/* 전체가 이미 비밀번호 미들웨어로 보호돼 있어 새 라우트도 자동으로 인증 대상.
  • 시트 컬럼 계약(PRD §4.2): A한자 B병음 C뜻 D모드1카운트 E모드2카운트 F(다음복습일|간격) G열~ 타임스탬프. 신규 등록 시 D/E는 빈칸 또는 0.

3설계

  • 행 재탐색: A2:A 열만 읽어 hanzi와 일치하는 인덱스를 찾고(+2로 실제 행 번호 환산), 못 찾으면 404. 행 번호 캐시 금지(PRD 4.2) 요건과 일치. 대안 기각: 매 요청마다 GET처럼 전체 A2:F를 읽고 필터링 — 컬럼 하나만 읽는 것보다 데이터 낭비가 커서 기각.
  • 행 전체 재조회: 행 번호를 찾은 뒤 {row}:{row}(행 전체, 컬럼 무제한 A1 표기)로 D~G+ 전체를 한 번에 읽고 parseWordRow(tab, fullRow)를 그대로 호출해 현재 m1/m2/nextReview/interval을 얻는다 — words.ts 주석이 예고한 재사용 지점.
  • 카운트 +1 쓰기: mode→열 매핑(m1→D, m2→E)으로 updateValues(tab, "{D|E}{row}", [[newCount]]).
  • F열 갱신 조건부 계산 (신규 lib/answer.ts에 캡슐화):
    • isReview===true → 간격 사다리 [1,3,7,14,30]에서 현재 interval보다 큰 다음 값(없으면 30 유지) → 오늘+새간격|새간격.
    • isReview===false이고 이번 +1로 증가 전 상태 기준 처음 D≥3&&E≥3 성립(졸업) → 내일|1.
    • 그 외(아직 학습 중) → F열 쓰기 자체를 생략.
    대안 기각: F열을 항상 재조립해 덮어쓰기(안 바뀌어도) — 코드는 단순해지지만 원본이 파싱 불가능한 값(수동 오타 등)이었을 경우 조용히 지워버리는 부작용이 있어 기각. 안 바뀌면 쓰기 자체를 생략하는 쪽을 택한다.
  • G열 append: 재조회한 행 배열에서 인덱스 6(G, 0-based)부터 값이 있는 동안 전진해 첫 빈 인덱스를 찾고, base-26 컬럼 문자 변환(Z 다음 AA)으로 열 문자를 얻어 updateValues(tab, "{col}{row}", [["{timestamp}|{mode}"]]). Sheets append API(뒤에 새 "행" 추가)는 같은 행 오른쪽 빈 칸에 쓰는 이 요구에 맞지 않아 사용하지 않는다.
  • 날짜 계산: 한국은 DST가 없으므로 now.getTime() + days*86400000를 Asia/Seoul로 포매팅해도 달력 날짜가 어긋나지 않는다 → time.ts에 addSeoulDays(date, days): string(YYYY-MM-DD)만 추가하면 충분, 별도 타임존 라이브러리 불필요.
  • 응답 구성: 쓰기 성공 후 재조회하지 않고 이미 계산한 값(newM1/newM2, 새 F 또는 기존 F)으로 바로 WordEntry를 조립해 반환 — 왕복 1회 절약.
  • 행 탐색 함수의 위치: 탭+한자로 행 번호를 찾는 로직은 PRD 4.2가 명시한 공유 계약이라 lib/words.ts에 둔다. 다음 이슈로 예상되는 POST /api/review-fail도 동일 탐색이 필요하므로 재사용 지점이 된다 (단, 지금은 이 이슈에 필요한 시그니처만 구현한다).

4작업 단계

  1. 날짜만(YYYY-MM-DD) 포맷 + N일 후 계산 addSeoulDays(date, days) 추가. worker/lib/time.ts
  2. 탭+한자로 행 번호를 캐시 없이 재탐색하는 findRowNumber(env, tab, hanzi): Promise<number | null> 추가 (A2:A 조회, PRD 4.2 계약). worker/lib/words.ts
  3. 신규 파일: 간격 사다리 상수·nextLadderInterval(current), base-26 컬럼 문자 변환 columnLetter(index), 첫 빈 G+ 셀 인덱스 탐색 findNextEmptyColumn(row), 그리고 이를 조합해 D/E/F 갱신값을 계산하는 순수 함수 computeAnswerUpdate(...). worker/lib/answer.ts(신규)
  4. 요청 바디 검증(tab/hanzi/mode/timestamp/isReview 타입·mode∈{m1,m2}) → findRowNumber로 탐색(없으면 404) → 행 전체 재조회 → lib/answer.ts 계산 → updateValues로 D/E, (조건부) F, G+ 순차 기록 → 갱신된 WordEntry 응답. worker/routes/answer.ts(신규)
  5. POST /api/answer 라우트 등록 — GET /api/words와 동일한 try/catch + 500 폴백 패턴. worker/index.ts
  6. 실제 시트(예: HSK6급 탭)로 wrangler dev + curl 왕복 테스트: 학습 중 정답, 졸업 순간, 복습 정답, 존재하지 않는 한자 4가지 케이스를 직접 호출해 시트 값 변화를 눈으로 확인. 코드 변경 없음 — 검증 단계

5리스크

리스크영향완화책
Sheets API가 컬럼 생략 행 전체 범위({row}:{row}) 표기를 예상과 다르게 처리행 재조회 실패 → 500wrangler dev로 실제 시트에 먼저 확인, 필요시 A{row}:ZZ{row}처럼 넉넉한 고정 범위로 대체
G열이 오른쪽으로 계속 append되며 언젠가 Z를 넘는 열(AA 이상)에 도달컬럼 문자 변환 버그 시 잘못된 셀에 기록base-26 변환 함수를 Z→AA 경계값으로 직접 검증
요청의 tab이 실제 존재하지 않는 시트 이름인 경우Sheets API 400 → 현재 설계는 500으로 처리(404 아님)PRD가 요구하는 404는 "존재하지 않는 한자"뿐이라 범위 밖으로 명시. GET /api/words도 탭 유효성 검사를 하지 않는 것과 동일한 신뢰 모델
쓰기 3~4회(카운트/F/G) 사이 사용자가 시트를 직접 편집해 그 행이 바뀌는 동시성드묾, 개인 단일 사용자PRD 7.4에서 이미 비범위로 명시(락 없음, "읽고→계산→쓰기" 단순 구현으로 충분)

6테스트 계획

  • 학습 중 단어(D<3 또는 E<3, isReview:false)에 정답 호출 → 응답에서 해당 모드 카운트만 +1, nextReview/interval은 그대로 null, 시트 F열 불변, G열 이후 새 타임스탬프 1개 추가 확인.
  • 같은 단어가 이번 호출로 D≥3&&E≥3 첫 달성 → 응답 nextReview=내일 날짜, interval=1, 시트 F열에 내일|1 기록 확인.
  • 이미 졸업한 단어(isReview:true)에 정답 호출 → interval이 사다리대로 상승(예: 7→14), F열 오늘+14|14로 갱신, 카운트도 +1 확인.
  • interval=30인 단어에 복습 정답 → 사다리 최상단 유지, F열 오늘+30|30 확인.
  • 여러 번 연속 호출해 G,H,I…로 타임스탬프가 순서대로 쌓이고 기존 값이 지워지지 않는지 확인.
  • 시트에 없는 한자로 호출 → 404, 시트 값 변경 없음 확인.
  • 필수 필드 누락/mode 오타(m1·m2 외 값) → 400, 시트 값 변경 없음 확인.

7완료 조건

  • 학습 중 단어 정답: 카운트만 +1, F열 불변
  • 졸업 순간: F열에 내일|1 기록
  • 복습 정답: 간격 사다리대로 상승(1→3→7→14→30→30) + F열 갱신
  • G열 이후 기존 타임스탬프 보존하며 다음 빈 셀에 append
  • 존재하지 않는 한자로 호출 시 404
  • (추가) 필수 필드 누락·mode 오타 시 400 — 이슈 명시 항목은 아니나 견고성을 위해 구현에 포함
