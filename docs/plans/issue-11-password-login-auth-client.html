#11기능 개선

비밀번호 로그인 화면 + API 인증 클라이언트

1요약

백엔드 인증 미들웨어(#6, PR #21 머지 완료)가 /api/*에 Bearer 검증을 하고 있으므로, 프론트에 그 짝을 만든다. (1) 공통 fetch 래퍼 — localStorage의 비밀번호를 Authorization: Bearer로 자동 첨부하고 401이면 저장값 삭제 + 로그인 화면 복귀. (2) 로그인 화면 실제 구현 — design-prd §2 사양(로고·타이틀·54px 입력/버튼, GET /api/health로 검증, 확인 중/오류 shake 상태)대로 플레이스홀더를 교체한다.

2현황 분석

  • 백엔드는 준비 완료: worker/index.ts:10이 모든 /api/* 요청에 isAuthorized()(worker/lib/auth.ts:8) 검사를 하고 실패 시 401을 반환. GET /api/health(worker/index.ts:14)는 인증 통과 시 200 — 로그인 검증 엔드포인트로 그대로 사용 가능.
  • 프론트는 플레이스홀더: src/screens/LoginScreen.tsx는 "들어가기" 버튼만 있는 껍데기(주석에 "#11에서 구현" 명시). src/App.tsx:10은 항상 'login'으로 시작하고, 저장된 비밀번호 확인·401 처리 로직이 없다.
  • API 호출 코드가 아직 없음: src/lib/에는 wordState.ts뿐. fetch 래퍼는 이 이슈에서 신설하고, 이후 화면들(#13 홈, #14 학습)이 가져다 쓴다.
  • 디자인 기반은 마련됨: src/index.css에 색 토큰(--red 등)과 @keyframes shake(:119)가 이미 있고, 목업(docs/design/prototype.dc.html:30-39)에 로그인 화면 마크업(로고 58px·radius 17·그라데이션, 입력/버튼 54px·radius 14)이 있다.
  • 테스트 환경: vitest(node 환경, vitest.config.ts)로 순수 함수 테스트만 수행 중(src/lib/wordState.test.ts). 컴포넌트 테스트 도구(testing-library/jsdom)는 없다 — 이 이슈에서도 도입하지 않고, 래퍼 로직만 단위 테스트한다.

3설계

  • src/lib/api.ts 신설 — 저장소 + 래퍼 + 검증을 한 모듈로:
    • getStoredPassword() / savePassword() / clearPassword() — localStorage 키 app-password.
    • apiFetch(path, init?) — 저장된 비밀번호를 Bearer로 첨부해 fetch. 응답이 401이면 clearPassword() 후 등록된 unauthorized 핸들러 호출, 그리고 그대로 응답 반환(호출부는 401 분기를 다시 신경 쓸 필요 없음).
    • setUnauthorizedHandler(fn) — App이 마운트 시 setScreen('login')을 등록. 커스텀 이벤트 대신 모듈 레벨 콜백을 택한 이유: 소비자가 App 하나뿐이라 이벤트 버스는 과함.
    • verifyPassword(candidate) — 후보 값을 Bearer로 GET /api/health 직접 호출(래퍼 미경유 — 아직 저장 전이므로). 200 → 'ok', 401 → 'invalid', 그 외/네트워크 예외 → 'error'를 반환해 화면이 메시지를 구분하게 한다.
  • App.tsx — 초기 화면을 getStoredPassword() ? 'home' : 'login'으로 결정(새로고침 후 재입력 없이 통과). 마운트 시 unauthorized 핸들러 등록.
  • LoginScreen.tsx — design-prd §2 그대로: 상태 3종(idle/verifying/error). 제출 → verifyPassword → 'ok'면 savePassword + onLogin(), 'invalid'면 shake + "비밀번호가 올바르지 않습니다", 'error'면 같은 오류 스타일에 "서버에 연결할 수 없습니다"(사양 외 보강 — 401과 구분). 입력 변경 시 오류 해제. 목업 각주("프로토타입 —…")는 넣지 않는다.
  • 스타일 — 기존 관례대로 src/index.css에 .login-* 클래스 추가(별도 CSS 파일 없음). shake는 기존 keyframes 재사용(0.4s), 오류 테두리는 var(--red). .screen-placeholder는 #13/#15가 아직 쓰므로 유지.

4작업 단계

  1. API 인증 클라이언트 모듈 작성 — localStorage 헬퍼, apiFetch(Bearer 자동 첨부 + 401 시 삭제·핸들러 호출), setUnauthorizedHandler, verifyPassword(health 검증, 'ok'/'invalid'/'error') src/lib/api.ts
  2. 래퍼 단위 테스트 — fetch/localStorage 스텁으로: 헤더 첨부, 401 시 저장값 삭제+핸들러 호출, 200 통과, verifyPassword 3분기 src/lib/api.test.ts
  3. App에 인증 상태 배선 — 초기 화면을 저장값 유무로 결정, 마운트 시 unauthorized 핸들러 등록(저장값 삭제는 래퍼가 이미 수행 → 화면만 'login'으로) src/App.tsx
  4. 로그인 화면 실제 구현 — 로고/타이틀/안내/입력(autofocus)/버튼, verifying·error 상태, 제출 핸들러 src/screens/LoginScreen.tsx
  5. 로그인 화면 스타일 — .login-* 클래스(목업 수치: 로고 58px·radius 17·그라데이션, 입력/버튼 54px·radius 14, 오류 13px red), shake 적용 src/index.css
  6. 검증 — npm test·npm run lint·npm run build, dev 서버로 실제 플로우(오답 shake → 정답 통과 → 새로고침 유지 → 저장값 오염 시 401 복귀) 확인 package.json scripts 결과: 테스트 19개·린트·빌드 그린. 브라우저(dev 서버)로 오답 shake/미저장 → 재입력 시 오류 해제 → 정답 통과·저장 → 새로고침 유지까지 확인. 401 복귀 실 UI 재현은 홈이 아직 API를 호출하지 않는 플레이스홀더라 불가(#13에서 연결) — apiFetch 단위 테스트로 커버.
  7. 플랜 체크오프 반영 후 커밋·푸시·PR(Closes #11) docs/plans/issue-11-password-login-auth-client.html

5리스크

리스크영향완화책
vitest가 node 환경이라 localStorage/fetch 부재테스트가 환경 의존으로 깨질 수 있음테스트에서 globalThis에 스텁 주입(모듈은 전역만 참조하는 단순 구조 유지)
로컬 dev에서 APP_PASSWORD 시크릿 미설정 시 isAuthorized가 항상 false(worker/lib/auth.ts:9)수동 검증에서 정답도 401로 보임.dev.vars(gitignore 대상)로 로컬 비밀번호 지정 후 확인
이후 화면(#13/#14)이 래퍼를 안 거치고 fetch를 직접 쓰면 401 처리 누락인증 만료 시 화면별 동작 불일치래퍼 모듈 주석에 "API 호출은 반드시 apiFetch 경유" 명시 — 후속 이슈가 참조
네트워크 오류를 401과 동일 취급하면 저장값이 억울하게 삭제될 수 있음일시 장애에 재로그인 강요래퍼는 정확히 401에서만 삭제, verifyPassword는 'invalid'/'error' 분리

6테스트 계획

  • npm test — 신규 api.test.ts: Bearer 헤더 첨부 / 401 → 저장값 삭제 + 핸들러 호출 / 200 정상 통과 / verifyPassword가 200·401·네트워크 예외를 각각 'ok'·'invalid'·'error'로 반환. 기존 wordState.test.ts 회귀 없음.
  • npm run lint, npm run build(tsc -b 포함) 통과.
  • 수동(dev 서버): 오답 입력 → shake + 오류 메시지 + localStorage 미저장 / 재입력 시작 → 오류 해제 / 정답 → 홈 진입 + localStorage 저장 / 새로고침 → 재입력 없이 홈 / localStorage 값 훼손 후 API 호출 → 로그인 화면 복귀 확인.

  • 틀린 비밀번호 입력 시 shake + "비밀번호가 올바르지 않습니다" 표시, 저장되지 않음
  • 검증 중 버튼 비활성 + "확인 중…" 라벨
