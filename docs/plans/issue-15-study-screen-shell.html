1요약

학습 화면의 오케스트레이션 레이어를 구현한다. 홈에서 만든 세션 큐를 프롭으로 받아, 문제 단위로 소비하면서 정답/오답 네 케이스에 맞는 기록 API 호출(POST /api/answer / POST /api/review-fail)과 학습 중 오답 1회 재삽입을 수행하고, 진행도(동적 분모)·복습 뱃지·모드 칩·논블로킹 정오 피드백 오버레이·세션 완료 화면(design-prd §4.1·§4.4·§4.5·§5)을 갖춘다. 모드 1·2의 실제 카드 UI는 #16/#17 몫이므로, 여기서는 셸↔모드 컴포넌트 계약(onJudged/onProceed)을 확정한 플레이스홀더로 네 케이스를 실제 구동 가능하게 만든다.

2현황 분석

  • 화면은 전부 플레이스홀더: src/screens/StudyScreen.tsx:7은 종료/세션 완료 버튼만 있는 스텁, src/screens/DoneScreen.tsx:6도 수치 없는 스텁. 화면 전환 상태기계는 src/App.tsx:12에 있음(login→home→study→done).
  • 큐 빌더는 완성됐지만 호출자가 없다: buildSessionQueue()(src/lib/sessionQueue.ts:33)가 #14로 구현·테스트 완료. SESSION_CAP = 60(src/lib/sessionQueue.ts:11)의 주석이 "§6.2의 오답 재삽입 상한도 같은 값을 쓴다(#15)"라고 이 이슈를 지목. 문제 타입 SessionQuestion<T>(src/lib/sessionQueue.ts:16)에 mode·isReview가 이미 있음.
  • API 클라이언트는 인증만 있다: src/lib/api.ts:29의 apiFetch(Bearer 첨부 + 401 처리)와 verifyPassword뿐. words 조회·answer·review-fail 클라이언트 함수가 없다.
  • 백엔드 3종은 완료 상태(#7·#8·#9): GET /api/words(worker/routes/words.ts:10)는 {fetchedAt, words[]} 반환. POST /api/answer(worker/routes/answer.ts:37)는 {tab, hanzi, mode, timestamp, isReview}를 받아 갱신된 단어 객체를 반환. POST /api/review-fail(worker/routes/review-fail.ts:27)은 {tab, hanzi}를 받아 간격 후퇴.
  • src ↛ worker import 경계: tsconfig.app.json:25이 include: ["src"]라 worker/lib/words.ts:11의 WordEntry를 src에서 가져올 수 없다 → 클라이언트 쪽 WordEntry 타입(§7.3 응답 계약 미러)이 필요. 타임스탬프 포맷터(formatSeoulDateTime, worker/lib/time.ts:27)도 같은 이유로 클라이언트 미러 필요. "오늘"은 getSeoulToday()(src/lib/wordState.ts:17)가 이미 src에 있다.
  • CSS 기반은 준비돼 있다: 색 토큰·popfb/fadeup/shake 키프레임(src/index.css:88), 그리고 .app-column의 position: relative가 "정오 피드백 오버레이(§4.4, #15)의 absolute 앵커"로 미리 주석돼 있음(src/index.css:82).
  • 목업이 참조 구현을 제공: docs/design/prototype.dc.html:66-167에 학습 프레임·오버레이·완료 화면 마크업, :272-280에 advance()의 재삽입 로직(!q.review && !q.re && queue.length < 60). 단 design-prd §6의 변경점 — m1AutoMs 딜레이 삭제(즉시 진행), 오버레이 0.68s→0.4s 논블로킹, 로컬 시간 로직 이식 금지 — 를 반영해야 한다.
  • 이슈 분담(design-prd §9): #15 = §4.1 공통 프레임 · §4.4 오버레이 · §4.5 진행 규칙 · §5 완료 화면. §4.2 모드1 카드 UI = #16(OPEN), §4.3 모드2 UI = #17(OPEN), 재시도 큐 = #18(OPEN), 홈 실제 UI = #13(OPEN).

3설계

데이터 흐름 — 큐는 홈이 만들고 셸은 받는다

  • HomeScreen(플레이스홀더 유지, 배선만 추가)이 "학습 시작" 클릭 시 fetchWords() → buildSessionQueue(words, getSeoulToday()) → onStart(queue)로 올린다. App이 큐를 상태로 보관해 StudyScreen에 프롭으로 전달하고, 완료 시 onComplete({correct, wrong}) 결과를 받아 DoneScreen에 넘긴다.
  • 기각: 셸이 /api/words를 직접 fetch — PRD §3 "홈 진입 시마다 재조회"와 이중 조회가 되고, 홈의 "오늘 세션 · n문제" 계산과 큐가 어긋날 수 있다. 큐 구성은 홈 책임으로 고정, #13은 이 계약 위에서 UI만 교체.

진행 로직 — 순수 리듀서 studySession.ts

  • vitest가 node 환경(RTL·jsdom 없음, vitest.config.ts:5)이므로, 검증 대상인 진행 규칙을 컴포넌트 밖 순수 모듈로 분리한다 — wordState/sessionQueue와 같은 저장소 컨벤션.
  • recordAnswer(state, correct) → { state, effect }로 네 케이스를 한 곳에서 판정:
    • 정답(복습/학습중 공통): correct+1, effect = {kind:'answer', question} (isReview는 큐 구성 시 값 그대로).
    • 복습 오답: wrong+1, effect = {kind:'review-fail', question}, 재삽입 없음(복습 문제는 단어당 1개뿐이라 "이번 세션 재출제 없음"이 자동 성립).
    • 학습 중 오답: wrong+1, effect 없음, requeued 플래그가 없고 queue.length < SESSION_CAP일 때만 {...q, requeued: true}를 큐 맨 뒤에 push → 분모 즉시 증가. 재삽입 복사본이 다시 틀려도 requeued가 재재삽입을 막는다("1회만").
  • advance(state)는 pos+1만. applyWordUpdate(state, word)는 answer 응답의 갱신 단어를 남은 큐 스냅샷(같은 tab+hanzi)에 반영 — 화면 영향은 없지만 이슈 기대 동작("응답으로 카운트/간격 갱신 반영")과 데이터 정합을 지킨다. effect에 타임스탬프를 넣지 않아 모듈을 순수하게 유지(시각은 셸이 발사 시점에 생성).

진행 템포 — 셸이 규칙을 소유(§4.5), 모드 컴포넌트는 판정만

  • 계약: 모드 컴포넌트는 판정 순간 onJudged(correct)를 호출. 셸이 기록(effect 발사)하고 — 모드1 O/X와 모드2 정답은 오버레이 + 즉시 advance, 모드2 오답만 홀드(모드 컴포넌트가 내부 결과 화면 표시) 후 "다음" 버튼의 onProceed()로 advance(오버레이 없음, §4.4).
  • 모드2 채점(트림 후 정확 일치, 빈 입력=오답)은 §4.3 소관이므로 모드 컴포넌트(플레이스홀더) 안에 둔다. 셸은 판정 결과만 받는다 — #17이 UI를 갈아끼워도 셸 무변경.
  • 문제 전환 시 key={pos}로 모드 컴포넌트를 리마운트해 입력값·공개 상태 잔존을 차단(재삽입 복사본도 pos가 다르므로 안전).

기록 전송 — 문제 단위 fire-and-forget

  • postAnswer/postReviewFail은 실패 시 throw하는 정직한 헬퍼로 두고, 셸이 .catch()로 무시(전송 실패가 진행을 막지 않음 — 재시도 큐는 #18이 이 catch 지점에 배선). 성공 시 applyWordUpdate 반영. await 없이 발사하므로 advance가 네트워크를 기다리지 않는다.

오버레이·완료 전환

  • 오버레이는 .app-column 앵커의 position:absolute; inset:0; pointer-events:none, O/X 글리프 130px + 배경 틴트, popfb 0.4s forwards. 발사마다 증가하는 시퀀스를 key로 써서 연속 정답에도 애니메이션이 재시작되게 하고 onAnimationEnd에서 제거. 다음 문제 카드가 이미 전환된 위에 겹쳐 재생(논블로킹).
  • 기각: 목업처럼 오버레이 종료 후 advance — 확정 사양 "진행을 막거나 지연시키지 않는다"(§4.4) 위배.
  • 예외 — 마지막 문제: 즉시 onComplete하면 StudyScreen이 언마운트돼 마지막 판정의 피드백이 유실된다. 이 경우만 오버레이 onAnimationEnd(0.4s)에서 onComplete를 호출한다. 문제 간 전환은 여전히 즉시.

5리스크

리스크영향완화책
마지막 문제에서 즉시 완료 전환 시 오버레이 유실마지막 판정의 O/X 피드백이 안 보임마지막 문제만 onAnimationEnd(0.4s) 후 onComplete — 문제 간 전환은 즉시 유지
재삽입·분모·집계가 여러 곳에서 갱신되며 어긋남진행도/완료 수치 부정확 (완료 조건 위반)큐 변형은 recordAnswer 한 곳에서만, 단위 테스트로 고정
fire-and-forget 실패 무시로 기록 유실오프라인 학습 기록이 시트에 안 남음v1 허용 범위(이슈 명시 — 재시도 큐는 #18). 헬퍼는 throw로 정직하게, 셸 catch 지점에 #18 배선 주석
같은 단어 재출제 시 모드 컴포넌트 상태 잔존이전 입력/공개 상태가 새 문제에 남음key={pos} 리마운트로 문제마다 초기화
학습 중 401 수신 시 화면 전환 충돌세션 도중 로그인 화면 강제 복귀기존 apiFetch+setUnauthorizedHandler(#11) 동작 그대로 — 세션 상태는 App에 남지만 재로그인 후 홈부터 시작이라 무해. 수동 확인만
#13·#16·#17과의 경계 침범후속 이슈와 충돌·재작업홈은 배선만(UI 불변), 모드 카드는 계약 확정한 최소 플레이스홀더로 한정

6테스트 계획

  • vitest — studySession: ① 복습 정답 → effect answer(isReview:true) ② 복습 오답 → effect review-fail + 큐 길이 불변 ③ 학습중 정답 → effect answer(isReview:false) ④ 학습중 오답 → effect 없음 + 맨 뒤 재삽입(requeued:true) + 분모 +1 / 재삽입 복사본 재오답 시 무재삽입 / SESSION_CAP=60에서 재삽입 생략 / 정답·오답 집계 / applyWordUpdate가 남은 동일 단어 스냅샷 갱신.
  • vitest — api: postAnswer가 /api/answer에 {tab, hanzi, mode, timestamp, isReview} JSON을 Authorization과 함께 POST, postReviewFail·fetchWords 동형 검증, 비정상 응답 throw.
  • vitest — time: 고정 Date로 formatSeoulDateTime이 YYYY-MM-DD HH:mm(Asia/Seoul) 반환.
  • 회귀: npm test 전체(기존 wordState·sessionQueue·api 테스트 포함) + npm run lint + npm run build(tsc -b로 src/worker 타입 경계 확인).
  • 수동(dev 서버, 실제 시트): 네 케이스 각각에서 네트워크 탭으로 API 호출/무호출 확인 — 복습 정답(answer, isReview:true) · 복습 오답(review-fail, 재출제 없음) · 학습중 정답(answer, isReview:false) · 학습중 오답(호출 없음, 큐 뒤 재삽입, 분모 즉시 +1).
  • 수동 — 진행 템포·오버레이: 모드1 O/X·모드2 정답 직후 다음 카드가 즉시 표시되고 오버레이가 그 위에 겹쳐 재생되는지(연속 빠른 진행 가능), 모드2 오답만 결과 화면+"다음"인지.
  • 수동 — 실패 내성: DevTools 오프라인로 answer 전송 실패시켜도 다음 문제로 진행되는지.

7완료 조건
