배포 파이프라인을 GitHub Actions에서 Cloudflare Workers Builds(대시보드 Git 연결)로 전환

1요약

빌드+배포를 Cloudflare Workers Builds(대시보드 Git 연결)로 이관해 .github/workflows/deploy.yml과 CF API 토큰(GitHub 시크릿)을 없앤다. 단, #23(배포 직후 인증 우회)에서 나온 회귀 안전망인 미인증 401 검사는 시크릿 0개 스모크 워크플로우로 분리해 남긴다. 이 스모크는 단순 200/401 재시도가 아니라, 미인증 /api/words 401 응답에 실려오는 X-Worker-Version 헤더가 바뀌는지를 게이트로 삼아 새 배포가 실제 반영된 뒤에만 200/401을 단언한다(구버전 상대 오탐 방지). Node 24는 리포에 .node-version을 추가해 버전 관리로 고정한다. 코드 변경(이 PR)과 대시보드/시크릿 수동 작업의 순서를 무중단으로 배치한다.

2현황 분석

  • 현재 배포 워크플로우: .github/workflows/deploy.yml:1-46 — main push·workflow_dispatch 트리거, concurrency: deploy-production, cancel-in-progress으로 순서 역전 방지. 단계: checkout → setup-node@24(npm 캐시) → npm ci → npm run build → cloudflare/wrangler-action@v3(secrets.CLOUDFLARE_API_TOKEN·secrets.CLOUDFLARE_ACCOUNT_ID) → 스모크(배포 URL 200 + /api/words 401).
  • 빌드 산출물: npm run build = tsc -b && vite build(package.json:8). vite에 @cloudflare/vite-plugin의 cloudflare()가 적용돼(vite.config.ts:4-8) 배포 설정을 생성하므로 빌드 커맨드는 npm run build가 필수다.
  • Worker/인증: 단일 Worker가 정적 자산 + /api/*를 겸해 서빙(wrangler.jsonc:7-13). /api/*는 isAuthorized(Bearer APP_PASSWORD) 미들웨어로 보호돼 미인증 시 401(worker/index.ts:14-19, worker/lib/auth.ts:8-22). /api/health도 같은 가드 뒤라 미인증 401이다.
  • 버전 헤더: 401 응답에도 X-Worker-Version(CF_VERSION_METADATA.id)이 실린다(worker/index.ts:11-18, wrangler.jsonc:19-21). #23 배포 전파 윈도우의 신/구 버전 혼재를 관측하기 위해 노출한 값 — 미인증으로도 배포 버전을 읽을 수 있다(이번 스모크 게이트의 근거).
  • 기존 스모크 스크립트: scripts/deploy-auth-smoke.sh:13-53 — BASE_URL·DURATION(기본 60s) 동안 /api/health에 wrong/none 토큰 → 401, (SMOKE_PASSWORD 설정 시) correct → 200을 반복 검사하고 버전을 로깅. SMOKE_PASSWORD 미설정이면 401 경로만 검사 → 시크릿 불필요.
  • 프로덕션 URL: https://vocabulary-study.dambi626626.workers.dev (계정 서브도메인 dambi626626, CF API로 확인). 실측: root 200, /api/words 401, /api/health 401. wrangler.jsonc에 커스텀 route가 없어 이 workers.dev URL이 프로덕션이며, 리포엔 하드코딩돼 있지 않다.
  • Worker 이름: wrangler.jsonc:7 name: "vocabulary-study" — Workers Builds는 대시보드 Worker 이름이 이 값과 일치해야 빌드가 성공한다.
  • 문서: docs/PRD.md는 Worker 아키텍처만 서술하고 CI/CD 파이프라인 전용 섹션이 없음 → 문서 수정 불필요.

3설계

빌드+배포는 CF Workers Builds로 이관하고, GH Actions에는 시크릿 0개 스모크만 남긴다. Workers Builds는 배포 후 검증 훅이 없으므로 회귀 안전망은 GH Actions로 유지한다. 세 가지 핵심 결정:

① Node 24 고정 — .node-version 파일

Workers Builds 기본 Node는 22.16.0이며 .node-version/.nvmrc/NODE_VERSION(대시보드 변수) 순으로 버전을 존중한다(CF 빌드 이미지 문서 확인). 리포 루트에 값 24인 .node-version을 추가해 버전 관리로 고정한다. 대안 대시보드 NODE_VERSION 변수 — 리포에 안 남아 불투명하므로 기각.

② 스모크가 “새 배포”를 실제로 검증 — X-Worker-Version 변경 게이트 (추천)

문제: 구버전도 이미 200/401을 반환하므로, 단순 “200/401 재시도”는 CF 새 배포가 반영되기 전 구버전 상대로 즉시 통과한다 → 이 커밋의 회귀(예: auth 우회)를 놓치는 형식만 남은 안전망이 된다. 게다가 스모크와 CF 빌드는 같은 push로 독립 트리거돼 완료 순서 보장이 없다.

해결(시크릿 0개 유지): ⓐ 워크플로우 시작 시 미인증 /api/words 401 응답의 X-Worker-Version을 베이스라인으로 캡처 → ⓑ 값이 바뀔 때까지 폴링(= CF 새 배포 반영 확인, 넉넉한 타임아웃) → ⓒ root 200 + 미인증 /api/words 401 단언. 버전이 끝내 안 바뀌면 타임아웃→red — 이는 CF 빌드 실패까지 함께 잡아주는 부수 효과가 있다.

대안 워커에 git SHA를 노출하고 github.sha와 일치 검사 — 워커 코드·빌드 주입 변경이 필요해 과함, 기각. 순수 200/401 재시도 — 위의 구버전 오탐 위험, 기각. 버전 게이트가 시크릿 0개를 유지하면서 안전망을 실질화하는 최소 변경.

③ deploy-auth-smoke.sh 재활용 — 전파 윈도우 401 감시

버전 게이트 통과 후 SMOKE_PASSWORD 없이 scripts/deploy-auth-smoke.sh $URL 60을 호출해 #23의 실제 회귀 시나리오(전파 윈도우 동안 wrong/none → 401 유지, 버전 값 로깅)를 반복 감시한다. 스크립트가 커버하지 않는 root 200만 인라인으로 추가한다. 스크립트 자체는 이미 시크릿 없이 401 경로를 검사하므로 변경 불필요.

URL 출처

https://vocabulary-study.dambi626626.workers.dev를 워크플로우에 하드코딩한다(공개 값·매우 안정적, “유지보수 소멸” 취지에 부합). 대안 리포 변수 vars.PRODUCTION_URL(시크릿 아님) — 워커 이름/서브도메인 변경 시 코드 수정 없이 대응. 하드코딩을 추천하되 주석으로 대안을 남긴다.

전환 순서 (무중단)

  • Phase 0 · 대시보드 연결·검증 수동 — Workers Builds에 리포 연결: 빌드 커맨드 npm run build, deploy 커맨드 기본 npx wrangler deploy, 프로덕션 브랜치 main, Worker 이름 vocabulary-study 일치, Node 24. 첫 CF 배포가 정상인지 확인(연결 시 초기 배포/프리뷰). 이 동안 deploy.yml과 병존 → main push 시 일시적 이중 배포는 허용.
  • Phase 1 · 이 PR 머지 코드 — .node-version 추가 + deploy.yml 삭제 + smoke.yml 추가. 머지=main push → Workers Builds 배포 + 새 스모크 실행, deploy.yml이 사라져 이중 배포 종료.
  • Phase 2 · 시크릿 제거·토큰 폐기 수동 — GH 시크릿 CLOUDFLARE_API_TOKEN·CLOUDFLARE_ACCOUNT_ID 제거 + CF 대시보드에서 API 토큰 폐기. 반드시 deploy.yml 삭제(Phase 1) 이후.
  • Phase 3 · 최종 검증 — 실제 main push 1회로 CF 배포 + 스모크 그린(버전 변경 감지 → 200/401 통과) 확인.
절대 순서 규칙: (1) Workers Builds 연결·검증 전에 deploy.yml을 삭제하지 않는다(배포 공백). (2) deploy.yml 삭제 전에 시크릿을 제거하지 않는다(deploy.yml red). (3) 토큰 폐기는 맨 마지막.
