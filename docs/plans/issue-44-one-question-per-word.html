학습 중 단어 세션 출제를 단어당 1문제로 변경 — 모드1·모드2 중복 출제 제거

1요약

현재 세션 큐는 학습 중 단어마다 모드1(D<3)·모드2(E<3) 문제를 각각 후보로 넣어 단어당 최대 2문제가 같은 세션에 출제된다. 소유자 결정(2026-07-19)에 따라 한 세션에서 한 단어는 한 모드로만 출제되도록 바꾼다: 둘 다 미달이면 모드를 무작위로 하나 선택(복습 문제와 동일한 방식), 한쪽만 미달이면 그 모드로 출제. 홈 화면의 "오늘 세션 · n문제" 산식도 단어당 1문제 기준으로 맞추고, PRD·디자인 PRD의 해당 절을 갱신한다. 부수 효과로 초기 큐에 같은 단어가 두 번 들어갈 수 없게 되므로, 인접 중복 분산 로직은 죽은 코드가 되어 제거한다.

2현황 분석

세션 큐 구성과 홈 집계는 모두 클라이언트 순수 모듈이다 (PRD §7.3). 변경 표면은 lib 2개 + 테스트 2개 + 문서 2개로 닫힌다.

  • 큐 구성 — src/lib/sessionQueue.ts:56-64: 학습 중 단어를 총 정답 수 내림차순으로 돌며 m1<3이면 m1 문제, m2<3이면 m2 문제를 각각 후보에 추가한다. 신규 단어(0/0)는 후보 2개 → 48단어 시트에서 후보 96개, 상한 60에 걸려 세션이 60문제가 된다.
  • 홈 산식 — src/lib/homeStats.ts:30-31: 같은 규칙으로 learningCandidates를 단어당 1~2개 센다. sessionCount = min(60, min(복습대기,60) + learningCandidates) (homeStats.ts:43).
  • 상태 판정 — src/lib/wordState.ts:36-38: m1<3 || m2<3이면 무조건 learning. 즉 학습 중 단어는 항상 최소 한 모드가 미달이므로, 단어당 1문제 규칙에서는 학습 중 후보 수 = 학습 중 단어 수가 된다.
  • 모드 무작위 선례 — 복습 문제는 이미 randomMode(rng)(sessionQueue.ts:93-95, rng()<0.5 → m1)로 모드를 정한다. 학습 중 양쪽 미달도 같은 함수를 재사용하면 된다.
  • 인접 분산 — sessionQueue.ts:105-143(wordKey·disperseAdjacentDuplicates·trySwap·pairOk)과 QueueWord의 tab+hanzi 요구(sessionQueue.ts:24)는 "단어당 2문제"가 만들던 인접 중복을 풀기 위한 장치다. 상태는 상호 배타(getWordState가 단일 상태 반환)라 복습·학습 양쪽에 같은 단어가 들어갈 수 없고, 단어당 1문제가 되면 초기 큐에 중복 자체가 불가능해져 전부 죽은 코드가 된다.
  • 오답 재삽입은 무관 — src/lib/studySession.ts:82-87: 학습 중 오답 시 같은 문제를 큐 뒤에 1회 재삽입하는 것은 세션 진행 중 로직으로, 이번 변경(초기 큐 구성)과 독립적이다. 이슈도 유지를 명시했다.
  • 소비처 — src/screens/HomeScreen.tsx:74(buildSessionQueue)·:37(computeHomeStats)뿐. 시그니처가 유지되므로 화면 코드는 변경 없다.
  • 문서 — docs/PRD.md:136-141(§6.1 큐 구성), docs/PRD.md:259-274(§12 결정 이력 표), docs/design-prd.md:75(§3 세션 수 산식)이 옛 규칙을 명시하고 있다.

3설계

  • 모드 선택 규칙 (이슈 기본안 채택): 학습 중 단어당 문제 1개 —
    • m1<3 && m2≥3 → m1 / m1≥3 && m2<3 → m2 (기존과 동일)
    • 둘 다 미달 → randomMode(rng)로 무작위 선택 — 복습 문제 모드 선택과 같은 함수·같은 확률(50:50). 이슈 코멘트에 다른 규칙 확정이 없으므로 기본안 그대로 간다.
  • 홈 산식 단순화: 학습 중 단어는 정의상 후보가 정확히 1개이므로 learningCandidates 카운터를 없애고 sessionCount = min(60, min(reviewDue,60) + learning)으로 쓴다. 큐 산식과의 동치가 코드에서 자명해진다.
  • 죽은 분산 로직 제거: 초기 큐에 같은 단어 중복이 불가능해지므로 disperseAdjacentDuplicates 일가를 제거하고 QueueWord의 tab+hanzi 요구도 걷어낸다. 방어적으로 남겨두는 안은 기각 — 도달 불가 코드는 "중복이 있을 수 있다"는 거짓 신호를 주고, 유일성은 테스트로 고정하는 편이 명확하다. (제네릭 제약은 WordProgress로 좁아지며 호출부는 WordEntry라 그대로 호환.)
  • rng 호출 순서: 복습 모드 → 학습 중 모드(양쪽 미달 단어만) → 셔플 순서로 소비된다. 시드 고정 테스트는 이 순서 전제로 다시 짠다.

4작업 단계

  1. 큐 구성 변경: 학습 중 후보 루프를 단어당 1문제로 — 한쪽 미달은 그 모드, 양쪽 미달은 randomMode(rng). 상단 doc 코멘트(§6.1 서술)와 "후보 1~2개" 코멘트를 새 규칙으로 갱신. src/lib/sessionQueue.ts:26-29, 55-64
  2. 죽은 코드 제거: wordKey·disperseAdjacentDuplicates·trySwap·pairOk 삭제, buildSessionQueue의 분산 호출 제거, QueueWord 타입(tab+hanzi 요구) 제거 후 제네릭을 WordProgress 제약으로 단순화. src/lib/sessionQueue.ts:23-24, 71, 105-143
  3. 큐 테스트 재작성: ① "단어당 최대 2문제" 테스트를 모드 선택 규칙 테스트로 교체 — 한쪽 미달(갑 3,1→m2 / 을 0,4→m1) + 양쪽 미달을 상수 rng(()=>0→m1, ()=>0.7→m2)로 고정. ② 동률 정렬 테스트를 후보 수 절반에 맞게 재구성(예: 고득점 59 + 동률 2 → 61후보에서 컷). ③ 분산 테스트 2건을 "큐에 같은 단어(tab+hanzi)가 두 번 없다" 유일성 테스트(시드 다수)로 교체. ④ 신규 시나리오: 신규 단어 48개(0/0) → 큐 길이 48. ⑤ 상한 테스트(복습30+학습40)는 코멘트만 갱신(후보 80→40, 단언은 유효). src/lib/sessionQueue.test.ts
  4. 홈 산식 변경: learningCandidates 카운터 제거, sessionCount = min(60, min(reviewDue,60) + learning). HomeStats.sessionCount doc 코멘트의 산식 표기 갱신. src/lib/homeStats.ts:16, 24, 30-31, 43
  5. 홈 산식 테스트 갱신: 혼합 케이스 기대값 6→5, 신규 단어 48개 → sessionCount 48 케이스 추가 (큐 테스트 ④와 같은 시나리오로 산식·큐 일치를 양쪽에서 고정). src/lib/homeStats.test.ts
  6. 문서 갱신: PRD §6.1 2항을 "단어당 1문제 — D<3만: 모드1 / E<3만: 모드2 / 둘 다: 무작위"로, 3항에서 "같은 단어 인접 분산" 문구 제거(초기 큐 중복 불가). §12 결정 이력 표에 소유자 결정(2026-07-19, 단어당 1문제·양쪽 미달 시 무작위, #44) 행 추가. design-prd §3 산식을 min(60, min(복습대기,60) + 학습중 단어 수)로 갱신. docs/PRD.md:136-141, 263-274, docs/design-prd.md:75
  7. 검증 후 커밋·PR: npm test(vitest)·tsc 통과 확인 → 레포 스타일 커밋(기능/테스트/문서) → push → Closes #44 PR. package.json scripts 기준 ✓ vitest 89개·tsc -b·oxlint 전부 통과. 계획 대비 이탈 없음 — 이 플랜 커밋 직후 push·PR 오픈.
