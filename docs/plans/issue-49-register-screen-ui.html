1요약

단어 등록 시스템(전체 플랜 docs/plans/word-registration-system.md) 3단계 중 2단계 화면이다. 홈 화면에 저강조 링크로 진입해, JSON 붙여넣기 → 형식·pinyin-pro 병음 검증·시트 중복 대조 → 검증 결과 테이블 → 탭 선택/생성 → 중복 확인 → 제출까지 이어지는 클라이언트 플로우를 구현한다. Worker API(GET /api/tabs, POST /api/words/register, #48)는 아직 미구현이라 계약(플랜 §6)만 근거로 병렬 진행하고, 종단 검증은 #48 머지 이후로 미룬다. pinyin-pro가 gzip 약 145KB로 무거워, 지연 로딩으로 학습 플로우 번들에 영향을 주지 않게 설계한다.

3설계

주요 설계 결정과 기각한 대안:

항목결정기각한 대안
번들 크기 대응Screen에 'register' 추가 + React.lazy/Suspense로 RegisterScreen 지연 로딩. pinyin-pro가 gzip 약 145KB(의존성 0개)이고 등록 화면은 비핵심 경로라, 즉시 import하면 로그인·홈·학습 초기 번들이 그만큼 무거워진다.즉시 import — 이슈 자체 완료 조건("번들 크기 확인")에 실질적으로 대응하지 못하고 초기 로드만 저하시킴.
검증 로직 위치src/lib/registerValidation.ts에 순수 함수로 분리 — wordState/sessionQueue/studySession과 동일하게 로직·컴포넌트를 계층화.컴포넌트 내부에 검증 로직 인라인 — 이 저장소는 화면 컴포넌트 테스트 관행이 없으므로(§2) 로직을 분리해야만 단위 테스트 커버리지를 확보할 수 있음.
병음-한자 일치 판정한자를 코드포인트 단위로 순회, 각 글자의 후보 배열(pinyin(char, {multiple:true, type:'array'}) — 공식 문서상 multiple은 text 길이 1일 때만 유효)을 구해 카티전 곱 중 하나라도 공백 제거한 입력 병음과 정확히 일치하면 통과.붙여넣은 병음 문자열을 음절 단위로 직접 분리하는 파서 — 음절 경계가 모호해(예: "jing"+"ji" vs "ji"+"ngji") 범용 분리가 어렵다. 카티전 곱 비교는 이 문제를 완전히 우회한다. 성조 부호 형식 검증도 별도 정규식 없이 자연히 겸한다 — 후보는 항상 성조 부호를 포함하므로 무성조·숫자성조 입력은 어떤 후보와도 일치하지 않아 저절로 차단된다.
행 분류 우선순위오류(형식·병음불일치·빈 값·입력 내 중복) > 중복(시트 내, 선택 탭 기준). 입력 내 중복 한자는 어느 쪽이 진짜인지 판별 불가하므로 시트-중복보다 먼저 차단으로 분류.입력 내 중복도 "중복"으로 묶어 스킵 허용 — 사용자가 실수로 둘 중 하나만 남기려던 의도를 놓칠 위험이 있음.
#48 의존 처리GET /api/tabs/POST /api/words/register 클라이언트 함수는 플랜 §6 계약대로 선(先)구현하고 mock fetch 단위 테스트로 커버. 화면은 탭 조회 실패를 홈 화면과 동일한 에러 카드 패턴으로 흡수해 크래시 없이 동작.#48 머지까지 대기 — 이슈 본문이 명시적으로 "#48과 병렬 진행 가능"이라 밝힘.
중복 확인 UI모달 대신 인라인 확인 배너 + "확인하고 제출" 2단계 버튼.모달 다이얼로그 — 이 저장소에 모달 컴포넌트 관행이 없고(StudyScreen "종료"도 design-prd §4.1에 따라 확인 없이 인라인 즉시 처리), 전부 인라인 상태 전환으로 화면을 구성하는 기존 스타일과 어긋남.
새 탭 이름 클라이언트 검증트림·_시작 차단을 UX 개선 목적으로 클라이언트에도 적용하되, 최종 강제는 #48 Worker 책임으로 남김.클라이언트 검증 생략 — 플랜 §2 "신뢰 경계"가 2·3단계 모두 재검증하도록 명시하므로, 클라이언트 검증은 우회 가능해도 방어선 하나를 보태는 것으로 충분함.
