#43기능 개선

복습 오답 기록(POST /api/review-fail) 전송 실패 시 간격 후퇴 유실 — 재시도 큐 미적용

1요약

재시도 큐(#18, PR #40)는 POST /api/answer 전용으로만 구현되어 있어, 복습 오답 시 호출하는 POST /api/review-fail이 실패하면 StudyScreen.tsx:69에서 그냥 무시되고 간격 후퇴가 유실된다. 큐 저장 포맷을 answer 전용 배열에서 종류가 태그된(tagged) 엔트리 배열로 확장해 review-fail 실패분도 같은 FIFO 큐에 담고, 앱 로드 시·다음 API 성공 시점에 종류별로 올바른 엔드포인트로 재전송한다.

2현황 분석

재시도 큐(#18)는 answer 전용 구조다. src/lib/retryQueue.ts:20의 readQueue()는 저장된 JSON을 AnswerRecord[]로 취급하고, flushRetryQueue()(retryQueue.ts:60-84)는 큐 맨 앞 항목을 항상 postAnswer(retryQueue.ts:69)로만 재전송한다. 큐에 review-fail 실패분을 넣을 방법 자체가 없다.

review-fail 실패는 지금 완전히 삼켜진다. src/screens/StudyScreen.tsx:67-70의 fireEffect에서 answer 분기는 실패 시 enqueueAnswer(record)(retryQueue.ts:43)로 큐에 적재하지만, review-fail 분기는 postReviewFail(word.tab, word.hanzi).catch(() => {})로 끝 — 실패가 그냥 사라진다.

두 엔드포인트는 요청 바디 형태가 다르다. src/lib/api.ts:112-122의 postAnswer(answer: AnswerRecord)는 {tab, hanzi, mode, timestamp, isReview} 전체를 바디로 보내는 반면, api.ts:125-135의 postReviewFail(tab, hanzi)는 {tab, hanzi} 두 필드만 보낸다. 즉 큐 엔트리를 단일 타입으로 합칠 수 없고, 종류를 구분해 저장·재생해야 한다.

재전송 배선은 이미 종류 무관하게 재사용 가능하다. src/App.tsx:26-30은 마운트 시 flushRetryQueue()를 1회 호출하고 setApiSuccessHandler로 이후 모든 API 성공 시점마다 재호출되도록 배선해 두었다(src/lib/api.ts:37-39) — 이 배선 자체는 큐 내용물의 종류를 몰라도 되므로 변경이 필요 없다. src/screens/HomeScreen.tsx:53의 getRetryQueueLength() 기반 인디케이터도 큐 길이만 세므로 종류 무관하게 그대로 동작한다.

순서 보존 요구. 이슈는 "같은 단어의 answer/review-fail이 섞여 있어도 시트 상태가 일관되도록 큐에 담긴 순서대로 재전송"을 요구한다 — 현재 flushRetryQueue가 이미 단일 배열을 FIFO로(맨 앞부터, 성공 시에만 제거, 첫 실패에서 중단) 처리하므로, answer/review-fail을 같은 배열에 태그를 붙여 넣기만 하면 이 요구는 자동으로 만족된다.

3설계

채택안 — 큐 엔트리를 종류 태그가 붙은 유니온으로 확장.

type RetryQueueEntry =
  | { kind: "answer"; record: AnswerRecord }
  | { kind: "review-fail"; record: ReviewFailRecord }  // { tab, hanzi }

readQueue/saveQueue는 RetryQueueEntry[] 기준으로 그대로 재사용하고, flushRetryQueue는 큐 맨 앞 엔트리의 kind에 따라 postAnswer 또는 postReviewFail을 호출하도록 한 줄만 분기한다. 기존 FIFO·재진입 가드·"첫 실패에서 중단"·"성공 시에만 제거" 정책은 엔트리 종류와 무관하게 그대로 유지된다.

기각한 대안 1 — review-fail 전용 별도 큐(별도 localStorage 키). 큐가 둘로 나뉘면 같은 탭에서 answer와 review-fail을 별도 순서로 재전송하게 되어, 이슈가 명시한 "섞여 있어도 순서대로" 요구를 어긴다 — 예를 들어 같은 단어에 대해 review-fail(후퇴) 이후 answer(갱신)가 발생했는데 두 큐를 독립적으로 비우면 answer가 먼저 반영되어 최종 갱신 값을 review-fail이 덮어쓸 수 있다.

기각한 대안 2 — 저장 포맷을 {type, payload} 대신 판별 불가능한 유니온으로 두고 필드 존재 여부로 추론. mode/timestamp/isReview 필드 유무로 종류를 추론하는 방식은 taggedunion보다 취약하고, review-fail 레코드가 우연히 mode 유사 필드를 갖게 되는 미래 변경에 깨지기 쉽다. 명시적 kind 태그가 더 안전하다.

레거시 큐 데이터 처리. 배포된 앱이 이미 (#18/PR #40) 기준 AnswerRecord[] 평면 배열 포맷으로 큐를 저장 중일 수 있다. readQueue에서 파싱한 배열의 각 항목에 kind 필드가 없으면 레거시 answer 레코드로 간주해 { kind: "answer", record: item }로 승격시켜 마이그레이션한다 — 배포 시점에 남아있던 미전송 항목이 파싱 실패로 통째로 버려지는 것(유실 방지라는 #18의 취지 위반)을 막기 위함.

4작업 단계

  1. 큐 엔트리를 RetryQueueEntry 태그드 유니온으로 확장한다. readQueue는 레거시(플랫 AnswerRecord) 항목을 kind: "answer"로 승격시켜 반환하고, saveQueue는 그대로 저장한다. enqueueAnswer는 {kind:"answer", record}로 래핑하도록 내부만 바꾸고 시그니처는 유지하며, enqueueReviewFail(record: ReviewFailRecord)를 신설한다. flushRetryQueue는 맨 앞 엔트리의 kind로 postAnswer/postReviewFail을 분기 호출하도록 수정한다. src/lib/retryQueue.ts 완료 — 계획대로 구현.
  2. ReviewFailRecord 타입({ tab: string; hanzi: string })을 export하고, postReviewFail의 주석을 postAnswer와 같은 톤으로 갱신해 재시도 큐 연계(#43)를 명시한다. 함수 시그니처·동작은 변경하지 않는다. src/lib/api.ts 완료 — 계획대로 구현.
  3. fireEffect의 review-fail 분기에서 .catch(() => {})를 .catch(() => enqueueReviewFail({ tab: word.tab, hanzi: word.hanzi }))로 교체한다. answer 분기와 대칭되는 형태로 맞춘다. src/screens/StudyScreen.tsx 완료 — 계획대로 구현.
  4. 기존 answer 전용 테스트를 새 엔트리 스키마에 맞게 갱신하고, review-fail enqueue·flush·레거시 마이그레이션·answer와 review-fail이 섞인 큐의 FIFO 순서 보존·review-fail 실패 시 큐 유지 케이스를 추가한다. src/lib/retryQueue.test.ts 완료 — 계획대로 구현, 9개 테스트 추가/갱신.

5리스크

리스크영향완화책
배포된 앱에 이미 쌓여있는 레거시(플랫 AnswerRecord[]) 큐 데이터새 포맷 배포 후 kind 없는 기존 항목이 파싱되지 않아 재전송 자체가 실패하거나 항목이 통째로 버려질 수 있음2단계에서 readQueue가 kind 없는 항목을 answer로 자동 승격 — 마이그레이션 테스트로 확인
review-fail도 flushRetryQueue의 "첫 실패에서 중단" 정책을 그대로 따름큐 앞쪽에 review-fail 실패가 있으면 뒤의 answer도 이번 플러시에서 재전송되지 않음(기존 answer 전용 정책과 동일한 트레이드오프)기존 PRD §10 트레이드오프를 그대로 계승 — 새 리스크 아님, 유실 방지 우선이 의도된 동작

6테스트 계획

  • retryQueue.test.ts에 enqueueReviewFail 단위 테스트 추가 — 빈 큐 적재, 기존 항목 뒤 append, 변경 이벤트 발화 확인.
  • flushRetryQueue가 review-fail 엔트리를 POST /api/review-fail로, answer 엔트리를 POST /api/answer로 각각 올바른 바디({tab,hanzi} vs 전체 AnswerRecord)로 보내는지 확인.
  • answer/review-fail이 섞인 큐를 시딩해 FIFO 순서 그대로 재전송되는지(호출 순서가 큐 삽입 순서와 일치) 확인.
  • review-fail 재전송 실패 시 해당 항목과 잔여가 큐에 유지되는지(기존 answer 중간 실패 테스트와 대칭) 확인.
  • kind 필드 없는 레거시 플랫 AnswerRecord[] JSON을 시딩했을 때 answer로 정상 파싱·재전송되는지 확인.
  • 기존 answer 전용 회귀 테스트(빈 큐, append, 파손 JSON, 재진입 가드, 전송 중 신규 적재)가 모두 통과하는지 npm test로 확인.
  • 수동: 네트워크 차단 상태에서 복습 오답 → 큐에 review-fail 항목 적재 확인(이슈 완료 조건 1) → 네트워크 복구 후 재로드 → 시트 F열 간격 후퇴 반영 확인(완료 조건 2) → 재전송 성공 항목이 큐에서 제거돼 중복 후퇴 없는지 확인(완료 조건 3).

7완료 조건

  • 네트워크 차단 상태에서 복습 오답 → 재시도 큐에 review-fail 항목이 쌓인다. 단위 테스트(fetch reject 시뮬레이션)로 동등 검증 — 실제 브라우저·네트워크 차단 수동 시연은 이 세션에서 미실시(개발 서버·실기기 필요).
  • 네트워크 복구 후 앱 재로드 시 재전송되어 F열 간격 후퇴가 시트에 반영된다. 단위 테스트로 재전송 로직(올바른 엔드포인트·바디) 검증 — 실제 구글 시트 F열 반영 수동 확인은 이 세션에서 미실시(실 시트 연동 필요).
  • 성공적으로 전송된 항목이 큐에서 제거되어 간격이 중복 후퇴하지 않는다.
  • 기존 answer 재시도 큐 동작에 회귀가 없다(기존 테스트 통과) — npm test 93개 전체 통과.
  • 레거시(비태그) 큐 데이터가 새 배포에서도 파싱되어 유실되지 않는다.
