API 비밀번호 인증 미들웨어 — Bearer 토큰 검증

damdam6/vocabulary-study 브랜치 feat/6-api-password-auth-middleware ← main 라벨 enhancement 2026-07-15

1요약

현재 worker/index.ts는 /api/* 요청에 대해 어떤 인증 검사도 하지 않아, APP_PASSWORD 시크릿이 등록되어 있음에도 사실상 전체 공개 상태다. 게다가 프로덕션 APP_PASSWORD 값은 PR #2 이후 임시값(changeme)에 머물러 있다. 이번 작업은 /api/* 전용 인증 미들웨어를 추가해 Authorization: Bearer <APP_PASSWORD> 헤더를 검사하고 불일치·누락 시 401을 반환하도록 하며, 프로덕션 시크릿을 실제 값으로 교체한다.

2현황 분석

worker/index.ts:1-10의 fetch(request) 핸들러는 env 인자를 받지 않고, url.pathname === "/api/health"일 때만 무조건 200을 반환하며 그 외는 404다. 현재 정의된 /api/* 라우트는 /api/health 하나뿐이고, PRD에 명시된 /api/words·/api/answer·/api/review-fail은 아직 구현되지 않았다(별도 이슈).

worker-configuration.d.ts:5-8의 Env 인터페이스에는 APP_PASSWORD: string이 이미 타입으로 반영되어 있고, PR #2 기록(gh pr view 2)에 따르면 프로덕션 시크릿 3종(SHEET_ID/APP_PASSWORD/GOOGLE_SERVICE_ACCOUNT_KEY)이 wrangler secret으로 이미 등록되어 있으나, APP_PASSWORD는 "실제 값 결정 후 교체 필요"라는 노트가 남아 있는 임시값(changeme) 상태다.

docs/PRD.md:178·docs/PRD.md:219-224(§7.3, §8)에 명세된 인증 규칙: 모든 /api/* 요청에 Authorization: Bearer <APP_PASSWORD> 필수, 불일치 시 401. 레이트리밋·세션 만료 등 추가 보안은 개인용 서비스라 v1 비범위로 명시되어 있다.

worker/lib/google-auth.ts·worker/lib/sheets.ts는 이미 "파일 상단에 목적을 요약하는 주석 + env: Env를 첫 인자로 받는 함수" 컨벤션을 쓰고 있어, 인증 로직도 같은 패턴으로 worker/lib/ 아래 분리하는 것이 기존 스타일과 맞는다.

src/App.tsx:38-46는 C3 스캐폴딩 데모로 Authorization 헤더 없이 fetch('/api/health')를 호출한다. 이번 변경 후 이 버튼은 401을 받게 되지만, 실제 로그인 화면(PRD §9)은 이번 이슈 범위 밖이다.

3설계

  • 모듈 분리: worker/lib/auth.ts를 신설해 isAuthorized(request, env) 함수를 둔다. index.ts에 인라인으로 넣는 방식은 기각 — /api/words 등 라우트가 늘어날 이후 이슈에서 중복되고, google-auth.ts/sheets.ts가 이미 lib 분리 컨벤션을 쓰고 있다.
  • 적용 범위: 라우트별로 개별 검사를 넣지 않고, url.pathname.startsWith("/api/")인 모든 요청에 공통으로 적용한다. 이렇게 하면 후속 이슈에서 /api/words 등을 추가할 때 인증 검사를 매번 새로 넣지 않아도 된다.
  • 비교 방식: Authorization 헤더를 "Bearer " 스킴으로 파싱해 env.APP_PASSWORD와 단순 문자열 비교(!==)한다. 타이밍 세이프 비교(constant-time compare)는 채택하지 않음 — PRD §8이 "그 이상의 보안(레이트리밋, 세션 만료 등)은 개인용이므로 v1 비범위"라고 명시했고, 단일 사용자 대상 서비스에서 네트워크 지연 대비 타이밍 공격의 실효성이 낮다. 리스크 섹션에 트레이드오프로 기록.
  • 실패 응답: 헤더 누락·스킴 불일치·값 불일치 모두 동일하게 401, 바디 없음 — 현재 /api/health의 404 응답 스타일(new Response(null, { status: ... }))과 일관되게 유지.

4작업 단계

  1. Authorization 헤더에서 Bearer 스킴을 파싱하고 env.APP_PASSWORD와 비교하는 isAuthorized(request, env) 함수를 신설한다. 기존 lib 파일들과 같은 스타일(상단 목적 주석, env: Env 인자)을 따른다. worker/lib/auth.ts (신규)
  2. fetch(request, env)로 시그니처를 확장하고, url.pathname.startsWith("/api/")이면서 isAuthorized가 false일 때 401을 반환하도록 라우팅 앞단에 검사를 추가한다. worker/index.ts
  3. 로컬 검증용으로 .dev.vars(gitignore 대상, 미커밋)에 APP_PASSWORD 임시값을 설정하고 wrangler dev로 200/401 케이스를 curl로 확인한다. .dev.vars (로컬 전용, 커밋 안 함)
  4. 프로덕션 APP_PASSWORD를 임시값(changeme)에서 실제 값으로 교체한다 (wrangler secret put APP_PASSWORD). 실제 비밀번호 값은 사용자가 직접 정하며, 이 세션은 값을 대신 결정하거나 대화·코드에 노출하지 않는다. 프로덕션 시크릿 (코드 변경 없음)

5리스크

리스크영향완화책
프로덕션 시크릿 교체(wrangler secret put)는 되돌리기 어려운 운영 작업비밀번호 값을 잘못 정하거나 분실하면 서비스 접근 불가사용자가 직접 값을 정하고 명시적으로 확인한 뒤에만 실행. 값 자체는 대화 로그·코드에 남기지 않는다
src/App.tsx의 /api/health 데모 버튼이 인증 헤더 없이 호출배포 후 데모 버튼이 401을 반환 — 기능 저하로 보일 수 있음이 이슈의 완료 조건에 포함되지 않음. 로그인 화면(PRD §9)은 별도 이슈 범위이므로 리포트에서 명시적으로 알림
단순 문자열 비교(!==)는 이론상 타이밍 공격에 노출개인용 단일 사용자 서비스라 실질 위험은 낮음PRD §8에서 v1 비범위로 명시. 필요해지면 후속 이슈로 constant-time 비교 검토

6테스트 계획

  • wrangler dev 로컬 실행 후 curl -H "Authorization: Bearer <정답>" .../api/health → 200 및 기존 JSON 바디 확인
  • curl (Authorization 헤더 없이) .../api/health → 401 확인
  • curl -H "Authorization: Bearer <오답>" .../api/health → 401 확인
  • curl -H "Authorization: Basic ..." (Bearer가 아닌 스킴) .../api/health → 401 확인
  • curl .../api/nope (정의되지 않은 /api/* 경로, 인증 없음) → 401 (404가 아님 — 인증 검사가 라우팅보다 먼저 실행되는지 확인)
  • npm run build(tsc -b + vite build) 에러 없음 확인
  • npm run lint(oxlint) 통과 확인
  • 프로덕션 시크릿 교체 후 실제 배포 엔드포인트에서 정답/누락/오답 세 케이스 재확인

7완료 조건

  • 올바른 비밀번호(Authorization: Bearer <APP_PASSWORD>)로 /api/health 등 /api/* 호출 시 정상 통과(200)
  • Authorization 헤더 누락 또는 값 불일치 시 /api/* 전 경로에서 401 응답
  • 프로덕션 APP_PASSWORD 시크릿이 임시값(changeme)에서 실제 값으로 교체 완료
