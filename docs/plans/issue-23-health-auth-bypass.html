#23버그 수정

/api/health — 틀린 Authorization 토큰이 일시적으로 401 대신 200 반환

damdam6/vocabulary-study 브랜치 fix/23-health-auth-bypass ← main 라벨 bug 2026-07-18

1요약

배포 직후 몇 초간 틀린 Bearer 토큰으로 /api/health가 200을 반환한 현상은, 코드 결함이 아니라 Cloudflare 엣지 전파 윈도우 동안 인증이 없던 구버전(pre-#6)이 일부 요청을 처리한 것으로 판단된다. 무인증→인증 최초 전환 배포에서만 존재할 수 있는 1회성 노출이며, 이후 배포에서는 신·구 버전 모두 인증을 강제하므로 재발 경로가 없다. 이 가설을 실측으로 확정하기 위해 응답에 서빙 버전을 노출하는 관측성 변경(version_metadata 바인딩 + X-Worker-Version 헤더)과 배포 직후 인증 스모크 스크립트를 추가하고, 인증 순서를 고정하는 회귀 테스트를 넣은 뒤, 실제 배포로 재현 불가·버전 혼재 여부를 확인해 이슈에 기록하고 PR로 마무리한다.

2현황 분석

현재 코드의 요청 흐름은 다음과 같고, 로직상 우회 경로가 없다.

  • worker/index.ts:10 — /api/* 전체가 !(await isAuthorized(request, env))이면 즉시 401 (WWW-Authenticate: Bearer). 인증 검사가 /api/health 분기(worker/index.ts:14)보다 앞선다.
  • worker/lib/auth.ts:8-22 — APP_PASSWORD 미설정 시 거부, Bearer 접두사 검사, SHA-256 다이제스트 상수시간 비교. 다이제스트는 isolate 스코프 cachedPasswordDigest(worker/lib/auth.ts:6)에 캐시.
  • wrangler.jsonc:13 — run_worker_first: ["/api/*"]이 스캐폴딩 커밋(783951a)부터 존재 → /api/*가 SPA fallback(에셋)으로 새어나간 적 없음.
  • 프런트 소비처: src/lib/api.ts:52의 verifyPassword가 후보 비밀번호를 Bearer로 GET /api/health 호출 — 응답 JSON에 필드를 추가해도 호환된다(상태코드만 사용).

git 이력 (인증 도입 전후):

  • 316b920 (pre-#6): /api/health가 무인증 200 — 당시 설계상 정상.
  • 516c9a9 (#6 병합) → 84edb6a(빈 비밀번호 우회 방지) → 75d7c38(상수시간 비교, async 전환) → 895ef4b(다이제스트 캐시): 모든 상태에서 인증이 health보다 앞서며, async 전환 커밋(75d7c38)에서 호출부 await도 동일 커밋에 함께 적용되어 missing-await 상태로 배포된 이력이 없다.

3원인 분석

유력 원인 — 배포 전파 윈도우의 신·구 버전 혼재. Cloudflare Workers 배포는 전 세계 엣지에 수 초~수십 초에 걸쳐 전파되며, 그동안 같은 콜로 안에서도 요청별로 다른 메탈(구버전 isolate)이 응답할 수 있다. 관측된 현상과 정확히 맞아떨어진다:

  • 틀린 토큰 → 200: 구버전(316b920)의 무인증 /api/health가 응답. 헤더 유무와 무관하게 200을 주는 코드였다.
  • 헤더 없음 → 401: 신버전이 배포된 메탈에 맞았거나 전파 완료 후 테스트된 요청.
  • 수십 초 뒤 일관된 401: 전파 완료. "재현 불안정"도 요청별 라우팅 복불복으로 설명된다.

배제한 대안 가설:

  • cachedPasswordDigest (이슈의 1순위 의심) — isolate의 env는 배포 버전 수명 동안 불변이고, 비교가 200이 되려면 제시한 토큰의 SHA-256이 캐시된 비밀번호 다이제스트와 일치해야 한다. "명백히 틀린 값"으로는 수학적으로 불가능. 캐시는 신선도 문제를 만들 수 없다.
  • 엣지/에셋 캐시 — Worker 생성 응답은 기본적으로 엣지 캐시되지 않고, 코드 어디에서도 Cache API를 쓰지 않으며, run_worker_first가 처음부터 있어 /api/health가 캐시 가능한 에셋(index.html)으로 응답된 적 없다. curl은 클라이언트 캐시도 없다.
  • 코드 결함(인증 순서·await 누락) — 2번 섹션의 이력 검토대로 배포된 어떤 상태에도 존재하지 않았다.

귀결: 무인증 엔드포인트 위에 인증을 처음 씌우는 배포에서만 생길 수 있었던 1회성 노출이다. 이후 배포는 신·구 버전 모두 인증을 강제하므로 틀린 토큰 200은 재발할 수 없다 — 이 예측을 4번 섹션의 실배포 스모크로 검증한다. 단, 검증 중 401이 아닌 응답이 하나라도 관측되면 가설을 폐기하고 신버전 결함으로 재조사한다(리스크 표 참고).

4작업 단계

  1. 버전 메타데이터 바인딩 추가 — wrangler.jsonc에 "version_metadata": { "binding": "CF_VERSION_METADATA" }를 넣고 npm run cf-typegen으로 타입 재생성. 이후 어떤 배포 윈도우에서도 응답별 서빙 버전을 식별할 수 있게 하는 기반. wrangler.jsonc, worker-configuration.d.ts(재생성) 노트: 워크트리에 gitignored .dev.vars가 없어 typegen이 시크릿 타입(APP_PASSWORD 등)을 떨궜음 — 메인 체크아웃의 .dev.vars를 심링크(로컬 전용, 미커밋)해서 해결.
  2. 응답에 서빙 버전 노출 — worker/index.ts의 401 응답과 /api/health 200 응답에 X-Worker-Version 헤더를 추가하고, health JSON에 version 필드를 추가(env.CF_VERSION_METADATA?.id ?? "unknown"). 틀린 토큰이 받는 401에도 헤더가 실리므로 스모크 로그만으로 버전 혼재를 직접 관측 가능. 프런트 verifyPassword는 상태코드만 보므로 호환. worker/index.ts
  3. 인증 순서 회귀 테스트 — 신규 worker/index.test.ts: 목 env(APP_PASSWORD 설정)로 fetch 핸들러를 직접 호출해 ① 틀린 토큰 → 401 ② 헤더 없음 → 401 ③ 잘못된 스킴 → 401 ④ APP_PASSWORD 미설정 → 401 ⑤ 올바른 토큰 → 200 + ok: true + X-Worker-Version 헤더를 고정. "인증이 health보다 먼저"라는 순서가 다시는 뒤집히지 않게 하는 안전망. 모듈 스코프 다이제스트 캐시 때문에 테스트 전체가 단일 APP_PASSWORD 값을 사용. worker/index.test.ts(신규) 노트: 핸들러가 satisfies로 선언돼 fetch는 2-인자 시그니처(ctx 없음)로 호출하고, new Request()의 cf 타입 불일치는 WorkerRequest 캐스트 헬퍼로 해결. 총 8개 케이스 추가(스위트 37개 그린).
  4. 배포 직후 인증 스모크 스크립트 — 신규 scripts/deploy-auth-smoke.sh: 인자로 받은 URL에 N초(기본 60초) 동안 틀린 토큰·무헤더 curl을 반복해 시각·상태코드·X-Worker-Version을 로깅하고, 401이 아닌 응답이 하나라도 있으면 exit 1. 선택적 SMOKE_PASSWORD 환경변수로 정상 토큰 200 경로도 확인. 이슈의 "배포 직후 반복 curl 테스트" 완료 조건을 재사용 가능한 형태로 충족. scripts/deploy-auth-smoke.sh(신규)
  5. 로컬 그린 확인 — npm run lint · npm run test · npm run build(tsc -b 포함) 모두 통과 확인. package.json 스크립트 사용(변경 없음) 확인: oxlint 0건, vitest 37/37, tsc -b + vite build 성공.
  6. 실배포 검증 — npm run deploy 직후 스모크 스크립트 실행(워커 URL은 npx wrangler deployments list 등으로 확인). 기대: 틀린 토큰·무헤더 전부 401 = 재현 불가 확정. 부수 관측: 전파 윈도우 동안 X-Worker-Version 헤더가 없는 401(구버전)과 있는 401(신버전)이 섞여 나오면 버전 혼재 메커니즘의 직접 증거. 로그를 보존해 이슈/PR에 첨부. scripts/deploy-auth-smoke.sh 실행 로그 결과: 75초간 222건 전부 401(위반 0) → 재현 불가 확정. 동시에 구버전(version=<none>) 89건과 신버전(version=d8f3dbb6) 133건이 배포 후 71초까지 교차 관측 — 전파 윈도우 신·구 버전 혼재의 직접 증거 확보. 상세 로그는 이슈 #23 코멘트에 첨부.
  7. 이슈 기록 + PR — 근본 원인·배제 근거·스모크 실측 결과를 이슈 #23에 코멘트로 기록하고, 커밋(플랜 HTML 최종 상태 포함)·푸시 후 Closes #23 PR 생성. docs/plans/issue-23-health-auth-bypass.html
